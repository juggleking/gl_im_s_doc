Pre-upgrade check list

Check the database structure. Is a dump and load recommended?

Execute a speed test in GLIMS 8 to be able to compare it with GLIMS 9.

What's the customer's overall feeling of the performance in GLIMS 8? Server and client (speed test must be done at least 2 times).

Beware of network latency, this can cause performance problems. Check the latency between client and server: must be lower than 400 microseconds. Best tool to use is HRping http://www.cfos.de
Ensure that the virus scanner is disabled for the GLIMS 9, GLIMSDB and PROGRESS folder.

GLIMS version 9.5.x requires Progress 11.3 SP 2 and .NET 4.0

GLIMS version 9.6.x requires Progress 11.6 SP 2 hotfix 006 and .NET 4.0

GLIMS version 9.8.x:

  • If used with Progress 11.6* => .NET 4.0 is required.
  • If used with Progress 11.7* => .NET 4.6 is required.

* The supported Progress version per GLIMS 9.8.x version can be found in the system requirements.

GLIMS version 9.9.x requires Progress 11.7 SP 5 and .NET 4.6.

Note

The .NET Framework patch level can be upgraded, however, you must stay within the supported minor version.

Example

Given a supported .NET Framework version of 4.6; any 4.6.x patch level above 4.6 and lower than 4.7 can be assumed as supported.
Check the Progress system requirements
If VMware is used, check the VMware settings. Avoid overcommitting the CPUs: there must be enough power left for ESXi itself and also check that power-management is turned off on all levels (BIOS, ESX and Windows)

Check if the Progress CD-ROM is available on site.

Make sure your Progress licenses and your installkit (CD, dvd or install file (*.tar, or *.exe) match. To verify, look at the Media ID on the license paper and verify that this number matches your product below. Eg. IBM AIX 32bit product would be "??", the 64bit product of IBM AIX would be "??".

This table comes from the Progress Product availability guide and can also be found on

http://communities.progress.com/pcom/docs/DOC-32338

Before performing a structural GLIMS upgrade, always make sure that you have a valid backup of the GLIMS databases.

Also backup the *.st files.
Check if you have the latest functional and corrective service pack for the new structural GLIMS version (MIPS support website)

It is recommend not to install GLIMS in the commonly used 'Program files' directory.

The directory should not contain spaces. Example: "D:\glims"

Check if you have all necessary license keys for GLIMS.

GLIMS 9 requires an OE Application Svr Ent license for Progress as well.
Make sure you have a Windows administrator account including the password.
Verify if a local user account with the same name and password as the domain login account exists. If not, add the new local user to the local Administrator group. This will allow the user to disconnect other users from the database started by AdminServer.
Check the required disk space on the server machine.
Check the parameters in session.pf (More information: Help > System management > Parameter tuning > session.pf)

Check the parameters in glimssv.pf, genrwsv.pf (More information: Help > System management > Parameter tuning > glims/genro/genrwsv.pf)

In Glims 9 we double the # of clients, because each client can have its own AppServer agent with its own connection to the database so:

-n >= (#clients+#services)*2+10

#clients <= (-Mn*-Ma).
When working on a SAN, check if there are no backups scheduled during the conversion, because this will affect the performance during the upgrade.
Before upgrading to production, export the ribbon configuration from the test environment.

Make sure the server is connected to the network with the first network adapter in the server.

This is to avoid connection problems with AppServer and translators.

Customer upgrading to GLIMS 9.X and using a G_LDT or G_MTS driver, should first upgrade these drivers to the most recent version:

  • G_LDT 3.13.0
  • G_MTS 3.4.0

Post-upgrade check list

After the major upgrade, GLIMS clients must not yet login!
Stop GENRO, GENRW + GLIMS databases

Databases back-uppen (offline)

Probkup glims glims.bck
After the installation of GLIMS 9.5.x, install the most recent service pack.

If the customer uses ODBC:

  • Set ODBC connections in the Openedge Explorer

  • Change Glimssv.pf and genrwsv.pf for use with ODBC

  • Perform grants on the databases. GENRW + GLIMS

    • sqlexp -H hostname -S service -db glims -user SYSPROGRESS -password SYSPROGRESS

      Hostname = Server name, service = port number ODBC

    • Remark, only the grants from the GLIMS8 folder are sufficient

    • Open the sqlexp via a dosprompt in the folder where the Grants are:

    • Once logged on, you can perform the grants by typing in @run grant8.sql.

Check the parameters in Progress.ini (Windows) and .glimsrc (UNIX).

Compare the new Progress.ini / .glimsrc with the one that was used!

Make sure that the environment variable MA_LC_ALL is NOT set in progress.ini or .glimsrc
Check the propath in the progress.ini.
Start the GLIMS 9 interface on the database server with administrator user account otherwise you cannot start or stop services and translators
The first post-conversion will be started as soon as a user logs on to the GLIMS 9 database.

In GLIMS, take the following actions:

  • Regenerate the menu (Start > System management > Menu > Regenerate)

    Note

    From GLIMS 9.6 on, menus are automatically regenerated during the migration to major and minor versions. This means that, when a version contains new functions, the corresponding new menu items are automatically added to the menus. Manual addition of new menu items to the menus (see chapter Menu customization) is only necessary when upgrading to a corrective version.

  • Synchronize sequences (Start > System management > Scheme > Sequences > Synchronize)

  • Check integrity of gp_Text and gp_SiteFunction (Start > System management > Scheme > Tables > Select table and choose Check integrity)
Stop databases to activate Afterimage. This can only be done after the upgrade, otherwise we risk problems with after image extents.

Start AfterImage at the new production environment. Remark, do not start the databases.

In glimssv.pf/ genrwsv.pf, check the AI parameters:

  • -aistall
  • -aiarcdir f:\glimsbck\ai
  • -aiarcinterval 900

 

Mark the databases as backed up.

  • Rfutil glims -C mark backedup
  • Rfutil genrw -C mark backedup
  • rfutil glims -C aimage begin
  • rfutil glims -C aiarchiver enable
  • rfutil genrw -C aimage begin
  • rfutil genrw -C aiarchiver enable

 

Start the database.

  • Proaiw glims
  • Proaiw genrw
  • rfutil glims –C aimage extent list
  • rfutil genrw –C aimage extent list

Import the ribbons from the GLIMS test environment

Before starting the import, do the following in the new database:

  • Delete all ribbons (4GL run procedure: GP_DeleteAllRibbonTabs)

  • Delete all privilege assignments

  • Delete all privileges

  • Delete all roles

 

Import the csv files with the script. Only create new entries, no updates for function parameter sets and query parameter sets to avoid problems between test - production !
Copy the translators from the Glims8\bin folder + check the JCE.

Check the options in progress.ini from GLIMS 8 – GLIMS 9
Import the license file
Start services and translators

In several places in GLIMS, you can use text modules containing HTML (externalization zones in browsers, data sheets, etc).

This HTML may refer to a fixed style sheet such as: <LINK REL="stylesheet" HREF="c:\glims8\Dat\Data-Sheet.css" TYPE="text/css">

After an upgrade to v9, this link may be broken. Solution: in the text browser, use the filter option to search for e.g. "glims8".

Check the communication services

Check backup & cleanup scripts,

There are MAM scripts for Windows. In the mam folder you can find how to use it.

Beware in progress.ini you will see:

GLIMSDBS=genro,genrw,glims

Change this to:

GLIMSDBS=glims,genrw,genro

Mam script uses this entry, this to avoid problems with sequences in the database .
When a virus scanner is installed, we advise to exclude the Glims directory. After an upgrade this must be checked as the directory name may have changed (e.g. glims9 i.s.o glims8)
Do not use interactive MISPL functions in the order entry option Evaluate after scheduling
The parameters of the MISPL function Order.PrintLabels have changed.
MISPL function CreateRequest should be replaced by MarkAsUnsollicited in the ResultOutput Triggers.
MISPL function: Specimen.LastRequest.Urgency changed to Specimen.Attribute("Urgency")
Activate Report builder service for the Report builder reports
Recover order entry layout from test environment
