GLIMS version terminology

MAJOR RELEASE - E.g. 9.0.0

Major scheme changes are involved, requiring a down time of the database and off-line conversion procedures to be run.

MINOR RELEASE - E.g. 9.4.0

No or only 'new-objects' schema changes are involved, enabling a customer to upgrade autonomously.

The 'auto db upgrade mechanism' will automatically modify the data model and trigger required conversion procedures. Down time should be minimal. The resulting database is backwards compatible with previous minor versions within the same major version.

Notes

About schema changes of type 'new objects'

If we limit schema changes to adding only certain types of new objects in the database, we have the following advantages concerning migrating from e.g. GLIMS 9.3 to 9.4:

  • Minimal down time, no index rebuild with database down is required.
  • Schema changes can in principal be done online, while the lab is working; best practice however is to bring lab activity down.
  • After migration, it is technically possible to revert the new software version. The migrated database remains compatible with the previous version. No need to restore a backup of the database.
  • Benefits on future migrations: by eliminating MIPS-defined site attributes, converting them to db fields is no longer necessary when doing future upgrades (major or minor). This reduces down time when doing major upgrades.

CORRECTIVE RELEASE - E.g. 9.4.1

Corrective version.

Note

We no longer talk about 'functional' or 'structural' version. What comes in which version (version number level), should depend on the amount of down time which is associated with migrating to the new version, and of course also the amount of risk the new version of the software introduces.

Schema changes are no longer done exclusively by major version upgrades. Some schema changes can be supported on a lower release version: minor release will from now on involve minor schema changes.
