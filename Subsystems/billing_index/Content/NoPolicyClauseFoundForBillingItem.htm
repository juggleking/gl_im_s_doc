No Policy clause found for Billing item

Verifying the log

Information found in the error:

  • Order set label (1)
  • Object external ID (2)
  • Billing code (3)
  • Lab, Department and Date of request that is linked to the Billing code (4)

This kind of error can have different causes. Here, we list the most common causes that you can check.

Checking the Order set

First you have to make sure that there isn't a mix of orders with different Encounter types in one Order set.

Navigate to the Order set which gives the error by using the query. You can find the label of the Order Set in the error log (1).

This query will give you a list of all the Order sets with this specific label. In this list, you will have to find the specific Order set linked to the Object in the error log (2). You will have to find the external ID of the Object in the Reimbursement party column.

Once you have found the Order set label with the correct external ID of the Object in the error, you can get a list of all the orders that are bundled in this Order set.

Some Order sets will only contain 1 order. In that case, the encounter type doesn't need to be checked.

If the Order set contains different Orders, you need to check if all the Orders have the same Encounter type (Ambulatory / Hospitalized). You can't have a mix of different Encounter types in one Order set.

If you find a mix of Encounter types, you will need to split them up under 2 different Order Sets.

The first thing you can do is a reevaluation of the Order set label:

This should spread the orders among different Order sets with the correct Order set label, depending on the Encounter type.

If this action hasn't resolved the issue and the different Encounter types can still be found in the same Order set, you can split them up manually by adjusting the Order set label on the Order with the different Encounter type. This way, the Order with the different Encounter type will be placed in this new Order set.

Example:

If the Order 511-005355 needs to be unlinked from the Order set, the label will need to be adjusted too, for example, 20110629_H_bis.

This way, the Order 511-005355 will be linked to this new label. The other Orders will stay linked to the original Order set and will have the original Order set label.

REMARK

An Order gets an Order set label from the moment it is created in GLIMS. The Order set label linked to the Order is a collection of Orders for one Object and one Encounter type for the same day.

This means that when you adjust the Encounter type in one Order of the Order set after an Order has been created, there will be a mix of Encounter types for the Order set label and the Order set will be incorrect.

When you adjust the Encounter type on an order you can start the reevaluation of the Order set label manually as described above, but normally you will have a 'TAR0'-task that will do this reevaluation automatically on a daily basis.

Step 3 - Determining which Clauses are missing

To determine which clause is missing you need the information that you gathered before:

Example:

a) Reimbursement class: KR

b) Encounter type: Hospitalized

c) Policy: KGR-ZVK

d) Subcontracting yes/no: yes

We suggest to get the list of clauses starting from the Policy, as this usually is the most manageable list.

In this list, you can find the Reimbursement class, linked to the clause. For this example, only the Clauses linked to KR are interesting.

Now we need to look at the linked Encounter type of these Clauses.

The column 'R.T.' stands for Residence type. This can be A (Ambulatory) or H (Hospitalized). The Label of a Clause can also give you an indication of the Encounter type.

The Clauses that have a label with _H, will be clauses that can be used for Encounter types that are 'Hopitalized', these with _A can be used for Encounter types that are 'Ambulant'.

You can also check this by opening the clause.

For this example, only the Clause for Residence Type 'Hospitalized' is interesting.

The column 'Subc.' stands for Subcontracting. This can be yes or no. You can also check this by opening the clause.

On this Clause we now need to check if Subcontracting is checked. This is not the case, so the Clause won't be available for this specific situation.

REMARK:

If the correct Clause seems to be present, make sure the 'To charge' toggle is checked on this specific Clause.

By doing an 'INSERT' in the list of Clauses, a new Clause can be added on the Policy or Reimbursement class. Normally, Clauses for 3 kinds of Reimburser types: Fund (invoice for Fund), Object Person (invoice for Patient), Dispatching lab (used for Subcontracting) can be found.
