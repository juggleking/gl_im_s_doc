Order series

Introduction

GLIMS features a dedicated program to generate a set of orders.  It can be accessed from the main menu via Start -> Routine -> Orders -> Create series. 

Main page

General order info

Study

Allows selecting a study for the orders.  If the study correspondent (or its issuer, or its responsible) is a known PIN provider, it will be automatically used for subsequent PIN selecting.  If the study has a default request definition, it will be proposed by default.

Order count

Allows specifying the number of orders to be generated for each selected patient (see "Patients" page).

PIN source

When selecting patients by PIN (in the "Patients" page), a browser will be displayed showing the patient identifications issued by this PIN provider.

Request, object time, issuer, agent and department for each order  

Just as in the manual order entry program, the issuer and agent can be selected via their identifications.  It is possible to specify a single request or a multiple request list like "Na,k,Cl".

Just like in the manual order entry program, the order series generator allows typing Ctrl-F in the request list in order to add requests to the existing list by selecting one or several request definitions from a popup selection browser.  Note that, as opposed to the PIN selection browser or the Code Scan screen, the already entered request mnemonics are not pre-selected in the browser: the browser is only used to ADD request mnemonics to the list.

Post-commit

Allows specifying an order-based, string-valued site function to be executed after the order has been created, just like the manual order entry program.

When this site function returns a non-empty string, it is considered to be an error message and will be shown as such to the user. Note that this error message does not prevent the order from being written to the database.

This post-commit site function, as well as the PIN provider, Order Short Id post-processing site function and the person selection method is stored in the user preferences for later reuse.

Short id layout

The layout of the order short id can be specified using the following fields: prefix, suffix and start from. For instance, if prefix is 'ORD', suffix is 'a', start from is '0' and order count is '100' then 100 orders will be created with short id's from 'ORD00a' to 'ORD99a'.

The Post-processing function can be used to add a check digit to the short id.  The following list of order data can be used in the function:

  • The object
  • The issuer
  • The agent (if supplied in the user interface)
  • The department (can be the department from the user interface or the user's department if none supplied in the user interface)
  • The lowest object time
  • The order short id

Other options

Root and work specimen labels

This option is saved in the user preferences.  Labels of orders are printed after a series of orders has been created. This means that all labels for an order series will be printed consecutively.   When labels have to be printed, the order label is printed first.

Confirm specimens

Allows the user to decide to apply automatic root specimen confirmation or not.  This option is saved in the user preferences.

Patients page

This page allows selecting the patients for whom orders are to be created.

Selection

Indicates how you want to select your patients.

Select persons

This mode allows selecting patients by using the fields:

Add person

Uses the person search screen.

Add PIN

Select a patient through its PIN (PIN provider is set in the main page).

Add group

Allows selecting a correspondent group: only members of type "Person" will be added.

Use dialysis encounters

When set, the encounter referred to by the field Person.DialysisEncounter is used as the new Order.Encounter. Persons without such an encounter are ignored.

Scan codes

You must first enter a last name and then scan the codes which will be used as first name.  The program checks if a patient with this last name and the scanned code as first name is already known. If so, the patient is reused. If not, a new patient is created.  Additionally, during the scanning of the codes, a visual indication is added to already existing patients to inform the user of this fact.  The maximum number of different codes that can be scanned is 2000.  Identifications containing a comma will not be accepted.  

Person browser

Shows all selected patients for which orders will be created.  Deleting patients from the selected list is possible by using the <Delete> key in the browser.

Person series

GLIMS features a dedicated program to generate a series of persons.  It can be accessed from the main menu via Start -> Correspondents -> Persons -> Create series.  The following screen will be displayed:

The layout of the name can be specified using the following fields: prefix, suffix and range (from - to) and first name. For instance, if prefix is 'series', first name is john and from is '0' and to is '99' then 100 persons will be created with names from 'john series00' to 'john series99'.

Optionally, you can specify an object template, the person's birth date and sex.

The fields Encounter type, Institution and Maximum validity date are used to automatically create encounters for the new persons. When the institution has an Encounter ExternalId MISPL, an encounter is automatically created for that person, using the specified encounter type and validity date. The EncounterExternalId MISPL of the institution is used to determine the external id.

When the institution has an identification code (via Institution.Correspondent.IdentificationCode), an identification is automatically created, using the specified validity date and the identification code.

When an existing person is found with the same last name, first name and birth date as for the series to be created, a warning is given.  However, the user can still accept and the series will be created anyhow.  

Note

You risk to bump into the maximum amount of duplicate lastname/firstname/birthdate combinations when using this frequently. This amount is high, but limited. The discriminator will count from "0" to "ZZ". After "ZZ", you will no longer be able to create a person with that lastname/firstname/birthdate combination.
