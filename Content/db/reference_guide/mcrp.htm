This table stores the definition of microbiology procedures on a material.

Fields

Properties > Conclusion property

This field contains the property used to store the conclusion of the examination procedure. Can be of type enumerated, string,... The conclusion property allows to put the conclusion text (specified in the microbiology screen) on the report. By including this property in a property classification, you can determine where the conclusion should be put on the report.

This is a reference to the input material, i.e. the material of the specimen used for this procedure.

A microbiology procedure is valid for one particular Material. Note that you must specify a root material, i.e. a material which is not derived from a parent material.

Properties > Microscopy property

The Microscopy property (Property MicroscopyProperty) is the property used to represent the microscopy phase. Specifying this property causes a microscopy phase to be added to the other phases, under the condition that there is either no microbiology request code for the specimen or that for such a microbiology request code the flag "Add microscopy" is OFF.

Negative > Negative answer > Final negative answer

This string will be printed on the report if the MB-action has been confirmed or validated and is considered as finished while no (reportable) isolations have been detected (negative cultures).
It is valid for status 'Ready' and 'Closed', but will not be used for a lower status.
For preliminary answers use the field 'Prelim negative answer' instead.

This field is also MISPL-enabled (based on the current microbiology action).
Example:

The exam was conducted for {= ToString( .Procedure.IncubationPeriod, "%x" ) } days, but no growth has been detected.

Properties > Report property

The report property should always be specified. If not, the result of the corresponding microbiology action will not be displayed on the report. The property report is a dummy property which must have the data type Culture! By including this report property in a property classification, you can determine where the microbiology results should be put on the report.

Property ReportProperty

Seq no

If specified, microbiology actions will be sorted first by this sequence number. This applies to:

- The microbiology screen: sorting of microbiology procedures first by sequence number, then by creation order (sequence in which requested);
- Specimen.Attribute("MicrobiologyProcedure...List") and Order.Attribute("MicrobiologyProcedure...List") MISPL functions
- Microbiology statistics that classify by microbiology procedure (in rows/columns/tables).

Main > SOP

The SOP is a symbolic reference to an external document describing the associated Standard Operating Procedure.

The generic Edit menu contains an option "Hyperlink". Activating this option will cause the focused widget to be scanned for Hyperlink information. If any hyperlink information is found, the Hyperlink will be followed. Hyperlink information conforms to the following syntax:
HyperlinkInformation: Link=(<LinkInfo>)
LinkInfo: <ClassExternalizationInfo> | <WindowsApplicationInfo>
ClassExternalizationInfo: Method=(ClassExternalization | CX) (Class | C)=<ClassName> (Externalization | X)=<Externalization>
WindowsApplicationInfo: Method=(WindowsApplication | WA) (Application | A)=(Word97 | InternetExplorer) (Document | D)=<URL>

Example: Link=(M=WA A=Word97 D=(C:\Standard Operating procedures\SOP0001.doc))
Example: Link=(M=CX C=Order X=(19980801-00001))

String SOP

Main > Specimen scope

The scope of a specimen defines by which procedures this specimen can be used. If specified, the specimen can only be used by procedures having the same scope. If an order requires 2 procedures for the same material but with a different specimen scope, 2 specimens will be scheduled, each destined for its own procedure.

Exemple:

If you request two microbiology procedures (or a combination of regular procedures and microbiology procedures) which would normally be able to use the same specimen, you can indicate that each procedure requires its own specimen by defining a different specimen scope for each procedure.
