Logging > Log file keep

"File keep and File keep unit are used together to indicate after how long a period log files should be automatically deleted. If not specified, you will have to delete the old log files manually. You can't delete a log file of a running queue processor.

PositiveInteger LogFileKeep

Logging > Log file keep unit

"File keep" and "File keep unit" are used together to indicate after how long a period log files should be automatically deleted. If not specified, you will have to delete the old log files manually. You can't delete a log file of a running queue processor.

Mandatory LoggingTimeUnit LogFileKeepUnit

One of: Hour, Day

Logging > Log file max size

Sets the maximal size (in MB) of the log file.

PositiveInteger LogFileMaxSize

Logging > Log file refresh

"File refresh" and "File refresh unit" are used together to indicate after how long a period open log files should be closed and a new log file should be opened. The actual log file names will be different for each new version by suffixing them with a creation time stamp.

PositiveInteger LogFileRefresh

Logging > Log file refresh unit

"File refresh" and "File refresh unit" are used together to indicate after how long a period open log files should be closed and a new log file should be opened. The actual log file names will be different for each new version by suffixing them with a creation time stamp.

Mandatory LoggingTimeUnit LogFileRefreshUnit

One of: Hour, Day

Logging > Log file refresh on max size

Determines what happens when the maximal log file size is reached. If set, a new log file is created. Else, the queue processor exits to prevent disk overflow. In such case, the associated message queue service is informed and alerts all users of type "system manager" by email.

Mandatory Automatic Logical LogFileRefrshOnMaxSiz

Login user

Is a reference to the user on account of which the queue processor will run. Double click the field to select a 'batch' user. Batch users are users of type 'System manager', with session type 'batch', 'Auto login' enabled and 'Enabled'='Yes'.

Mandatory Invariant sc_User LoginUser

Logging > Log verbosity

Is either "Low", "Medium" or "High". Use it to set the amount of logging detail. We suggest setting this to "Medium". Info, warning and error messages are always logged, independent from the log verbosity. The log verbosity levels offer the following options:

Low: TCP/IP communication not logged
Medium: TCP/IP communication fully logged
High: byte per byte logging (for debugging purposes only)

Mandatory Verbosity LogVerbosity

One of: Low, Medium, High

Message queue

Is a reference to a service of type "Message queue".

Mandatory Invariant gp_Service MessageQueue

Name

Is the identification of the queue processor. It should not contain any characters other than letters, digits and underscores. The first character must be a letter or an underscore.

Mandatory Unique Invariant Mnemonic Name

Port

Should be any unoccupied TCP/IP port on the server. Do not use port numbers smaller then 1024, because they are reserved for system processes; we recommend using numbers between 3100 and 4000.

Mandatory Invariant String Port

Process id

Set automatically when the queue processor is started.

Automatic String ProcessId

Seq no

Determines the relative sequence in which queue processors associated with a given Message Queue are started and stopped by the services startup and shutdown functions, respectively. Shutdown uses the inverted order.

PositiveInteger SeqNo

Synchronize 

Synchronizing a queue processor means setting the parameters of the running queue processor equal to the queue processor parameters specified in the database. Use Synchronize from the queue processor contextual menu. The synchronize dialog allows you to specify whether you want synchronization of logging setup, interface setup, or both.
