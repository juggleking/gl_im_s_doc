Label

This field is used during the assignment of orders to order sets. This assignment is based on the uniqueness of the combination of order set label and reimbursement party. Hence, orders of the same reimbursement party (e.g. patient) that need to be tariffed together need to specify the same label. Examples:
In Belgium; all orders, prescribed on the same day of an ambulatory patient need to be tariffed simultaneously (for supplement calculation). This can be achieved by spcecifying the presription date as order set label.
In commercial environments payment agreements may span several orders. If these agreements need to be detected automatically, all the related orders need to be tariffed simultaneously.
If order grouping is not required, specify order internal id as order set label.

Reimbursement party

This field provides access to the order set's reimbursement party. It is a party that is related (cfr party type) with the associated order(s) and that is principally responsible for the reimbursement of the costs. This does not mean that this party has to pay! The payment agreements applicable for this order(set) are either general payment agreements, or payment agreements of the reimbursement party. Examples:
In health care environments the reimbursement party is usually the patient. This means that invoices will be sent to the patient's fund or to the patient if the patient is not insured or if his fund fails to pay.
In commercial envirionments the reimbursement party may be the issuer. For vetrinarian orders, the reimbursement party may be the issuer (the vet) or the animal owner.

Mandatory Invariant Correspondent ReimbursementParty

Specific party

Correspondent, assigned as specific reimbursement party. This field is only relevant if the order set's party type is set to 'Specific'. This field was introduced to relate a correspondent with the order(set), which is not involved with the order for other reasons. In this way that correspondent's payment agreements can be applied.
Example: In Germany the 'Rechnungsempfanger' is implemented in this way.

BCIData 

This function returns data (specified by a dedicated parameter) of billing code instances (BCI) that meet all the selection criteria. Billing code instances are temporary records during a tariffication session that correspond with future billing items. Hence this function is only available during the tariffication program. This function is frequently used for supplement calculation.

Billing code group mnemonic. If specified, restrict billing code instances for which the billing code belongs to the specified group.

Parameter specifying which data needs to be returned of the selected billing code instances:

- PriceTotal: The amounts that are summed are the prices calculated by the price formule.
- GrossAmountTotal: The amounts that are summed are the gross amounts (exclusive VAT and discount) of the invoice items (billing assessments) of the selected billing code instance.

- MainNomenclatureCode: Nomenclature code with the highest base value.
- BillingCodePanelOrPropertyDescriptions: returns a list of descriptions (only applicable during the order set tariffication such as in the policy supplement expression). The following descriptions are returned in descending preference: the BillingCode.ExternalDescription, the panel description or the Property / Material / Bloodproduct / MicrobiologyProcedure / Pathologyprocedure description.
- ExecutingDepartmentList: Comma seperated list of executing department(s) mnemonic(s)
- ExecutingLabList: Comma seperated list of executing lab(s) mnemonic(s)

FrenchSupplements 

This function calculates the official french supplements
('Majorations', 'Prélèvements', 'Déplacements') for labs
and based on the order information (sampler, distance,...) and charged billing codes (tariffication before evaluation of the supplement expression).
The return value is a comma seperated list of billing code codes (with each an optional \Reason tag).
IF no supplements are to be applied, the return value is an empty string ''.
This function can be used (only) in the Policy.Supplement expression and can reduces the expression to:
RETURN(.FrenchSupplements(?, ?));

conventions:
-The calculated supplements are conform the table 'Tariffs france metropolitaine' (not tariffs Dom-Tom)
-The format of the returned billing codes is NNNNCle; example 9050K
in case of possible ambiguety, the factor is added too; example 9050K[1.5/2]
these codes should be available (preconfigurated) - note that most of them are not included in the nabm nomenclature file(import)
-the 'Prelevements' are based on the Material.SamplingCode:
PS = 'Prelevement sanguin (par ponction veineuse directe)' (multiples and babies<5y are automaticly recognized)
PGY= 'Prelevements gynecologiques'
PPM= 'Prelevements asceptique musqueuses et peau'
PIA= 'Ponction Intra Arterielle'
PIV= 'Injection Intra Veineuse'
PCU= 'Cathererisme Ureal femme'
PTG= 'Tubage Gastrique'
-The 'Nuit', ferier,.. urgencies are based on the Order prescription time (unless explicit specified in the Options parameter)

String FrenchSupplements (String Types, String Options)

String Types
indicate which supplements should be calculated.
Possible values are:
? all
P 'Prélevements' (AMI, TB, KB, PB, K)
M 'Majorations' (Nuit, Dimanche, B4 sanguin, B5 bacterio)
D 'Déplacements' (IK,ID)
or a combination of these values; for instance "P,M"
String Options
\UrgencyReferenceTime=ObjectTime|PrescriptionTime
Reference date and time used to determe the night, holiday supplement
The default behaviour is PrescriptionTime
