Value entry > Allow general result codes

Indicates if the non-property specific result codes (i.e. the general result codes) can be used for this property.

Mandatory Logical AllowGeneralCodes

Value entry > Allow multiple result codes

When this field is set, the results of this property can contain multiple result codes. This means that every word of a result is checked to see if it is a result code. When a result contains multiple result codes, the highest result code severity is used to set the result severity. The other fields of the result codes (Action, Assessed, To charge) are not taken into consideration when there is more than one result code.

Value entry > Datatype

One of the following options:

- Numeric: if the property is numeric, a unit must be specified
- String: text
- Enumerated: if enumerated, it must have an associated list of valid values (see choice lists)
- Titer: display 800 as 1:800. Customers preferring an alternate character (e.g. "/", like in "1/8") should specify their choice in the SpecificSite.TiterSeparator field. Leaving this field unknown ("?") assumes the default ":" character. For performance reasons, the field is fetched only once, when first needed. An update will become effective after a session exit & re-login only. Properties of data type 'Titer' are always treated as standard numeric results in GLIMS. The prefix is only added for visualization in result browsers or when reporting. This means that:

- A titer property can be used as input for a formula (in PropertyOutput.Value). The property mnemonic will now expand to its numeric value instead of "?".
- When updating a titer result in the confirmation/validation browser, the result cell will change from e.g. "1:800" to "800" on entry.

- Culture: indicates that this property is used to generate a microbiology culture result, starting from the isolations referring to the actual result record.
- Antibiogram: will be used to indicate that this property is used to generate a microbiology antibiogram result, starting from the antibiotic results referring to the actual result record
- Image: indicates that the results of the property will be images
- Graph: indicates that the results of the property will be graph data
- Foreign: properties of which the results do not belong to one of the above-mentioned data types

Mandatory PropertyDatatype Datatype

One of: Numeric, String, Enumerated, Titer, Culture, Antibiogram, Image, Graph, Foreign, Genetics, Genetic List

Value entry > Informational (entered during order entry)

Indicates that the result of such a property is considered to be information passed on during order entry, without being assessed in the laboratory. Such results are considered to be 'secondary' results: they are never considered as 'new' for a report, are not a trigger for the scheduling of a report and do not influence the status of an order in the same way as a regular result. The result will automatically be validated if no assessment method can be found (so result origin is unknown).

Report > Report format

The report format determines how the results will be displayed on reports.

The option 'Text' means "Starting from the Value column, until the page border". This means that there is no more space for any subsequent column.

When this field is set to "Hidden", then the actual result is completely hidden on the report (no line for this property is generated). The main purpose is the possibility to steer the automatic printing and completion status of the report via a property, which itself is never printed on the report.

Scheduling > Use available specimen

allows to indicate that this property should only be scheduled on an explicitly confirmed specimen. This will typically be used when the test can be derived from several materials, and the scheduler should wait until a suitable material is confirmed for this order.

In this context, 'explicit confirmation' can be obtained in the following ways:

- explicitly requesting the specimen in order entry
- adding a specimen automatically via the regular scheduling while AutoSampleLogin is active
- scanning and either creating or confirming a specimen via the specimen receipt program
- scanning a specimen in the specimen scan or work list generation program
- reacting to a query or result upload message from an analyzer or sample distributor connection

When a specimen is explicitly confirmed after order entry, and the order still has at least one non-scheduled result, the action scheduler will try again to schedule the non-scheduled tests.

Mandatory Logical UseAvailableSpecimen

Replace 

The goal of this function is to replace all request definitions referring to the original property with request definitions (= requestables) referring to the newly chosen property. This is done by closing the existing request definitions using the specified date and creating a new request definition referring to the new property with a start date equal to the specified date. This ensures that from that date on, all usage of the original request code will result in the new property being requested rather that the old property. The newly created requestable inherits all tariff and accounting class records of the original request definition, as well as its requestable variable settings.

The end effect is that the users can keep using the old request codes, while the system actually requests the new property instead.

MultiRecord Void Replace (Property NewProperty, Date Date)

Property NewProperty
The mnemonic of the new property, which replaces the old one.
Date Date
Date that will be used to close the existing request definitions and start the new request definitions.
