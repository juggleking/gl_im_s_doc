End

End date+time of the approach, depending on the status:
- As long as the start date is empty (approach status is ‘Initial’), the end date is also empty.
- If it is not finished yet (status ‚Running‘), it contains the expected end date.
This expected date is calculated as:
Start date + Number of days TAT of the approach plan (config).
- If the approach is finished, it contains the effective date that the end node was reached.
- If the approach status is Discontinued or Error, the end date is also empty.

Specimen Specimen

Start

This is the date+time that the approach was actually started to be executed. Or more accurately, date-time on which the first (classic) Action started.
The execution of the approach can be delayed.
e.g. when created on a Friday but the execution should start on Monday.
How is this start marked in Glims?
As soon as an Action is created for a certain Activity in the approach, and this Action is ‘started’ (i.e. gets a start time), then the Approach record gets a start time.

DateTime StartTime

Status

Initial: approach was created but execution still has to start.
Note: will never exist in reality, because plan immediately starts. Foresee this status anyway.
Running: execution has started and is running, but not finished yet. So at least 1 activity in the Approach has status Running.
Error: One of the contained activities has run into an Error status.
Finished: Execution of the approach has finished (end of the plan has been reached)
Discontinued: e.g. due to discontinue of the order

Mandatory ApproachStatus Status

One of: Initial, Running, Error, Finished, Discontinued

GetVariantResult

Returns a reference to a specific variant result of the order

VariantResult GetVariantResult (String VariantName, VariantResultStatus MinimalStatus, Integer MinimalSeverity, Integer MinimalClassification, VariantRetestStatus RetestStatus, Integer Index)

String VariantName
Only VariantResults will be selected where VariantResult.Variant.Name = VariantName if <> ?, otherwise the VariantName is not taken into account.

VariantResultStatus MinimalStatus
One of: Available, Confirmed, Validated, Discontinued

Only VariantResults will be selected where VariantResult.Status is at least this status; if not specified, VariantResultStatus["Available"] is assumed. Previously this was VariantResultStatus["Validated"]. This means that customers that depend on the old behavior by specifying ?, must explicitly state VariantResultStatus ["Validated"] in their current MISPL.
Integer MinimalSeverity
Only VariantResults will be selected where VariantResult.Severity >= minimal Severity.
Integer MinimalClassification
Only VariantResults will be selected where VariantResult.Classification >= minimal classification; when empty, 1 is assumed.
VariantRetestStatus RetestStatus
One of: Not requested, To do, Success, Failed

Only return variant results which have the same retest status. If unknown is provided then this filter is ignored.
Integer Index
This is the only mandatory parameter which must be <> 0. This is a loop counter variable:
- when 1, the first variant result is returned (in sequence of field availability time)
when 2, second result etc
... until the return value becomes unknown.
- when -1, the last variant result is returned (in sequence of field availability time)
when -2, the one but last result is returned etc
... until the return value becomes unknown.
