Compatibility > Only autologous units

A person can have autologous blood bags. Those blood bags must be transfused to that person. Yet, if more blood bags are required than the number of available autologous blood bags, one has to decide whether or not other (non-autologous) blood bags will be transfused. This decision however is site-dependent. This toggle disables the possibility to transfuse other (non-autologous) blood bags to an autologous person.

Compatibility > Check bag's antigens against patient's antibodies? > Check antibodies<->antigens

If this option is enabled, only blood bags will be selected of which the antigens have the opposite value of the corresponding antibodies in the patient. If a conflict is found, the "Matching score" of the blood bag can be influenced by the score mismatch factor that is defined in the antigen configuration. This allows completely rejecting the blood bag (score factor = 0) or just lowering the matching score.

Do not temporarily switch off this toggle to circumvent the built-in security! (MIPS cannot be held responsible in that case!)

Compatibility > Check bag's antigens against patient's antigens? > Skip check for person positive for the antigen

This option is only applied when the option "Check antigens <-> antigens" is enabled.
If this option is enabled and a patient is positive for a specific antigen, then no mismatch will be registered. This also implies that the antigen-antigen mismatch score defined for that antigen will not be applied. The blood bag will be considered compatible for that antigen.
All other values for the availability of a person antigen (Negative or Unknown) can result in an antigen mismatch. If a patient is negative for an antigen and the blood bag is positive for that antigen, an antigen mismatch will be detected and the corresponding antigen-antigen mismatch score will be applied.

Do not temporarily switch off this toggle to circumvent the built-in security!
(MIPS cannot be held responsible in that case!)

Mandatory Logical NoCheckForPosAntigen

Bag receipt > (for future use)

This trigger is executed after a new blood bag has announced (electronically) before it's presence has been confirmed.
With the blood bag diskette program, this trigger will run as soon as the diskette file has been read.
With the 'blood bag scan' program, announcement and presence confirmation happen together, so this trigger will be executed at the same time as (but just before) the 'When received' trigger.
If an order has to be created also for this blood bag, this trigger runs BEFORE the order is created.
If the trigger returns 'FALSE', the blood bag will not be stored in GLIMS, so this offers a way to check things before blood bags are stored in GLIMS.

gp_SiteFunction TriggerWhenAnnounced

Triggers > When checked

This trigger runs after a blood selection (and hence also the blood bag) has reached the status "Checked". This can happen in 2 cases:
1. a blood bag is selected for a product that doesn't need a crossmatch
2. a blood selection is promoted to "Checked" because of the trigger of a crossmatch result.
This can be useful e.g. to print a transfusion form.
Caution: the result of this function (TRUE or FALSE) will be ignored.

Triggers > Utmost transfusion time

MISPL to update the utmost transfusion time of a blood selection.
This field is a blood selection based MISPL of datatype DateTime, to calculate the utmost transfusion time.
By default the utmost transfusion time is initialized at order entry as the due date/time. If it is not updated to a later time, the transfusion cannot take place.
So it is updated by the return value of this MISPL function. This is done at the moment when the blood bag reaches status 'Checked', which is
- either after a successful crossmatch
- or after selecting a bag for which no crossmatch is required.

The new utmost transfusion time can be based on various attributes of the blood selection, the order, etc..

Example:
RETURN .Order.ReceiptTime + 2; /* two days from the order date onwards */

Example:
RETURN DateAndTimeToDateTime(Today(), Now()) + 3; /* three days after the moment of selecting a blood bag */

Remark: it is still possible to update the utmost transfusion time in the selection trigger or in the crossmatch property trigger. Don't use more than 1 possibility together, otherwise the utmost transfusion time will be updated more than once.

gp_SiteFunction WhenAntibodiesFunc

Compatibility > Check bag's antigens against patient's antigens? > Check antigens<->antigens

There are situations where a patient has no antibodies (yet), but the presence or absence of the corresponding antigen is known already (positive or negative).
For certain patients it can be desirable in that case to select only equal blood bags, where this antigen is also present or absent (same as in patient).
So if patient Kell = neg => blood bag Kell must be neg
if patient Kell = pos => blood bag Kell must be pos

This MISPL function decides whether -for the actual patient- the system should perform this (built-in) check to make sure that the bloodbag's antigen pattern is exactly the same as the patient's antigen pattern.

This check can be switched on or off using the previous toggle.
When the toggle is ON, you can use this MISPL function to refine the conditions in which the check should be performed.
When the toggle is ON but no function is specified, the default is 'always'.
