String Description

Host

Specify the name of the Windows computer where the Service has to run in Host. This can be the Windows computer where you are currently logged in or any other Windows server or client computer. The only restriction is that you have to be logged in currently as a user with administrator privileges on the computer you specify. Normally, you will run all Services on the server computer. In that case, you can leave Host unspecified. The application will then take the value of MA_DEFAULTSERVICEHOST in progress.ini, which is the name of your server computer.

Is either "Low", "Medium" or "High". Use it to set the amount of logging detail. We suggest setting this to "Medium". Info, warning and error messages are always logged, independent from the log verbosity. The log verbosity levels offer the following options:

Specific options

Extra service specific options. They are specified as a list with format { -<name> [ <value> ] }.

String SpecificOptions

Type

Is either "Translator server" or "Task scheduler".

Translator servers are used to exchange information between the database and one or more translators (for performance reasons, a maximum of 2 is recommended).

Task schedulers are used to carry out scheduled tasks. Note that, when you define only a single scheduler, you define only a single queue for task execution. This implies that tasks may effectively execute later than their scheduled time. If you have sufficient resources, you can also define several task schedulers.

Mandatory Invariant ServiceType Type

One of: Translator server, Task scheduler, Service controller, Report builder, Message queue, Java communication engine

Register as Windows service 

Before a Service can be started, it has to be registered with the Windows operating system. Use the Register As Windows Service function in the context menu of the Services browser or editor. A dialog appears to specify a Windows user account that will be used to run the Service.

You need to register a service only once. When the service is registered, it is visible in the Windows services window.

MultiRecord Void RegisterAsWindowsService (String UserLoginName, String UserPassword, Logical CopyWindowsExecutable)

String UserLoginName
In the User login name field, you can enter the user account that should be used to run the service. For Services, you should enter "LocalSystem". For Translators, use a real user name in the format <DomainName>\<LoginName> if the application is running in a domain, or else <ComputerName>\<LoginName> (e.g. GLIMS\Administrator).
String UserPassword
Should be set to "LocalSystem" when you are running the service with user "LocalSystem", or use the matching Windows password when you are running the service using a real user account.
Logical CopyWindowsExecutable

Start 

Choose Start from the service's contextual menu to startup the associated process.

MultiRecord Void Start (ServiceDependingTranslators StartDependingTranslators)

ServiceDependingTranslators StartDependingTranslators
One of: None, Autostart, All
Possible values for Start depending translators are: "None", "Autostart" and "All". This determines which, if any, depending translators are started immediately after the service. This can reduce the amount of user manipulations required to start translator servers and their associated translators. In case of "Autostart", only translators marked as such will be started along.

Start profiler 

This function requires minimal user type 'Developer' and should therefore not be executed without the assistance of a MIPS employee.

As user of type "Developer", choose Start profiler from the service contextual menu to activate Progress profiling of the service's process. Profiling generates a report with data which may be useful for pinpointing performance bottlenecks in the code. Developers can load these reports into the Progress profile info viewer for further analysis.

MultiRecord Void StartProfiler (ServiceProfilingScope Scope, Integer MinimalProcessingTime)

ServiceProfilingScope Scope
One of: Session, Detail
Is either "Session" or "Detail".

When "Session", one single report (in file <LogFileName>_<Date>_<Time>_S<SeqNo>.out) will be generated, scoped to the entire process.

When "Detail", one report (in <LogFileName>_<Date>_<Time>_D<SeqNo>.out) will be generated per message processed (for translator servers) or task executed (for task schedulers).
Integer MinimalProcessingTime
Is expressed in seconds. It is taken into account only when Scope is "Detail". It constitutes a criterion for messages or tasks to be eligible for report generation. If they require less than the specified amount of time, report generation is skipped.
