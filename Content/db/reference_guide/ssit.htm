Billing > Functions > Bank account check

This field references a temp-table (w_BankAccount) based, character-valued MISPL expression: it runs whenever a bankaccount (optional with bankid) is inserted/modified and checks on valid control digits and format.

This function can return an empty string (indicates a valid check), a tagged list (contains reformatted values) or an error text : if a backslash separated tagged list is returned, then the following format is expected : "BankAccount=<account>\BankId=<bankid>"

When correspondent-data is inserted/modified and the bankaccount/bankid are inputted the MISPL runs : invalid bankaccount/bankid are rejected, an error text is displayed and the user can correct the inputted value; reformatted values (taggedlist returned, reformat conform expected format) are accepted; an empty string indicates valid data, the inputted values are accepted.

When payments are imported, the bank account is checked running the MISPL :
when the check is not OK, the import does not interrupt, but a clear warning is logged; if reformatting an account was necessary and possible, then the reformatted value (out of returned tagged list) will be accepted as input.

General > Labels > Bar code parser

For pratical usage, see tutorials: Basic routine -> Specimens -> Specimen and order search via barcode. The goal of the function is to detect what bar code was scanned and to return this information in a so-called tagged list. Example: when scanning 0023234201, this may result in a return value like
"Specimen=ExternalId=23234201~\Order=ShortId=23234-2"

This return value can be created by using the PutTag MISPL function:

STRING SpecimenInfo;
STRING OrderInfo;
STRING ReturnValue;

ReturnValue := ?;

IF LEN(.BarCode) = 10 AND SUBSTR(.BarCode, 1, 2) = "00"
THEN
SpecimenInfo := PutTag("ExternalId", Substr(.BarCode, 2, 8), SpecimenInfo, "\");
OrderInfo := PutTag("ShortId", Substr(.BarCode, 2, 5) + "-" +
Substr(.BarCode, 7, 1), OrderInfo, "\");
ENDIF;

IF Specimeninfo <> ?
THEN
ReturnValue := PutTag("Specimen", SpecimenInfo, ReturnValue, "\");
ENDIF;

IF OrderInfo <> ?
THEN
ReturnValue := PutTag("Order", OrderInfo, ReturnValue, "\");
ENDIF;

General > Results > Entry > Choices have precedence over result codes

By default, any result entered is first checked against the result codes valid for this property. This has the consequence that result codes have precedence over property choices of enumerated results.

Although we believe this is the best approach, the SpecificSite.ChoicesHavePrecendence option allows that the result is first checked against the property choices for an enumerated property. When the entered result matches with a property choice, it is selected and the result code table is not checked any more. Note that this option only works for properties of datatype enumerated. The result code selector (accessible by hitting CTRL-F in a result entry field) takes the precedence setting of choices and result codes into account.

Please consider carefully the possible side effect before activating this option! Remember that for instance a choice match is detected also when the entered result unambiguously matches the start of one of the choices, while for a result code an exact match is required.

Mandatory Logical ChoicesHavePrecedence
