Status > Completeness time

This field allows to indicate whether the report is an interim report, a complete report, a corrected report or a duplicate report. It stores the date and time upon which this report was first generated as a complete report.

During text report printing, this field aids in deciding the report status:

- Interim: an incomplete result was detected,
- Complete: the decision whether a report is considered to be complete, depends on two default report fields:

*DefaultReport.CompletenessStatus: when one of the results has a status lower than the CompletenessStatus, this result (and the report) is considered as incomplete.
*DefaultReport.MBCompletenessStatus: when the microbiology action of a culture result has a status lower than the
MBCompletenessStatus, this result (and both its culture result and the report) is considered as incomplete.

When these values are not specified, the following defaults apply :

* For 'status' reports, CompletenessStatus is set to 'Validated' and MBCompletenessStatus is set to 'Closed'
* For 'value' reports with MinimalResultStatus = 'Validated', CompletenessStatus is set to 'Validated' and MBCompletenessStatus is set to 'Closed'
* For 'value' reports with MinimalResultStatus = 'Confirmed', CompletenessStatus is set to 'Confirmed' and MBCompletenessStatus is set to 'Ready'
* For 'value' reports with MinimalResultStatus = 'Available', CompletenessStatus is set to 'Available' and MBCompletenessStatus is set to 'Ready'

- Duplicate: complete and Report.CompletenessTime was already filled in,
- Modified: complete and completeness time filled in and a new result was detected.

This status keyword is passed as variable with name "ReportStatus" to the expansion of the report headers and footers. There, you can print it directly on the report, or examine it via MISPL. Example:

Status:{= ENTRY(LOOKUP("{ReportStatus}", "Interim,Complete,Duplicate,Modified", ","), "Partial,Complete,Copy,Changed report", ","}

will transpose the keyword in whatever you like to print on the report.

Note that the Report.CompletenessTime member is only filled when a complete report is generated with the Mark option active.

Indicates if a seperate report should be created for each target (e.g. issuer).

For instance, for issuer-targeted object reports:

- When not set, there will be only one object report per patient and per report template: the report follows the last issuer!.
- When set, there will be a separate report record per patient per template and per issuer

NewResultCheck NewResultCheck
One of: Ignore, New result required, Only new results, Only new results if complete
Prevents report sections from being printed if they don't deliver any new results. Possible values are:

- Ignore: the new-ness of results plays no role
- New result Required: at least one new result has to be present in the report, before the entire report is printed
- Only new results: only new results are put on the report. However, in case of text reporting (not for electronic reports), a full report is sent when it is complete.
- Only new results when complete: only new results are put on the report AND these results are only reported if the report is complete (i.e. Report.CompletenessTime <> ?). This is typically used to report all the tests that were added after the initial creation of an order.

A report section is a set of adjacent result columns occupying the width of a page. Suppressing sections on Encounter or Day reports saves trees. Informational properties are never considered as being 'new' on the report. This implies that a report for which the only results which already contain a value are informational results (Property.Informational = YES), will not be generated when the New Results parameter of the report generator function setup is set to New Results Required. Moreover, columns on the report which ONLY contain informational properties are completely suppressed.

A report with DefaultReport.TriggerStatus >= 'Available' is considered to be a 'value' report and a report with DefaultReport.TriggerStatus < 'Available' is considered to be a 'Status' report.

For 'Value' reports, the 'New Results' parameter of the report generation function has a slightly different interpretation: when the value of the result was already reported in a previous printout (Result.Status was already >= Report.MinimalResultStatus before Report.LastOutputTime), the result is not considered as 'New' with respect to this report.

If the report generation parameter 'New Results' is set to 'New Results Required', then the report will only be printed when either a new VALUE is reportable, or the report has become complete since the last printout.

For 'Status' reports, a result is considered as 'New' when one if its Creation, Availability, Confirmation, Validation or Rejection times is more recent than the last report output time and this is sufficient to generate the report when the 'New results' restriction has been set to 'New Results Required'.

Note that a status report is marked with NeedsChecking = YES as soon as the report is created.

Logical DestinationFileAppend
Allows to reuse an existing report file for updates instead of generating a new report file.

Prerequisites:

* A report file must already exist with the same file name
* The file name pattern does not use an automatic counter mechanism (e.g. {###})

Usage

If the 'File append' option is set, an existing report file with the same name is reused for updates instead of generating a new report file. If the option is not set, than an existing file will not be updated and a new file will be generated with {##} to ensure uniqueness.

This option applies to the following report template generators: Online (report generation "To file" selected), HL7, EDI1_0, HPRIM, VariableText, EDIMEDV1_0, HEALTH1, MEDIGEST, MEDIDOC, MEDICARD, Electronic reporting to file (Report Destination: Directory, Medibridge and Floppy).

Caution: using this approach, the destination file will be written in multiple write operations. The receiving side (the software that imports the GLIMS report files) must be aware of this ! It could be possible that the receiving system is trying to delete the report file while GLIMS is updating it. These actions must be timed as such that they do not interfere.

Example:

File name pattern: '{Date}Test.rpt' (assuming that Date = 20030201)

* 'File append' is enabled:
-> file '20030201Test.rpt' will be created if it does not yet exist
-> file '20030201Test.rpt' will be updated if it already exists

* 'File append' is disabled:
-> file '20030201Test.rpt' will be created if it does not yet exist
-> file '20030201Test01.rpt' will be created if '20030201Test.rpt' already exists.

bt_ActualFunctionParameterSet InvoicePrintParameterset
Allows defining a function parameter set for the Invoice.Print function. If specified then the most recent (Fund) Invoice will be printed with these settings. If no invoice with payer type 'Fund' is available then the one with payer type 'Object person' will be printed. If there is no fund nor patient invoice and if there is only one payer type then this invoice will be printed. Note invoice printing is only done for order-scoped reports.

This is also true for other grouping mechanisms: when the invoice print parameters do not specify where the output is to be sent, the settings of the report print program are inherited.
