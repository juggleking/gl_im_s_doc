Maximal urgency

Indicates that any request, created from this request definition, should not exceed the specified urgency.
When a higher urgency is requested (e.g. at order entry time) it will silently be truncated to the specified limit.
When applied to a panel request definition, all panel members will inherit the truncate limit, unless they have a lower urgency limit by themselves.

Example:

The following urgencies are defined in GLIMS: N,P,C,S,V

A request definition has minimal urgency P and maximal urgency S

The order is requested with urgency N => the request definition will have urgency P (minimal urgency)
The order is requested with urgency V => the request definition will have urgency S (maximal urgency)

SiteEnumerated MaximalUrgency

Minimal urgency

If specified, this minimal urgency will be applied to all requests created with this request definition.
This implies that the urgency of a request is calculated as the maximum of the urgency specified in the order entry screen, and the urgency specified for the request definition.
If the request definition is a panel request definition, its urgency is inherited by its members, unless they have a higher urgency by themselves (again the maximum of both).

Billing > To charge

When set to "No", this implies that all requests entered with this request definition are not to be charged. When set to "Yes", the requests have to be charged.
When charged, the tariffication will look for billing code assignement(s) in this order:
1) First it will look on this request definition;
2) If not found there, it will look for billing code assignment(s) on the (property/Material/...) default request definition;
3) If not found, it will seach for billing codes assigned on the property (or material,... depanding on the request type);
4) otherwise it will give an 'Tariffcation' error.
In case of panels, this varies according to the setting of the panel specific option "Charge".

Mandatory Logical ToCharge
