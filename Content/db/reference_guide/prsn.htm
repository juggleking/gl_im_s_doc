One of the correspondent types is the (private) person. In medical environments persons are typically embodied by patients. Persons have a self-reference to indicate their mother (relation baby-mother, for instance), and a reference to a health care provider to indicate their family doctor.

Extra > Birth date status

This field contains an indication to the completeness of a person's birthdate.

- If set to 'Standard' then the birthdate is complete.
- If set to 'Year known' then only the year specification of the birthdate is valid, so the birthdate must be interpreted as 00/00/yyyy. In order for GLIMS to have a correct birthdate, it should be set to 1/1/yyyy.
- 'Year and month known' indicates a 00/mm/yyyy date: the day is unknown.
- 'Unknow' indicates that no birthdate is known.

OtherAntigens 

Two MISPL functions are available on the Person table : RhesusPhenoType & OtherAntigens.

These functions will produce a string formatted in the following way: <AntigenMnemonic>+/-<space>. There will be several of these combinations depending on the way that GLIMS has been configured. The RhesusPhenoType functions will contain all antigens equal or lower then the Kell-antigen (as positioned in the fixed list of "meanings"). All other antigens will be displayed with the OtherAntigens-function.

A possibility has also been added for the system administrator to override the behaviour of these MISPL functions by defining customized MISPL functions through an option in the menu (via Start -> System Management -> Customize -> GLIMS transfusion module). The fields are called "Person rhesus formula" & "Person other antigens". If a MISPL function has been defined then these will be used in stead of the default behaviour as provided by GLIMS. The system administrator must keep in mind that the MISPL functions added to override the GLIMS behaviour must have a String as return value in order for the system to work correctly. The possibility to override the functions has been added to allow changing the standard way GLIMS generates the strings or if for any reason you do not agree with the way GLIMS has decided which antigen is in the RhesusPhenoType-function or the OtherAntigens-function.

Example of such a customized MISPL function:

Logical GroteC;
Logical KleineC;
Logical GroteE;
Logical KleineE;
String List;

GroteC:= .Object.Person().GetAntigen("C");
KleineC:= .Object.Person().GetAntigen("c");
GroteE:= .Object.Person().Getantigen("E");
KleineE:= .Object.Person().Getantigen("e");

IF GroteC OR KleineC OR GroteE or KleineE
THEN
List := IF GroteC AND NOT KleineC
THEN "CC"
ELSE IF GroteC AND KleineC
THEN "Cc"
ELSE IF NOT GroteC AND KleineC
THEN "cc"
ELSE "??";
List := List +
IF GroteE and NOT KleineE
THEN "EE"
ELSE IF GroteE AND KleineE
THEN "Ee"
ELSE IF NOT GroteE AND KleineE
THEN "ee"
ELSE "??";
ENDIF;

RETURN List;

Relation 

Returns the n-th PersonRelation record of the specified 'Type' (PersonRelationType) of the person. If there are many, the sort order will be: by RelationType first, then alphabetically.

PersonRelation Relation (String Type, PositiveInteger Number)

String Type
This parameter puts a restriction on the Type of the relation. If specified, only relations that satisfy the restriction will be returned. This restriction can be:

- the name of a particular PersonRelationType: e.g. "Parent" will return only the parents of the person;
- "Family" will only return the relations which are family, not e.g. blood donors;
- "No-family" will only return the relations which are not family-bound: e.g. blood donor relations will only appear with "No-family" (even though these persons can also be family of course).
PositiveInteger Number
Return the n-th PersonRelation record (satisfying the 'Type' restriction) of the person. If n is unknown, 1 will be assumed.

gp_SiteFunction RankingFunction
This Person-based, integer-valued function returns the 'level of reliability' of the person record (e.g. records created by user "hl7batch" are more reliable than manually created records).

Identical records will be displayed in ranking order (descending, i.e. highest ranking first), and the record with the highest ranking will be passed as the 'good' record to the Person.Merge() function. When no ranking function is specified, or when two records yield the same ranking, the most recently created record has preference.
