Introduction

The Privileges tab page of the Security Center allows to configure privileges and protect GLIMS tables, functions and fields.  The screen is composed of different sections which are documented below.

List of privileges

The list of privileges shows all privileges that are defined in GLIMS.

  1. Select a privilege to see which tables, functions and fields are protected by this privilege (see below).
  2. Use the Insert key to define a new privilege.
  3. Use the Delete key to remove a privilege.

List of GLIMS tables

  1. By default, all GLIMS tables are shown. Enable the option Only show protected tables to limit the list to those tables that are protected by the selected privilege.
  2. Enable the check boxes if you want to protect the table for Creation, Updating or Deletion.
  3. If the last column contains an arrow icon, this means that at least one individual function or field of this table is protected by the privilege (see below).

List of table functions

In this phase, it is important to understand how functions are structured in GLIMS. Each function always has a base table. The Validation function for instance is executed on result level and therefore its base table is Result. Even functions without a real context have a base-table: e.g. all functions from the main menu have gp_Site as base-table. This way, all functions in GLIMS can be protected by privileges.

  1. By default, all functions of the selected table are shown. Enable the option Only show protected functions to limit the list to those functions that are protected by the selected privilege.
  2. Enable the check box if you want to protect the function for Execution.

  3. If function parameter sets are defined for a specific function, they will be shown indented below the function.  In the screen example above, cyberlab is a function parameter set defined for the Generate function on table Report.

Note

Each function has a minimal user type assigned to it by MIPS. This can be considered as the minimal authority or experience level a user should have to activate the function.

List of table fields

  1. By default, all the fields of the selected table are shown. Enable the option Only show protected fields to limit the list to those fields that are protected by the selected privilege.
  2. Enable the check boxes if you want to protect the function for Reading or Updating.
  3. The column labeled T (Type) indicates if it concerns a true database field (F) or a site attribute (S).

Protection of 'foreign' fields

Some editors (e.g. order entry screen, Person editor) contain fields which do not actually belong to the base table, but which can be modified via this user interface. We will refer to such fields as 'foreign' fields.

Example

Order entry screen

  • Object.Comment
  • Correspondent.Addressline1

Person editor

  • Object.Comment
  • Object.ObjectTemplate
  • Correspondent.AddressLine1


These fields are also protected by the field privileges: this implies that if for instance the Object.Comment field is protected for update, then the user will not be able to change this field via the order entry or person editor.

Note

Note that some widgets in the user interface may actually belong to the base table, even though they contain the contents of a field of another table.

Example:

In order entry, you can type in an issuer name (= Correspondent.Name) to specify the issuer of the order. So, this widget actually represents the Order.Issuer field. This means that it is governed by the Order.Issuer protection and not by the Correspondent.Name protection!

Note

When a table itself is protected for update (e.g. Person), all its fields are update-protected when they appear as 'foreign' fields.

Example:

When a user does not have the privilege to update the Object table, he will not be able to change the Object.ObjectTemplate or Object.Comment fields in the order entry screen or Person editor.
