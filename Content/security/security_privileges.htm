Introduction

Privileges allow to protect access to GLIMS tables, functions and fields. Only users with the required privileges will have access to the associated tables, functions and fields.

The configuration of privileges consists of two steps

  1. Create privileges and define which tables, functions and fields they protect
  2. Assign the privileges to a user or role (see below)

Example

In the theoretical example below, the following rules are defined:

  1. The Result.Confirm function is protected by a privilege named Confirmation. Only users (e.g. Dr Koch) with that privilege or users that are logged in with the role Lab technician will have access to this function.
  2. The Result.Validate function is protected by a privilege named Validation. Only users (e.g. Dr Koch) with that privilege or users that are logged in with the role Lab biologist will have access to this function.
  3. The field Person.BirthDate is protected for updating by a privilege named Application management. Only ;users that are logged in with the role Application manager will be able to update the birth date of a person.
  4. The table sc_User (creation) is protected by a privilege named System administration AND a privilege named Application management. In the example below, no one is currently authorized to create new users as no one has BOTH privileges.

Role privileges versus user privileges

Privileges can be granted on two levels:

Role-level

By assigning privileges to a role, all users logging in with that role will be granted these role privileges.

Role privileges allow a single definition of a common set of privileges. This eases the maintenance of the assigned privileges. If an individual user needs extra privileges, the role privileges can be extended by individual user privileges (see below).

User level

Allows to assign privileges to individual users. Mostly used to make exceptions to the role-based privileges.

Tip

The applicable privileges for a user are the union of his/her user privileges and the role privileges of the role chosen at log-in time.

Configuration

The entire privilege - user - role concept can be setup in the Security center.

Privilege checks

The validation of privileges depends on the user type of the role chosen during login. The four user types are, in increasing order of authority.

Guests

Guests have no privileges at all. It is impossible to assign a privilege to a guest as GLIMS will not take this into account. As a consequence, guests only have access to fields, functions or tables which are not linked to a privilege.

Users

Users have access to all functions for which they have the appropriate privilege or for which no privileges have been defined.

System managers

System managers have access to all fields, functions and tables. They also have the authority to assign privileges to users.

Tip

There is 1 exception: system managers will not have access to functions, tables or fields protected by the _SysAdm_NoPriv1 and _SysAdm_NoPriv2 privileges if these privileges are not explicitly assigned to them.

Developers

Warning

This user type is reserved for MIPS employees. Functions with minimal user type Developer should therefore not be executed without the assistance of a MIPS employee!

Effects on user interface

Start menu / Contextual menus

Start menu items (tool -> function) for which the user does not have the required privileges will be hidden.

Ribbon

Ribbon items (tool -> function) for which the user does not have the required privileges will be hidden.

Side bar

Side bar items (tool -> function) for which the user does not have the required privileges will be hidden.

Browser

  1. When a user does not have the Table privilege to create / delete records from a given table, it will not be possible to add / remove records in the associated browsers.
  2. When a user does not have the Read privilege for a Field, the corresponding column in the associated browsers will not be shown.

Editor

  1. When a user does not have the Table privilege to update records from a given table, the fields in the associated editors will be read-only.
  2. When a user does not have the Read privilege for a Field, the corresponding field in the associated editors will not be shown.
  3. When a user does not have the Update privilege for a Field, the corresponding field in the associated editors will be read-only.
