General Data Protection Regulation (GDPR)

Introduction

The General Data Protection Regulation (GDPR) (Regulation (EU) 2016/679) is a regulation by which data protection for all individuals within the European Union is strengthened and unified.

This topic explains how GLIMS complies with the following articles of the GDPR, related to patient rights:

  • Article 17: Right to erasure ('right to be forgotten')
  • Article 18: Right to restriction of processing
  • Article 15: Right of access by the data subject
  • Article 20: Right to data portability

Patients (called 'data subjects') can direct their questions related to these topics to an application manager (called 'data controller'). Below you find the procedures to respond to their questions.

Articles

Article 17: Right to erasure ('right to be forgotten')

This article describes that a data subject can request a data controller to erase all his personal data stored in a database. Please take your time to read this document entirely before beginning the process. Follow the instructions in sequence. If you do not use a specific module then skip that section. If a module is not mentioned, there are no specific steps required for the deletion of data in that area.

Logging in as a system administrator will be required for this process. A basic understanding of GLIMS operations is assumed for this document.

Note

Please note that other laws within your country requiring you to keep the data either medical or financial may take precedence. We recommend not deleting any personal data if the person has any data within the mandatory retention period. However, for the data that is not subject to a country-specific law, we advise to follow your own GDPR policy.

Warning

The removal of the personal data is irrevocable!

  • Right mouse click Object… > Show all… > Browse results.

  • Ensure the name is that of the person to be deleted.

4. General – If Person is an Issuer/Agent

If the person is used as an issuer or agent then follow these steps:

  • Open Order consultation

    • Blank all fields.
    • Fill in the Correspondent with the info from the person.
    • Set search as: Issuer.
    • Set Received since to a time before GLIMS was in use (such as 50 years).
    • Ensure Order status option is set to All.

    • Click OK.

      Tip

      You can go through the orders here to view reports, which may contain personal data. These can allow you to find other locations where you may be storing personal data outside of GLIMS (such as e-mails, file storage and printed reports).

    • Press Ctrl + A to select all orders.
    • Right mouse click Change state > Discontinue.
    • Provide a Justification such as "GDPR deletion" and press OK.
    • If any orders were already discontinued, an information message may display for each such order, simply click OK.
    • Press Delete on keyboard.
    • Click Yes.
    • Redo above with search set as: Agent.

5. General GLIMS

  • Open Persons > By last name, first name.

  • Find the Person record to be deleted.

    • Right mouse click Object… > Show all… > Orders (or click Orders in the ribbon).

    • Go to Options in the Orders browser:

      • Ensure Received since and until are set to blank (click field and type "?").
      • Ensure Correspondent is set to "?".
      • Ensure Order status option is set to All.

      • Click OK.

        Tip

        You can go through the orders here to view reports which may contain personal data. These can allow you to find other locations where you may be storing personal data outside of GLIMS (such as e-mails, file storage and printed reports).

    • Press Ctrl + A to select all orders.
    • Right mouse click Change state > Discontinue.
    • Provide a Justification such as "GDPR deletion" and press OK.
    • If any orders were already discontinued, an information message may display for each such order, simply click OK.
    • Right mouse click Advanced > Purge.
      • Tick all boxes
      • Select today's date
      • Click OK

6. Delete logs to ensure no data remains

  • Select Start > System management > Logging > Log types.
  • Select Audit Person.
  • Right mouse click Show all… > Logs.
  • Press Ctrl + A to select all logs.
  • Press Delete on keyboard.
  • Click Yes.

7. GLIMS Archive

If you are using GLIMS Archive then perform the following steps:

  • Open GLIMS Archive.
  • Open Search > Search using the person information (e.g. PIN, last name + first name).

  • Verify that all orders are for the correct person. If so:

    • Press Ctrl + A to select all Orders.
    • Press Delete on keyboard.

If they are not for the correct person, go through the list and delete only those related to the person.

Article 18: Right to restriction of processing

Article 18 of the GDPR gives data subjects the right to restrict the processing of their personal data in certain circumstances. When processing is restricted, the data controller is permitted to store the personal data, but not use it.

If the restriction of processing is requested, apply the following changes in GLIMS:

  1. Change the Person's last name, first name and birth date:

    • Last name: "Restricted_XXXX" where XXXX is a sequence number

      • Add this sequence number to an external register (for instance: an Excel file in which the person's identity is listed).
    • First name: GDPR
    • Birth date: date on which the restriction was requested

    When the restriction of processing can be lifted, the initial values of the above-mentioned fields can be recovered from the external register.

  2. Optionally: add a Person-based Logical site attribute called "GDPR_18" that can be set to "Yes" when processing is restricted.

Note

A MASTER Patient Administration System can overwrite the last name, first name, birth date, sex when patient data is linked to this MASTER application via national or institutional identifications.

In this case, it might be advised to have the restriction documented in the MASTER application as well.

Article 15: Right of access by the data subject

GDPR stipulates that a person has the right to request confirmation as to whether or not personal data concerning him or her are being processed, and, where that is the case, access to the personal data.

  1. Provide the categories of personal data concerned (e.g. address details, insurance details, clinical results...).
  2. Provide a copy of the personal data in a commonly used electronic form. Can be e.g. PDF, Word or Excel.

Article 20: Right to data portability

This article describes that a data subject can ask for a copy of his personal data in a structured and machine-readable format (csv, xml...) to transmit the data to another controller.

Within GLIMS we have a tool for exporting the data to CSV. We recommend you establish a policy for what data needs to be sent to the data subject. This is because much data in GLIMS is not understandable without the full GLIMS configuration to provide context. Please take your time to read this document entirely before beginning the process.

Export tool

Data within GLIMS can be exported to CSV using the Export tool in the Tools ribbon that is available on most browser screens:

Selecting this option will show the Export options screen:

The Folder section determines where the file is exported to.

Make sure to set the Separator according to the regional settings of your target system:

Warning

Something to be careful of is to ensure that no one else’s personal information is sent to a requesting data subject via another field. Be particularly careful of site attributes and comment fields, as they can be exported along. It is recommended to always verify the output before handing it over to the data subject.

Appendix A - Person Medical Record Export

The export of a person's medical record requires the creation of a Tool on the PersonMedicalRecord table. Follow these steps:

  • Go to Start > System Management > Scheme > Site domains.
  • Open ABODeterminationStatus and copy List of values to a text editor.

 

  • Go to Start > System Management > Scheme > Tables.
  • Navigate to and select the PersonMedicalRecord table.
  • Right mouse click Show all… > Texts.
    • Insert a Text with Mnemonic: GDPR-CSV-Header.
    • Copy Appendix A-1 into the Text field.
    • Click OK.
    • Insert another Text with mnemonic: GDPR-CSV-Body.
    • Copy Appendix A-2 into the Text field.
    • Replace the value for ABODeterminationStatus with the List of Values copied earlier.
    • Set the variable Separator to the separator used within ABODeterminationStatus.
    • Click OK.
  • Close the Texts browser.

 

  • Select the PersonMedicalRecord table.
  • Right mouse click Functions > By Mnemonic.
  • Select RunProcedure.
  • Right mouse click Show All… > Tools.
  • Insert a Tool with the following information:
    • Mnemonic = GDPR-CSV-Export
    • Label = GDPR CSV Export
    • Tooltip = Export for GDPR portability request
    • Table = PersonMedicalRecord
    • Click OK.
  • Double click the new Tool and select the Function parameter set.
  • Insert a function parameter set with Short description = GDPR CSV Export.
  • Select the function parameter set.
  • Right mouse click Setup.
    • Set Function to Report.
    • Click OK.
    • In First page header type GDPR-CSV-Header.
    • In Body type GDPR-CSV-Body.
    • Set Output to file to checked.
    • Click OK.
    • Set whatever directory you would prefer the file to be exported to.
    • Set the name of the file as desired such as {##}prmr.csv.
    • Click OK.
  • Close all windows till back at the Tables browser.

 

  • Select the PersonMedicalRecord table.
  • Right mouse click Menu > Edit.
    • In the menu tree editor, insert a new node.
    • Set Tool = GDPR-CSV-Export
    • Set Label = GDPR CSV Export
    • Click OK.
    • Close the menu editor.

 

  • Open Persons > By last name, first name.
  • Select the person.
  • Right mouse click HLA > Person medical record.
    • Right mouse click GDPR CSV Export.
    • Click OK twice.
    • File will now be exported to specified location.
