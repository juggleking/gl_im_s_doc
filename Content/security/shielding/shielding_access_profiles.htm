Shielding - Access profiles

Content: Introduction - Configuration - Activation

Introduction

Shielding based on access profiles allows the specification of the rights to access result, order, object attribute and person records created or entered in another department. The specification is based on a user-specific profile rather than on the fact that the user works for a specific department (see Department group shielding). The profile shielding allows for different settings for technicians, administrative personnel and biologists. It also allows independent setting of the shielding for results, orders, object attributes and persons.

Configuration

Access profiles

The first step is to define one or more access profiles via Start -> System management -> Security -> Access profiles.

  1. Press Insert to create a new access profile.
  2. Enter a name and possibly some comments.
  3. Save the access profile.
  4. Double-click on the created access profile in the browser to configure the access lists:

A profile contains access lists which determine the user's access rights for each department and type of access ("shield class"):

  • Result
  • Order
  • Object attribute
  • Person

You can specify different access rights for each shield class. The radio buttons at the top right indicate for which shield class the access rights should be shown in the list underneath.

To create an access list:

  1. Click in the list and press Insert to add a new row.

  2. Specify the department and class for which this access list applies (if the radio button is set to a specific class, it will automatically be specified).

  3. Set the access rights (create, read, view, modify, confirm, validate, discontinue). Activating an access right means allowing users to perform this operation on the records of this shielding class.

    Example

    Example

    If users should

    • be able to modify results created in Department DEPN, add "DEPN" in the column Department, "Results" in the column Classand activate the Modify check box.
    • not be allowed to see orders created in Department DEPZ, add "DEPZ" in the column Department, "Order" in the column Classand do not activate the Read check box.

    For orders and results, the Read and View access rights must be distinguished:

    • Read: user is allowed to see that result/order records exist (they for instance appear in browsers). However, the user does not have access to the details of these records (it is not possible to open their editors, result values are replaced by "XXX", etc).
    • View: user is allowed to see that result/order records exist AND has access to the details of these records (s/he can open the corresponding editors, see results values, etc).
    Warning

    Warning

    The shielding right View is not taken in consideration for so called "generated" browsers and editors. This means, for instance, that if users can see a record in a browser (because they have the Read access right), then they will also see the details of that record in the browser columns and be able to open the record's editor.

    Generated browsers and editors are automatically created when a table is added to the database. They all have the same layout. Examples are the Order browsers and the Result browsers.

  4. Granted access rights have green background, the others have a red background.
  5. Some access rights (like Confirm or Validate) have no sense for some classes (like Order) and cannot be activated (shown with a white background).

If order and/or person shielding are deactivated (see below), this is indicated in the information box at the bottom of the screen.

Notes

  • The program automatically tries to make the shielding settings consistent: e.g. Modify always implies Read, View always implies Read, for classes Person and Object attribute, Create always implies Read + View, ...

  • If no access profile or no access list is defined, nothing is allowed. For instance, if department DEPN is not mentioned in the Access lists editor of user-profile "Assistant", then users with user-profile "Assistant" will have no access rights for the records whose department is DEPN. However, if a record has no department, the shielding configuration is not taken in consideration: all the access rights are granted.

Linking an access profile to a user

Once the access profiles have been defined, you must now assign them to the users. A user is linked to a department through an employee assignment. In that employment assignment you can select the access profile for that user.

Warning

If a user attempts to log in to GLIMS with an employee assignment on which no access profile is specified, an error message is displayed.

Activation

The general option Shielding method (accessible from the main menu via Start -> System management -> Customize -> GLIMS shielding) must be set to Access profile.

Order shielding

The configured access lists for class Order will only be applied if this parameter is activated.

Person shielding

The configured access lists for class Person will only be applied if this parameter is activated.

If order and person shieldings are deactivated,

  • this is indicated in the information box at the bottom of the access list screen,
  • only results are shielded.
