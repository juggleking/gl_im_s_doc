Shielding - Department groups

Content: Introduction - Configuration - Activation - Result shielding - Order shielding - Person shielding

Introduction

This chapter describes the shielding of result, order and person data based on department groups.

The purpose of the department group shielding is to allow or prevent users logged in department A from seeing, using or updating results, orders, persons, object attributes, ... created or entered by department B.

Note

GLIMS also offers a more flexible shielding mechanism via access profiles.

Configuration

The first step in implementing shielding via department group is to define department groups and their members (individual departments). This can be done via Start -> Correspondents -> Department groups.

General fields

Mnemonic - Name

Free to choose

Shielding

Departments of this group

These options determine the access rights (for results, orders, persons,...) between members of the same group. Enabling an access right means that the user is allowed this type of operation:

  • Create: user is allowed to create orders, results, persons of other departments in this group.
  • Read: user is allowed to see that order, result, person records of other departments in this group exist (they will for instance appear in browsers). However, the user does not have access to the details of these records (it is not possible to open their editors, result values are replaced by "XXX", etc).
  • View: user is allowed to see that order, result, person records of other departments in this group exist AND has access to the details these records (s/he can open the corresponding editors, see results values, etc).
  • Modify: user is allowed to modify orders, results, persons of other departments in this group.
  • Confirm: user is allowed to confirm results of other departments in this group.
  • Validate: user is allowed to validate results of other departments in this group.
  • Discontinue: user is allowed to discontinue results and orders of other departments in this group.

When saving the department group, the program automatically makes the shielding settings consistent. For instance, the Validate access right also requires the View and Read access rights.

Departments in another group

Determines the shield level (for results, orders, persons,...) for members of other department groups. The options are identical to the ones described in the previous section "Departments of this group".

Departments in no group

Warning

If a department does not belong to any group, no shielding is applied: all users have access the records created in this department.

Members

Choose the contextual menu/ribbon item Members to add departments to the department group.

Activation

The general option Shielding method (accessible from the main menu via Start -> System management -> Customize -> GLIMS shielding) must be set to Department group.

Order shielding

The configured shielding is only be applied to orders if this option is activated. If this option is not set, only results (and persons) are shielded.

Person shielding

The configured shielding is only be applied to persons if this option is activated. If this option is not set, only results (and orders) are shielded.

If order and person shieldings are deactivated,

  • this is indicated in the information box at the bottom of the access list screen,
  • only results are shielded.

Note

If you were already using department group based shielding in a GLIMS version lower than version 8:

  • The shielding activation for results, orders and persons was based on environment variables in .glimsrc on unix or Progress.ini on Windows: "MA_Shielding" for results, "MA_OrderShielding" for orders and "MA_PersonShielding" for persons.
  • We strongly suggest removing the MA_Shielding variables, since they may conflict with the new general options documented above. However, when using the department group shielding mechanism, these MA_Shielding variables still have precedence (backward compatibility).

Result shielding

Activation

Result shielding is active as soon as the general option Shielding method is set to Department groups (see above).

Determination of a result's department

The link between a result and its department is made via the Result.PropertyOutput.Procedure.Department reference. This implies that

  • only scheduled results can belong to a department.
  • results which cannot be attributed to a specific department and results of a department which does not belong to a department group are accessible to everyone.

Note

The option Updatable of the work list query makes is possible to only select work list for which you have an access right.

Order shielding

Activation

Order shielding is only active if the option Order shielding is enabled.

Determination of an order's department

The link between an order and its department is based on the Order.Department reference.

Usage

The different order queries (e.g. the order validation query) ignore orders referring to a department which is Read-shielded from the current user department, thus providing additional protection and improved performance.

Person shielding

Activation

Person shielding is only active if the option Person shielding is enabled.

Determination of a person's department

The person is assigned to a department by filling the current user's department in the Department field of the Person editor.

Usage

Once the Person shielding is activated, the shielding mechanism is consulted by the person, object, encounter and correspondent browsers, their respective reference controls (allowing to choose a person, object, encounter or correspondent), the person update check program and the person editor.

Warning

Using this option may have a negative effect on the performance of the person, object, encounter and correspondent browsers.
