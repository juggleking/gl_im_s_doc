Edifact v1.0 - GLIMS configuration

Result reporting sequence

The sequence (of Results) in which Edifact reports is set by the used ReportTemplate (Report.ReportTemplate). The specified ReportTemplate uses a certain PropertyClassification. This class actually defines the order for the Results used in the outgoing report.

All PropertyClassificationNodes point to a certain PropertyClassification Class.

For Edifact the PropertyClassificationNodes are a 2-level tree. At the first level the nodes represent a Section label (e.g. Chemistry, Hematology, ...) and at the second level we have the Property nodes (e.g. Na, Cl, K, ...). The order of the Sections and it’s Property’s will be reflected in the Report.

Edifact reporting only supports a 2-level property classification, i.e. only 1 SEC (section) segment will be generated.  When configuring a property classification with a more complex hierarchical structure, only the top level node will be translated into a SEC segment.  For Edifact reporting, we therefore recommend to limit the property classification structure to 2 levels.  All other levels will be ignored for Edifact reports.

The structure of this tree can be created/altered via:

Configuration/Report Templates/Property Classifications/Edit

We can now edit the selected Property Classification with the tree editor. The Section nodes only contain a label (e.g. Chemistry). The Property nodes do not contain a label but a corresponding Property (e.g. Na).

Caution:

Each PropertyClassificationNode can be set to Report Always. For example: if the Section Hematology is set to Report Always then a Section Label will be created in the Edifact message even if there are no Results for it.

Caution:

Properties that are not included in the Property Classification will not be reported.

Completeness status

The Default report field Completeness status is taken into account to determine whether or not an Edifact report is complete.

It is recommended that the Completeness status is set to Validated.

Note

If the Completeness status is not set, GLIMS will look at the Trigger status of the default report:

  • If the Trigger status is higher than Available, the Minimal result status will be taken into account if it is equal to or higher than Available.
  • If the Trigger status is lower than Available, the Completeness status is considered to be Validated.

How to identify target for message

E.g. HaagNet

The Organization HaagNet is identified via the Edifact ("EDIFACT") Communication Service in Glims

(HaagNet’s Organization.Correspondent.CommunicationService = "EDIFACT").

Target doctor Identification code or MailboxNumber

The MailboxNumber of a doctor is the same as the Identification Code that binds the EdifactMessage distributor Organization.Correspondent to the HCProvider.Correspondent.

Caution: Every targeted HCProvider must have this Identification record in order to determine it’s MailboxNumber.

Remarks

  1. KOP segments are not supported.
  2. If the data of an item in the EDI message is larger then the given length for that data (set by EDIFACT standard), then the message will not be created in the case that the item was mandatory. If, on the other hand, the item is not mandatory and its length exceeds the length given by the EDIFACT standard, then the data will be truncated to that length. Also, if the value of a mandatory field is unknown, a value = "Unknown" will be used (the value of "Unknown" (= English language) however is multilingual).
  3. Currently only report to file is supported for EDI reports.
