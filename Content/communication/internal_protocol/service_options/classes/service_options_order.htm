Order Server Options - to be set as driver options -

Order based Options

InputDataSource

Syntax

InputDataSource = <enumerated Reliability>[,<enumerated TraceValue>]

Values

Default is 'Reliable,TraceDefault'.

Enumerated Reliability
  • Reliable: Accept all patients (person, correspondent, identifications, payment agreement).
  • UnreliableAlways: Never accept patient related info updates.
  • UnreliableWhenUnknown: Never accept patient related info updates when the patient is NOT known (*) in GLIMS.
  • UnreliableWhenKnown: Never accept patient related info updates when the patient is already known (*) in GLIMS.

(*) known means, identified using a message specified identification code.

Enumerated TraceValue
  • TraceDefault:
  • TraceNever:
  • TraceAlways: Stores all person related data differences. The data can be viewed by using the log verify().

Note

This option can be set by the driver.

Example

Order:InputDataSource=UnreliableWhenKnown,TraceNever

OrderCancelSpecimenThreshold

Represents the specimen status for which it is still allowed to discontinue the order.
If some specimen for the order exceeds the specified specimen status, the order discontinuation will not be processed.

Syntax

Values

By default ?. This means that it is allowed to always discontinue an existing order.

Enumerated Values SpecimenStatus
  • Initial
  • Expected
  • Available
  • Discontinued

Note

This option can be set by the driver.

Example

OrderLookUpCriteria

Allows specifying the criteria that must be fulfilled to reuse an order.

Syntax

OrderLookUpCriteria = <enumerated Value>

Values

By default, the service will look up an order via internal id, short id, external id. If multiple lookup fields are specified, then all those fields should match, otherwise the service will generate an error message.
Enumerated Value
  • Object_ObjectTime_Issuer: the service will look for an order matching the object, the object time and the issuer. If other lookup fields are specified, then they must match as well (e.g. ShortId, ...).

Note

This option can be set by the driver.

Example

Order:OrderLookUpCriteria=Object_ObjectTime_Issuer

SynchronizeLowestObjectTime

Allow the order lowest object time to be recalculated or updated from message value on each order update.

Syntax

SynchronizeLowestObjectTime = <SynchronizeLowestObjectTimevalue>[,<CascadeValue>]

Values

By default 'Never'.

Enumerated SynchronizeLowestObjectTimeValue
  • Never: Never recalculate the Order.LowestObjectTime.
  • OnPreCommit: Set the Order.LowestObjectTime to the message specified value prior processing the order.
  • OnPostCommit: Recalculate the Order.LowestObjectTime after the order was updated, scan all specimens and use the lowest specimen sampling time.
  • NO: Only retained for backwards compatibility = Never.
  • YES: Only retained for backwards compatibility = OnPostCommit.
Enumerated CascadeValue
  • Cascade: Change the sampling time for all order specimens to the date-time required - cascade -.

Note

This option can be set by the driver.

Example

Order:SynchronizeLowestObjectTime = OnPostCommit,Cascade

UpdateOrderThreshold

Represents the order status for which it is still allowed to update the order.

If the order in GLIMS exceeds the specified order status, the order method specified via the option UpdateOrderThresholdMethod applies.
If the latter is not set the order update is rejected.

Syntax

UpdateOrderThreshold = <OrderStatus>[,<OrderMethod>]

Values

By default ?. This means that it is allowed to always update an existing order.

Enumerated OrderStatus
  • Pending
  • Initial
  • Partial
  • Complete
  • Closed
  • Discontinued
Enumerated OrderMethod
  • UpdateReject (default)
  • CreateNew

Note

This option can be set by the driver.

Example

Order:UpdateOrderThreshold=Partial,CreateNew

The following situation could occur in case of order import communication:

  • Orders are identified by their short id
  • External system creates a new order with short id "x"
  • External system discontinues order with short id "x"
  • External system wants to create a new order with short id "x"

In that case, GLIMS would not create the new order, as an order with short id "x" already existed in the database.
To solve this situation, the following option can be set: Order:UpdateOrderThreshold=<OrderStatus>[,<OrderMethod>]

RequestAccumulateDiscriminator

Syntax

RequestAccumulateDiscriminator = <Logical Value>

Values

Default is 'Yes'.

Logical Value
  1. YES (default value):
    The GLIMS database field Specimen.SamplingTime will be calculated using the Specimen.SamplingTime and Specimen.Discriminator specified in the message.
     
    Example:

    Order:RequestAccumulateDiscriminator=YES

    Message SamplingTime = 01/11/2011 09:00
    Message Discriminator = +03:00

    &rarr; Database Specimen SamplingTime = 01/11/2011 12:00
    &rarr; Database Specimen Discriminator = +03:00

  2. NO:
    The Specimen.SamplingTime/Request.ObjectTime specified in the message will not be recalculated using the Discriminator specified in the message. The Specimen.SamplingTime from the message will be copied unchanged to the GLIMS database field Specimen.SamplingTime.
     
    Example:

    Order:RequestAccumulateDiscriminator=NO

    Message SamplingTime = 01/11/2011 09:00
    Message Discriminator = +03:00

    &rarr; Database Specimen SamplingTime = 01/11/2011 09:00
    &rarr; Database Specimen Discriminator = +03:00

Note

This option can be set by the driver.

Example

Order:RequestAccumulateDiscriminator=YES

Request Error handlers

CreateRequestErrorHandler

Syntax

CreateRequestErrorHandler = <enumerated ErrorHandler>

Values

Default is 'Halt'.

Enumerated ErrorHandler
  • ContinueAndAccept: Continue with order creation and accept the inbound message.
  • ContinueAndReject: Continue with order creation and reject the inbound message.
  • Halt: Halt the order import, do not create the order and reject the inbound message.

Note

This option can be set by the driver.

The CreateRequestErrorHandler option extends the ScheduleRequestErrorHandler option. In other words when specifying CreateRequestErrorHandler=ContinueAndReject, you also implicitly set the ScheduleRequestErrorHandler=ContinueAndReject.

Example

Order:CreateRequestErrorHandler=ContinueAndReject

ScheduleRequestErrorHandler

Syntax

ScheduleRequestErrorHandler = <enumerated ErrorHandler>

Values

Default is 'Halt'.

Enumerated ErrorHandler
  • ContinueAndAccept: Continue with order creation and accept the inbound message.
  • ContinueAndReject: Continue with order creation and reject the inbound message.
  • Halt: Halt the order import, do not create the order and reject the inbound message.

Note

This option can be set by the driver.

The ScheduleRequestErrorHandler fine-tunes the CreateRequestErrorHandler. It only takes into account the request errors during the schedule process.

Example

Order:ScheduleRequestErrorHandler=ContinueAndReject

PaymentAgreement based options

CanUseDefaultPolicyName

Syntax

CanUseDefaultPolicyName = <Logical Value>

Values

Default is 'No'.

Logical Value
  1. YES: When GLIMS receives an order import message that does not explicitly mention a policy name, GLIMS will use the default policy name defined in GLIMS for the payment agreement (specified in the order import message).
  2. NO: when GLIMS receives an order import message that does not explicitly mention a policy name, the GLIMS configuration will NOT be taken into account.

Note

This option can be set by the driver.

Example

PaymentAgreement:CanUseDefaultPolicyName=YES

Order Server Options - not to be set as driver options -

Order based options, set as user-based order entry options

Some options are retrieved from the user's order entry options. In almost all cases, this is the batch user that started the service. Some drivers (homl, houi) can send a user in the communication message indicating that the order entry options from that specific message should be used.

The following order entry options are taken into account:

  • DefaultIdentificationSource

  • DefaultInstitution

  • DefaultIssuer

  • DefaultAgent

  • DefaultIssuerType

  • DefaultAgentType

  • DefaultSamplingLocation

  • EncounterSelection

  • EvaluateForStatusPending

  • PreCommitEvaluation

  • PostCommitEvaluation

  • PostSchedulingEvaluation

IdentifyableLookUpMethod

Represents the order in which the service will lookup a request code.

Syntax

IdentifyableLookUpMethod = <LookUpMethod>

Values

Enumerated Values LookUpMethod
  • Local: Only the local code = requestcode/Mnemonic will be used to lookup a request code.
  • Alternate: Only the coding system codes of the specified coding system will be used to lookup a request code.
  • PriorityLocal: The service will first try to lookup the request code via the mnemonic and if no match was found it will try via the specified coding system code.
  • PriorityAlternate: The service will first try to lookup the request code via the specified coding system code and if no match was found it will try via the mnemonic.

Note

This option can be set as a Service overload option (= in the command-line option-list of the used gp_Service record).

Example

-O IdentifyableLookupMethod=Alternate
