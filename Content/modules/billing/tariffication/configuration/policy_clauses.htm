Who will pay which part of the cost?

Introduction

In order to determine who will pay which part of the costs, GLIMS allows defining policies and underlying policy clauses.  Each policy has a number of policy clauses (at least one for each reimbursement class that occurs in the order). The policy clause specifies who will pay what under which conditions.

It is important that each payment agreement (which is linked to the policy via a policy name) covers all reimbursement classes for 100%.  In general, one payment agreement will correspond to one policy (exception example: third payer agreements in France).  One policy could for instance have two policy clauses, one for the fund and one for the patient (object person).  The two policies will determine how much the fund pays and how much the patient pays.

Conditions

The conditions allow specifying when the policy clause is to be applied.

Residence type: specifies the required residence type (? = no requirement). The clause is only applicable in case the order's residence type matches the specified value. An order's residence type is the residence type of the encounter type of the order's encounter. Note that orders, not referencing an encounter are considered as 'ambulatory'.

Sub contracting: indicates whether the policy clause is applicable in case the delivery (order) was dispatched.

Forwarded: this is an optional validity condition of the policy clause.  

  • Yes: A lab request is forwarded to another (not the receiver) lab for execution.  
  • No: A lab request is executed in the receiving lab.

Note

It is possible to use a different interpretation of the Forwarded condition:

  • If the condition is set to YES, the policy clause can only be used when the receiving and executing labs do not belong to the same site.
  • If the condition is set to No, the policy clause can only be used when the receiving and executing labs are identical OR different but belong to the same site ("internal labs").

To use this different interpretation, you need to

  • create and activate a site attribute based on the SpecificSite table and with datatype Logical, "RelaxedPlccForwarded":

  • activate this site attribute in the Billing general settings (Start -> System management -> Customize -> GLIMS billing):

  • close and re-open GLIMS.

Eligibility: site function (order based, logical valued) of the policy clause. The return value indicates whether the policy clause is applicable. Returning '?' is considered as 'no', no site function is considered as 'yes'.  This field can be used if several policy clause combinations can be valid and a selection has to be made.

Valid from - until: validity date range.

Specifications

The specification allow indicating what should happen if the policy clause is applied.

  • Reimburser type: the reimburser type is the one who should normally have to pay the costs.  For more information on the possible options, please check the chapter 'Who is involved in the billing process'.

  • Payer type: the payer type is the one who will actually pay the costs.  For more information on the possible options, please check the chapter 'Who is involved in the billing process'.

  • Specific instance: this correspondent can be used as an invoicing target (e.g. an insurance company which pays a specific set of tests independent from the patient's personal fund).

  • To charge: if set to 'no', the billing code which is being tariffed will not be charged and no error will be produced.

  • Multiplier:  used for the calculation of the amount to be invoiced to the specified payer type. Suppose the price calculated so far = 'Pt'.  The invoiced amount = Pt * Multiplier + FixedAmount.  This value is mandatory except for a '3rd payer' policy where the reimbursement part is defined in the '3rd payer payment agreement'.

  • Fixed amount: used for the calculation of the invoiced amount to the specified payer type.  In the example above, the fund will pay the entire amount minus 12.37.  So logically, another policy should be created indicating that the patient has to pay 12.37. 

  • Price list type:

    Standard

    Price list specified in the payment agreement. If not specified, price list specified in the general billing settings.

    Issuer's

    Price list specified in the Correspondent configuration of the issuer.

    Specific

    Price list specified in the policy clause.

    Fund

    Price list specified in the Correspondent configuration of the fund linked to the policy clause.

    If no price list is specified for the fund, an error will be produced during tariffication.
