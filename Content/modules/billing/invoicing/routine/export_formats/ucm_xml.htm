Financial export in "UCM XML" format

Introduction

GLIMS allows exporting financial data in the UCM XML format to CNS Luxembourg.

File check

If this option is disabled, then all data validity checks are disabled. The number of unreported errors is mentioned in the logging. The file is created irrespective of these errors.

Delete temp file

Allows to delete the temporary .json files containing the data to be exported (BilxXport.json) and the feedback regarding the conversion of the exported data into the UCM XML format (BilxFeedback.json).

License

The export of financial data in the UCM XML format is protected by a separate license named "UCMXML Financial Export".

Customize with MISPL

The FinancialShipment.Send function generates an intermediate .json file - containing the data from the financial shipment - which will be converted into the UCM XML format. This intermediate file can be found in the working directory and has the default file name "BilxXport.json".

Customizable fields

The values of the fields in the intermediate file can be customized by means of site functions:

  1. The name of the site function should match the pattern "Send_GE_xxx_Values", where xxx is the table short name.

    Note

    The supported table short names are:

    • Fshp (Financial shipment)
    • Invs (Invoice summary)
    • Inv (Invoice)
    • Iit (Invoice item)
    • Obj (Object)
    • Ord (Order)

  2. The site function should return a tagged list of the table's fields that need to be customized with their new value.

Technical notes:

  1. Fields in the intermediate file which have their default value ("null") will not be exported in the XML format.
  2. The "\" character should not be used when customizing field values via MISPL.

  3. The value of the following fields should not be overwritten:

    FinancialShipment

    • fshp_Id
    • fshp_Firm
    • fshp_Destination

    FinancialShipmentEntry

    • fshe_FinancialShipment
    • fshe_Id
    • fshe_FinancialTransaction

    InvoiceSummary

    • invs_Id
    • invs_FinancialTransaction
    • invs_Payer
    • invs_Firm
    • invs_Fund

    Invoice

    • inv_Id
    • inv_InvoiceSummary
    • inv_Payer
    • inv_Order
    • inv_Reimburser
    • inv_PaymentAgreement
    • inv_EncounterHcProvider

    Order

    • ord_Id
    • ord_Object
    • ord_Issuer
    • ord_Sampler
    • ord_SamplingAddress
    • ord_DispatchingLab
    • ord_ReceivingLab

    InvoiceItem

    • iit_Id
    • iit_Invoice
    • iit_CreditNote
    • iit_ExecutingLab
    • iit_ExecutingHcProvider

    Object

    • obj_Id

  4. Overwriting the values of fshp_NumberOfInvoices and inv_NumberofInvoiceItems is pointless as they are recalculated at the end of the export.
