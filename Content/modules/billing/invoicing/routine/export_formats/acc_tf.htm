FinancialShipment.Send -> Acc TF

Inhoud: Inleiding - Configuratie - Aanvraagnummer - Urgenties - Remgeld - Bestandsstructuur

Inleiding

De GLIMS functie FinancialShipment.Send laat toe om een verrekenbestand te genereren volgens de ACC TF (CS400) layout, versie 2.5.4 (België) voor LABOS (med.techn pakket soort = LAB).

  • Markeren: moet aangevinkt zijn
  • Bestandscontrole: moet aangevinkt zijn
  • Bestandsnaam: gebruik het symbool {ExtId} voor externe id van de verzending of {Date}{##}.txt) als ook symbolische directory naam zoals {TmpDir}.

Wanneer de zending succesvol verstuurd werd zal deze gefixeerd worden (zodat de zending en onderliggende facturen niet meer gewijzigd kunnen worden). Het commentaar bij de financiële zending zal ook een korte samenvatting van het aangemaakte bestand bevatten.

Indien bij het aanmaken van het bestand fouten optreden worden deze in de Glims logging opgenomen met vermelding van het GLIMS order interne id, factuur nummer, veldnaam, record type en de opgetreden fout.

Configuratie

Voorzorgsmaatregelen

Om vaak voorkomende problemen te vermijden, biedt GLIMS een aantal MISPL-gebaseerde controles:

  • 'Order evaluation' functie (geactiveerd via de orderinvoer opties)
  • 'Order set completeness check' (tijdens de tarificatie)
  • 'Payment agreement check' 
  • 'Identification code checks' (formaat/grootte/checkdigit) van dokters, fondsen en andere identificatienummers)
  • 'Nomenclature code check'

Identificatie codes

Alle doorgestuurde correspondent nummers (aanvrager, arts, lab enz.) zijn identificaties toegekend door de bestemmeling van de verzending (id verstrekker).  Als geen identificatie gevonden kan worden, wordt de interne id van de correspondent gebruikt.  De bestemmeling van de verzending moet dus overeenstemmen met de id verstrekker !

Zendingsnummer

Als uniek zendingsnummer (veld 3) wordt de externe id van de financiële zending gebruikt.  Automatische nummering van de financiële zending is mogelijk via de specific site financial shipment numbering expression.

Aanvraagnummer

Aangezien de order korte id gebruikt wordt als verplicht ‘aanvraagnummer’ in het ‘Hoofding’ record dient deze steeds ingevuld te zijn. Merk op dat deze automatisch kan toegekend worden bij de aanmaak van een order via de Order korte id site functie in de specifieke instellingen.  Een goede Order korte id site functie voor ACC communicatie is:

/* Geef een oplopend order short id terug in de vorm NNNNNNN */
RETURN Identifier("OrderShortId", "" /* Prefix /, 7);

Urgenties Nacht/Weekend

Acc wil deze urgenties zelf bepalen maar de urgentie code N/W/F/O/.. dient wel doorgestuurd en dus gedefinieerd te worden in het site domain ‘BillingUrgency’. Er moet een billing urgentie site functie gedefinieerd worden per labo.  Dit laat toe om aan de hand van de aanvraagcode urgenties en tijdstippen een billing urgentie te bepalen en op te slaan in de billing item.

Remgeld aan te rekenen

Elke vergoedingsklasse is van een bepaald type : Administratief Supplement (B), Urgentie Supplement (B), Normaal, Kost, enz.  Het remgeld is het deel van het administratief supplement (forfaitair honorarium) dat wordt betaald door de de patient zelf (plcc_ReimburserType).

Tijdens de aanmaak van een ACC-bestand, wordt gecontroleerd of de patient wel of niet remgeld moet betalen. Voor elke patiëntfaktuur wordt gecontroleerd of het persoonlijk aandeel van de patiënt, normaliter dus te betalen door die patient, in de betreffende polis gemarkeerd is als NIET AAN TE REKENEN.

Anders gezegd, controleer:

  • Zijn er vergoedingsklassen van het type Administratief Supplement
  • Indien het geval, check dan of er corresponderende clausules, in de gerelateerde polis (financiering), zijn, met als schuldenaarstype de Patient (plcc_ReimburserType)
  • Indien het geval, als al deze clausules gemarkeerd zijn als ‘Niet aan te rekenen’, dan moet in de ACC-prestatiefile de aanduiding ‘Geen remgeld aanrekenen aan patient’ worden meegestuurd; praktisch moet dit gebeuren door in de velden 26 tot 28 van record A01 (korting/supplement) de waarde ‘–100%’ mee te geven.

Bestandsstructuur

Het Acc TF formaat is een multi-record fixed length bestand (LEN=550). gebaseerd op het FT432 formaat

Opmerkingen

  • Forfaits alsook de urgenties (N/W) worden niet doorgestuurd naar ACC.  Deze worden wel in Glims berekend maar de versturing naar ACC functie filtert deze er uit op basis van de vergoedingsklasse die voor fortais en urgenties type ‘Adm supplement’ staat.  De indicatie N/W wordt wel doorgestuurd naar ACC.
  • Kosten zijn in Glims verrekencodes met als vergoedingklasse type = Cost
  • Pseudo nomenclatuurcodes: Bij het versturen van pseudo nomenclatuurcodes in het ACC-TF bestand, dan zullen de codes altijd als kost beschouwd worden, zelf als de gerelateerde vergoedingsklasse niet van het type 'Kost' is.  Dit betekent dat de daadwerkelijke kost (prijs) in eurocent verstuurd wordt !

Voordeel: bij het gebruik van pseudo nomenclatuurcodes is het niet langer nodig om een tweede verrekencode van het type 'Kost' te definiëren.  Men kan makkelijk een tweede tarief definiëren voor een officiële verrekencode (bv 54) maar dan met een pseudo nomenclatuurcode.  Tijdens de generatie van het ACC-TF bestand zal deze beschouwd worden las een kost (verstuurd in een A03 record)

Pseudo-nomenclature code zijn : code binnen het bereik (00000)1-(0)99999 / tussen 698000 and 698999 / >= 800000

  • Prijzen worden niet doorgestuurd behalve voor Kosten (veld 53,54 in A02 record (en pseudonomenclatuurnummers zie veld 46 in A02 record) doorgestuurd naar ACC.
  • ACC behandelt een ontvangen dossier (order) als een geheel: geaccepteerd of volledig verworpen. Prestaties voor een dossier kunnen ook maar eenmaal doorgestuurd worden (tenzij bij verwerping). Kleine correcties kunnen evenwel in het ACC systeem zelf gecorrigeerd worden.
  • Bij een pasgeborene verwacht ACC de patiëntnummer van de baby maar als opnamenummer deze van de moeder: dit kan door in Glims in deze gevallen de baby als order object in te geven en als opname (encounter externe id) deze van de moeder te gebruiken.  Om de code bevalling correct te kunnen doorsturen naar ACC dient men een site attribute te gebruiken (definieer een site attribute van het type enumerated met als waarden respectievelijk
    "1st kind,2de kind,….8ste kind,moeder" + specifieer dit site attibute als specific site ChildBirthAttribute).
