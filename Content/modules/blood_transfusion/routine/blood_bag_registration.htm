BTM - Registering blood bags

Contents: Introduction - Scan program - Import program

Introduction

A stock of blood bags has to be created in GLIMS. Blood bags can be entered in GLIMS by means of a scan program or a specific import program.

Tip

The blood product field When received allows specifying a MISPL expression that is executed when a blood bag has been entered.

Scan program

The scan program is available from the main menu Start -> Transfusion management -> Entry -> By scanning barcodes.

Barcode recognition

When scanning the barcodes from a blood unit, GLIMS will recognize the type of barcode and automatically enter the related fields in the scan screen.

This automatic barcode recognition is based on fixed prefixes which are determined by the ISBT (International Society of Blood Transfusion). When using ISBT-128, the barcodes that contain blood group, rhesus and phenotype information are decoded by GLIMS (so you do not need to configure this in the Typing table).

Note

When a barcode is scanned, GLIMS searches for a blood bag having that External id as barcode. In ISBT-128, this barcode has the following structure: = a pp pp yy nn nn nn ff K where 'ff' are flag digits and 'K' is a check digit.

According to the ISBT-128 specifications (June 2001, Version 1.4.0, Section 3, Page 3-18) by the International Society of Blood Transfusion (ICCBBA), Appendix F (Modifications to Tables and Re-Definition of International and National Data Identifiers), Table 2 (ISBT 128-Specified Values for Donation Identification Number Flag Digits), the value of the 'flag' characters (at positions 15 and 16) are only to be interpreted if they have certain values:

  • '01' till '04' means 'Container 1 through 4 of a set'
  • '15' till '19' means 'Container 5 through 9 of a set'

If the flag characters contain any other value, GLIMS will NOT interpret them. Thus, for instance, when a blood bag contains 2 barcode labels for its identification:

  • a label with '=N034003033796002' (flag '00'),
  • a label with '=N034003033796052' (flag '05'),

then GLIMS will always find the same blood bag, no matter which of both labels is scanned.

ISBT barcode for special antigens

The ISBT barcode for special antigens (which starts with the prefix "={" or "=\") is also interpreted by the blood bag scan program. This means that when this barcode is scanned:

  • the list of antigens is shown in the scan program
  • these antigens are stored automatically with the blood bag (in the BloodBagAntigen table), when OK is clicked.

  • the actual antigen barcode is stored (without its prefix) in a blood bag site attribute named _SpecialTestingCode. This barcode can then be used for MISPL purposes.

The actually interpreted antigens are: Kell, Cellano, Duffy-A/B, Lewis-A/B, Kidd-A/B, Luther-A/B, M, N, S, s, P1, Cw, CMV.

Use of extended antigen meaning list

GLIMS supports all antigen meanings that can be scanned using an ISBT-128 barcode.

Activation

In the BTM general settings, enable the option Use extended antigen list.

Antigen configuration

Antigens must be configured for each possible antigen meaning.

Usage

GLIMS will enforce that antigens are configured for all antigen meanings (similar to the standard antigens). This check is performed by several programs, such as the blood bag scan program. When scanning a blood bag, GLIMS will recognize extra antigens. Check the reference manual for the full list of supported antigen meanings.

For more information, please check the PDF document ISBT 128 Bar Code Symbology and Application Specification for Labeling of Whole Blood and Blood Components.

Other features

  • If a bag is scanned for a blood product of which the option Verification of contents is set, you can manually overrule this setting for that bag by enabling the Ignore verify option in the scan program.
  • The Blood bag typing option Obsolete can be used to deactivate certain typing records so that the scan program does no longer recognize these.
  • The identification of the division (the last 2 characters of the ISBT product code) is stored in the field Division. When an ISBT product code is scanned, the division id is stored in the dedicated Division field.
  • If a Blood bag typing or a blood product is chosen (either manually or with a barcode) that is marked as Autologous, the Donation type automatically switches to Autologous.

  • When a blood bag was discontinued (for example because it has been returned to the supplier) and afterwards it is again delivered to the lab (with same number of course), it is possible to re-use this same number in GLIMS.

    When doing so, the Internal id (and the External id as barcode) of the old blood bag (that had status Discontinued) is suffixed with "#1". For instance, blood bag '12345' will be renamed to '12345#1'. If this is done a second time, it will be renamed to '12345#2' etc.

Donation type when scanning blood bags

By default, the barcode scan program for blood bags requires that the Donation type (normal, directed or autologous) be indicated for each blood bag that is scanned as the Donation type option is always reset to its initial value, i.e Normal.

However, it is possible to indicate the Donation type for several blood bags at once. To this purpose, the gp_Site-based function EnterBloodbags can be pre-configured, i.e. it is possible to define a menu option / ribbon item with a tool that has a parameter set.

Skip setup enabled

If the Tool option Skip setup is enabled, and the user clicks on the menu option / ribbon item using this tool, the scan program is immediately opened for the donation type specified in the tool's function parameter set. For each scanned blood bag, this donation type will be used.

Skip setup disabled

If the Tool option Skip setup is disabled, and the user clicks on the menu option / ribbon item using this tool, a window appears allowing to change the donation type that will be used when scanning barcodes. Subsequently, the scan program is opened for the chosen donation type. For each scanned blood bag, the chosen donation type will be used.

Eurocode barcodes

GLIMS provides limited support for Eurocode barcodes in the blood bag scan program. The currently supported barcodes contain the expiration time and the blood group and antigen typing information. Any other type of Eurocode barcode is not supported yet, and will result in an error/warning.

  • Blood group and antigen typing

Format : !Rxxxx

!: Primary data identifer (Eurocode)

R: Secondary data identifer (blood group information)

xxxx are numbers (0-9)

Example

!R1131 encodes the red cell antig. of the unit A Rh pos (D pos) CcD.ee Kell pos.

Warning

The special(s) of the blood group / blood typing barcode (digit 9) are not supported!

  • Expiration date

!EYYYYMMDD: Expiry date (day)

!FYYYYMMDDHH: Expiry date (hour)

Example

!E20191224 encodes the expiry date of the unit: 24 December 2019.

Import program

Introduction

GLIMS supports the import of blood bags via disk. The blood bag import program is available from the main menu through Start -> Transfusion management -> Entry -> Read supplier diskette.

  1. Select the format of the import file.
  2. The blood bags will be imported and displayed in a browser.
Note

Note

It is possible to pre-configure the directory where the file should be found without knowing the file name in advance. Specify a path with a valid directory + a non-existent file name for the parameter FileName of the parameter set .

Example: //TheServer/GLIMS/BloodTransfusion/DeliveryDiskettes/*

Caution: the easiest way to achieve this is to first perform Setup on the parameter set and select a valid filename in this directory. After having set up the parameter set, go to its parameter FileName and replace the filename (at the end) by a *.

When the program is started, the file dialog directly opens in this directory to allow the choice of correct file.

Configuration

You can use the antigen's Meaning field to tell GLIMS which antigens have which medical meaning. This is important for those antigens that are specified in the import file. Before such a file can be read, these antigens must all have been defined and have their Meaning field set correctly.

Product code

In the Blood product specification configuration, the Barcode field must contain the product code according to your supplier's definitions.

You can also store these product codes in the Alternative codes for the blood product specification (it is allowed to map multiple codes to the same blood product specification).

Blood bag supplier

The Bloodbag supplier option in the BTM general settings allows the specification of the default blood bag supplier.

Blood bag browser

The blood bags read from disk are displayed in a browser.

Barcode

Allows scanning the blood bags one by one to make sure that the blood bags on the disk match the actual blood bags. Please also check the chapter Unique identification of an ISBT blood bag.

If a blood bag is scanned and all the fields are filled out correctly, the blood bag will be grayed out and the status will be displayed in the browser.

Tip

When a blood supplier disk has been read, it is also possible to enter all blood units into GLIMS at once, without scanning each individual unit separately.

This can be done with the key CTRL-F1. A confirmation question will be asked before entering the whole list into GLIMS.

Possible import issues

If an error occurred during import, the blood bag will be displayed in red and a message will be displayed indicating the cause. There are several reasons why a blood bag can be rejected:

Expiration date not available

The expiration date is not available (or lies in the past) in the import file. This field is not mandatory in the red cross format but is mandatory in GLIMS.

Solution: adapt the import files by adding an expiration date.

Blood product (specification) not found

The blood product or blood product specification defined in the import file is not found in GLIMS. ??? will be used to mark such fields.

Solution: define the blood product and blood product specifications in GLIMS.

Tip

The blood bag typing should not have antigens defined on it. This allows customers to define other typings with the same mnemonic and rhesus but with antigens defined on them.

Notes

Notes

  • The diskette import program is able to handle multiple blood bags with the same external id, but with a different product code. This can occur when reading Progesa diskettes.

    If a bag is scanned and its number is not unique, the program will ask to enter or scan the unit's product code, so it can select the correct unit from the list.

  • The identification of the division (the last 2 characters of the ISBT product code) are stored the field Division.

Import formats

General "Glims" blood bag import protocol

In order to avoid that a specific import protocol has to be developed for each customer, a general blood bag import protocol that each customer can use is available.

Import file

The import file should consist of:

  1. a header containing the name and the version of the import protocol.
  2. consecutive blocks starting with BLOODBAG.BEGIN and ending with BLOODBAG.END containing the data of each blood bag.

For instance:

The "Directed to" person should be provided in the format <identification code>:<pin provider>.

Example

BLOODBAG.DESTINEDPERSON.PIN=123456789:INS-A

Attributes can be provided as follows:

<attribute name> The attribute will be shown as being present.
<attribute name>:<Y|N> The value after the colon will decide on the attribute's presence.
<attribute name>:<info> The attribute will be shown as being present and will have extra information.
<attribute name>:<Y|N> <info> The attribute will be shown as being present or not and will have extra information.

EFS France (Etablissement Franais du Sang)

When an institution receives blood bags from the EFS, EFS will supply:

  1. The physical blood bags + a paper summary.
  2. Blood bag delivery ("LIV") file. This is the file that should be imported and checked against the physically delivered blood bags by use of barcode scanning.

Required configuration:

  1. An Institution must be defined with Mnemonic "EFS"

  2. All EFS sites (e.g. EFS_IDF, EFS_AquiLimou, ...) must be defined as Organizations with an identification assigned by the EFS institution.


Notes

Notes

  • In the import file, several volumes are defined for each blood bag. Only the "Volume total du produit" will be imported in GLIMS.
  • Antigens are identified directly by mnemonic.
DGTI (Germany)
  • If the rhesus phenotype field is empty (blanks), the blood bag will be linked to a blood bag typing with rhesus = not.ass. and neither of the antigens 'C/c/E/e' specified.
  • If the rhesus phenotype contains 'Cwc' the blood bag will be linked to a blood bag typing with the antigen 'C' negative.
  • If the phenotype contains 'CwC', the blood bag will be linked to a blood bag typing with the antigen 'C' positive.

Version

Version 2.2 of the DGTI import format should be used.

Additional identifier "!"

We recommend using the following MISPL in the Blood bag external id field of the BTM general settings.

String Hersteller, Result;

/* Strip off the leading ! when using ExternalIdAsBarcode */

IF LEN(.ExternalIdAsBarcode) > 1

THEN

IF SUBSTR(.ExternalIdAsBarcode,1,1) = "!"

THEN Result := SUBSTR(.ExternalIdAsBarcode,2,-1) ;

ELSE Result := .ExternalIdAsBarcode;

ENDIF;

ELSE Result := .ExternalId;

ENDIF;

RETURN Result;

Red Cross Luxembourg

Antigens

  • Only phenotypes "+" or "-" are supported as antigen values.
  • Antigens "CH/RG", "Ag Public" and "Ag Priv" are not supported.
  • Antigen values "indtermin", "double population", "non test", "u", "p" and "faible" are not supported.

Blood attributes

When unknown blood attributes are encountered during the import, GLIMS will automatically create new blood attributes using the code in the import file. The Mnemonic and Name fields of the blood attribute will by default be filled with the Code and should be updated manually.

Additional "@" or "A" identifier

We recommend using the following MISPL in the Blood bag external id field of the BTM general settings:

String ExternalId;

IF LOOKUP(SUBSTR(.ExternalIdAsBarcode,1,1), "@,A", ",") > 0

THEN ExternalId := SUBSTR(.ExternalIdAsBarcode,2,-1) ;

ELSE ExternalId := .ExternalIdAsBarcode;

ENDIF;

RETURN ExternalId;
