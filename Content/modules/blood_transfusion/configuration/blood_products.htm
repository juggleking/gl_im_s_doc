BTM - Blood products

Introduction

This table stores the blood materials. The blood product editor is available under Start -> Transfusion management -> Configuration -> Blood products.

Compatibility

Only autologous units

A person can have autologous blood bags. Those blood bags must be transfused to that person. Yet, if more blood bags are required than the number of available autologous blood bags, one has to decide whether or not other (non-autologous) blood bags will be transfused. This decision however is site-dependent. This toggle disables the possibility to transfuse other (non-autologous) blood bags to an autologous person.

Calculate score

Allows defining a BloodSelection-based integer-valued MISPL function that calculates a compatibility score for the blood bags. GLIMS also offers the possibility to define ranking rules via blood recipient types and donor rules.

Check antibodies <-> antigens

If this option is enabled, only blood bags will be selected of which the antigens have the opposite value of the corresponding antibodies in the patient. If a conflict is found, the "Matching score" of the blood bag can be influenced by the score mismatch factor that is defined in the antigen configuration. This allows completely rejecting the blood bag (score factor = 0) or just lowering the matching score.

Note that the typing advice of a person antigen also influences the antibody <-> antigen check. If it is set to No, the blood bag will always be accepted even if there are antibody-antigen conflicts.

If the antibody <-> antigen check should only be applied in certain cases, you can use the Only when field (see below) to indicate when the check should be applied.

Only when

When should the antibody-antigen check be executed? This eligibility MISPL is a function based on the 'BloodSelection' table and should return a logical value. This site function will only be executed if the option Check antibodies <-> antigens is activated.

This functionality allows you e.g. to switch on/off the special antibody checking or antigen checking according to the patient's case. You could return NO for normal patients and YES for patients where care should be taken, so that these patients receive more compatible blood.

Check antigens <-> antigens

When this option is enabled, only blood bags will be selected where the antigens have the same value as in the patient, if known. If a conflict is found, the "Matching score" of the blood bag can be influenced by the score mismatch factor that is defined in the antigen configuration.

This allows completely rejecting the blood bag (score factor = 0) or just lowering the matching score.

More precisely:

  • if an antigen is positive for the patient, only blood bags where this antigen is positive (not unknown) will be selected;

  • if an antigen is negative for the patient, only blood bags where this antigen is negative (not unknown) will be selected.

This check causes blood bags to be selected with e.g. the same rhesus-phenotype as the patient's.

Note

Note

If the typing advice field for this patient's antigen is set to NO, the antigen will be completely ignored; so that bags with another value for that antigen will still be allowed.

Example:

If the patient has antigens C+ c- E- e+, then blood bags with

  • C+ c- E- e+ will be OK

  • C+ c- E+ e+ will be rejected

  • C+ c- E? e+ will be rejected

Caution:

This rejection of blood bags can be

  • switched on or off per patient (using a MISPL function in the Only when field)

  • switched on or off or be made less strict by setting the score factors that are defined in the antigen configuration.

Warning

Warning

We advise not to switch off this option to circumvent the built-in security (MIPS cannot be held responsible in that case!).

 

Only when

When should the antigen-antigen check be executed? This eligibility MISPL is a function based on the 'BloodSelection' table and should return a logical value. This site function will only be executed if the option Check antigens <-> antigens is activated!

This functionality allows you e.g. to switch on/off the special antibody checking or antigen checking according to the patient's case. You could return NO for normal patients and return YES for patients where care should be taken, so that these patients receive more compatible blood.

Skip check for person positive for the antigen

  • This option is only applied when the option Check antigens <-> antigens is enabled (see above).
  • If this option is enabled and a patient is positive for a specific antigen, then no mismatch will be registered. This also implies that the antigen-antigen mismatch score defined for that antigen will not be applied. The blood bag will be considered compatible for that antigen.
  • All other values for the availability of a person antigen (Negative or Unknown) can result in an antigen mismatch. If a patient is negative for an antigen and the blood bag is positive for that antigen, an antigen mismatch will be detected and the corresponding antigen-antigen mismatch score will be applied.

Crossmatch

Crossmatch not applicable

This option should be set when no cross match is to be performed. The blood bag will then automatically be promoted from status Initial to Checked during selection of the bag.

Default cross match

Only for blood products where Crossmatch not applicable is deactived. If specified, a result for this property request definition will automatically be added as compatibility result, at the time when the blood bag is selected.

Overrule default crossmatch

If a cross match is only to be executed if certain conditions are fulfilled (or if different cross matches are to be executed depending on the situation), you can define a MISPL expression in this field.

It refers a blood selection based result id-valued expression. It can e.g. be used to create a cascade request of the cross match (typically using the MISPL function BloodSelection.CompatibilityCascade).

Triggers

The following remarks apply to all the trigger fields described below:

  • The return value is of type 'Logical'.

  • If a trigger returns NO, the action will be aborted.

  • Each trigger also has the possibility of specifying the reason why it returns NO. This is done by writing this reason into the variable 'NOK_Message' in the following way:

    PokeCharacter( "NOK_Message", "This patient is not ready yet !" );
    RETURN NO

    If the trigger does this, the user will see this reason in a message box. Else the message box will say "Action inhibited by '...' trigger".

    Example

    Example

    Example for trigger 'before select':

    if .Object.GetResult("IrrantiScr",?,1,?) = ?
    then
    PokeCharacter( "NOK_Message", "Patient has not been screened yet for irregular antibodies !" );
    RETURN NO;
    endif;

Before select

This trigger is executed first when either of the two selection functions Select compatible bags or Select blood bag manually is executed. It is executed first, before any other screen appears and can be used to check a number of conditions that should be met before the user can select a blood bag for that patient (example: is there a recent irregular antibody screening?). If this function returns FALSE, searching or selecting a blood bag will not be possible.

Note

Note

This trigger should avoid the usage of the old 'BloodProperty' eligibility MISPL expressions.

When selected

This trigger is executed at the moment that a blood bag is linked to a blood selection. Once a blood bag has been selected, this MISPL expression will be run. If this function returns FALSE, the selection of the blood bag will be aborted.

Utmost transfusion time

By default the utmost transfusion time is initialized at order entry as the due date/time. In this field, you can indicate a blood selection based MISPL of datatype DateTime which will re-calculate the utmost transfusion time (if it is not updated to a later time, the transfusion cannot take place). The returned result will be stored as utmost transfusion time in the blood selection record.

The update takes place when the blood bag reaches status Checked, that is

  • either after a successful crossmatch
  • or after selecting a bag for which no crossmatch is required.

The new utmost transfusion time can be based on various attributes of the blood selection, the order, etc..

If the utmost transfusion time of a blood bag is elapsed, GLIMS will no longer allow to check out or administer the blood bag.

Note

Note

Using this function allows you to avoid calling 'SetUtmostTransfusionTime' or 'SetAttribute' from within the selection trigger of from within the crossmatch property trigger.

Examples

Examples

RETURN .Order.ReceiptTime + 3; /* 72h after order was received */
or
RETURN .LastRequest(?).DueTime + 2; /* 48h after due time of request*/
or
RETURN .CheckResult.ObjectTime + 3; /* 72h after specimen for crossmatch was sampled from patient */

Note

Note

It is also possible to update the utmost transfusion time in the selection trigger or in the crossmatch property trigger (using the BloodSelection.SetUtmostTransfusionTime MISPL function). Don't use more than 1 possibility together, otherwise the utmost transfusion time will be updated more than once.

When checked

This trigger runs after a blood selection (and hence also the blood bag) has reached the status Checked. This can happen in 2 cases:

  1. a blood bag is selected for a product that doesn't need a crossmatch

  2. a blood selection is promoted to Checked because of the trigger of a crossmatch result.

This can be useful e.g. to print a transfusion form.

Note

Note

The result of this function (TRUE or FALSE) will be ignored.

Autom. check out if (for future use)

Indicates when a blood bag should be automatically checked out when it reaches the status Checked.

When checked out

This trigger is executed when the blood bag is checked out. When it returns NO, checking out will be cancelled.

Note

In this MISPL trigger, you can also access the fields specified when checking out a blood bag (Check out comment, Ward, In the past and To person).

When returned

This trigger is executed when the blood bag is returned to the lab, either towards status Initial or Verification. If this function returns FALSE, the status change of the blood bag will be aborted.

Note

In this MISPL trigger, you can also access the fields specified when checking out a blood bag (Check out comment, Ward, In the past and To person).

When administered

This trigger is executed when the blood bag has been set to status 'administered'. If this function returns FALSE, the status change of the blood bag will be aborted.

When expired

This trigger is executed at the moment that a blood bag is marked as 'Expired'. If this function returns FALSE, the status change of the blood bag will be aborted.

When discontinued

This trigger is executed at the moment that a blood bag is discontinued. If this function returns FALSE, the status change of the blood bag will be aborted.

When department changed

This trigger can be used when certain things should be done or checked whenever a blood bag of this type of blood product is moved into another department (=depot).

Example

Example

Trigger that rejects moving autologous units into department 'D1':

---------
  IF .DonationType = DonationType["Autologous"] AND .Department.Mnemonic = "D1"
  THEN
     PokeCharacter( "NOK_Message", "Department " + .Department.Name + " is not used for storing autologous units !" );
     RETURN NO;
  ENDIF;
  RETURN YES;
  ---------

After treatment

This trigger is executed whenever the method Give treatment is used on a blood bag. The specified site function should return a LOGICAL. If it returns NO, the treatment of the blood bag will be aborted, so this can be used to check some things.

The site function could typically be used to reset the expiration time of a blood bag using the BloodBag.SetExpirationTime( <newtime> ) MISPL function. The new expiration time must be specified as parameter.

Example

Example

To prolong the date with two weeks:

.SetExpirationTime( .ExpirationTime + 14 );

When wasted

This trigger is executed when the status of a blood bag changes to Wasted. If this function returns FALSE, the status change of the blood bag will be aborted.
