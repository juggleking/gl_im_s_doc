Introduction

The two most important tables are:

  • Isolation series: grouping the isolations of the same patient and same/similar anti-biotype
  • Hospital Infection: grouping the isolation series of different patients which cross-infected each other

With the following relationships:

* An isolation can refer to
  1. the isolation series to which it belongs
  2. the isolation of which it is the double (similar antibiogram and within x days detected before)
* An isolation has an 'infection update mode' which can be:
  1. automatic: the system will try to figure out whether it belongs to an isolation series and whether it is a double or original
  2. explicit double: the user explicitly declared it as being identical to another isolation
  3. explicit original: the user explicitly declared it as the first occurrence of an isolation series
  4. no infection: the user explicitly declared it as not being an isolation series at all (not important for statistics)
* An isolation series always refers to
  1. its first isolation (the isolation that was first detected)
  2. the microorganism
  3. the material in which it was first detected
  4. the patient
* An isolation series can refer to
  1. the hospital infection which it is part of

Practical guide

When an isolation is created, it refers to no isolation series at all.  Its 'infection update mode' is 'automatic'.

There is a contextual menu function on isolations called 'Infection status'. This function shows the actual state of the isolation: the update mode ('automatic',...) as well as the isolation series it is part of and the isolation which it is probably the double of.

Automatic detection

With the contextual menu function on isolations called 'Determine infection' you can let the system figure out if the isolation belongs to an isolation series, to which one, etc.  When a microbiology action is validated (either preliminarily or finally), the detection of isolation series is automatically executed for each isolation contained within the action. 

Practically it is advisable to configure a ‘Command’ to do all this at regular times (like .e.g. every night) for all isolations that could possibly have been added or modified since the last run.  For example: every night re-determine the isolation series for all isolations with sampling time between two months ago and today.

An isolation series is created if the isolation

  1. has been confirmed
  2. has a positive appraisal

The system considers an isolation as the duplicate of another one if

  1. it is the 'same' microorganism:

    The detection of doubles - which by default searches for identical organisms for a certain patient, can be defined to consider all organisms within a certain taxonomic group. The level of this group can be defined with the option "Slightly different microorganisms may be identical" in the general 'microbiology settings'. (see screenshot below (1)).

    E.g. suppose you have set up a structure like:

    [-] Campylobacter species
    [-] Campylo Coli (subspecies)
    - Campylo Coli type A
    - Campylo Coli type B
    - Campylo Jejeuni (subspecies)

    In this case suppose the doubles detection level is set to 'Species', then it would be possible that a Campylo Coli is considered to be a double of a Campylo Jejeuni (if the antibiograms match etc.).

    If the doubles detection level is set to 'Subspecies', this would not be possible, but a Campylo Coli type A can still be detected as a double of a Campylo Coli type B.

    If you don't want this, but you only want doubles within strictly the same organism, then you should leave the doubles detection level empty ('?').

    GLIMS will also compare the isolations' discriminator and if those are different, the isolations will be considered as different.

  2. it is the same patient
  3. the antibiograms of 2 consecutive isolations match with a maximal difference of 'x' (see screenshot below (2))
  4. the antibiograms of the first isolation and last isolation of the isolation series match with a maximal difference of 'y' (see screenshot below (3))

where 'x' and 'y' are decimal numbers that can be configured in 'System management -> Customize -> GLIMS microbiology'.

  1. the Maximum number of days between 2 consecutive isolations of a series (see screenshot below (4)) is not exceeded.

The following detection parameters can also be specified per micro-organism or micro-organism family:

  • maximum number of days between 2 consecutive isolations 
  • maximum antibiogram difference between 2 consecutive isolations 
  • maximum total antibiogram difference between first and last isolation of a series

This means that each parameter that is specified for a certain microorganism is automatically inherited by all descendant microorganisms.  

About antibiogram difference

The numeric difference of 2 antibiograms is defined as the number of differences 'Resistant-Sensitive' + half of the number of differences 'Resistant-Intermediate' and 'Intermediate-Sensitive'.

Glims also allows defining rules that determine the relevance of specific antibiotics.

It can happen that

  1. Some antibiotics are not important when comparing the antibiogram of 2 isolations.  This means that equal isolations can have different results in their antibiogram for the same antibiotic, although both are really the same.
  2. Some antibiotics are so important that, if there is a difference, the isolations are certainly different.

To support this in Glims, you can define rules for this.  This can be done in the tab page "Epid. relevance" of antibiotic configuration screen.

  1. Each rule is specific for a certain antibiotic and a certain micro-organism
  2. Each rule specifies the "relevance":

"Normal" is the default
"Difference is decisive"
"Difference is ignored"

"Normal"

This is the default behaviour (as if the rule didn't exist):  a difference between 2 antibiotic results is counted as 1 point (for R-S) or 0.5 point (for R-I and I-S).

"Difference is decisive"

In this case a different result for this antibiotic will immediately imply that both isolations are different. (Technically such a difference counts as an infinite number of points.)

"Difference is ignored"

This means the system should ignore results for this antibiotic, when comparing 2 isolations.  The antibiotic will have no influence on the decision if the isolations are different or equal. Technically such a difference counts as 0 points.

Examples:

"Oxacilline" "Staph.Aureus" => "Decisive"
"Ampicilline" "Staph.Aureus" => "Ignored"

Manual determination

With the contextual menu function on isolations called ‘Infection status’ the user can see and change the way the isolation is treated.

  • By setting the 'isolation doubles' status of the isolation to ‘Explicit original’, the system will create an isolation series with this isolation as the first one.
  • By setting the 'isolation doubles' status of the isolation to ‘Explicit double’, the user has the opportunity to choose himself which other isolation it is the double of. The user is presented a list of isolations that have been detected before in the same patient. By selecting one of them, the system will link both isolations and the last isolation will refer to the same isolation series as the selected one.
  • By setting the 'isolation doubles' status of the isolation to ‘No infection’, the system will make sure that the isolation is not treated as part of an infection.
  • By setting the 'isolation doubles' status of the isolation to ‘Automatic determination’ (default), the system will try to figure everything out by itself.

Overview

  • In the main menu there is a function ‘Microbiology -> Isolation doubles’ to get the list of all isolation series, sorted by the date/time of the last detected isolation. For each isolation series, the list of contained isolations can be viewed.
  • For each isolation there is a function to go to its isolation series.
  • In the microbiology statistics the doubles can be taken into account.

Note: it is possible to show only infections, cross infections and hospital infections in browsers if they have a minimum infection severity.  This minimum severity can be specified in the general epidemiology options (Start -> System management -> Customize -> Glims microbiology -> tab page "Epidemiology"), in the field "Only severity >=".

Cross infections

The function 'Determine infections' also creates a 'Cross-infection' record as a link between different infections (Isolation series) of different patients that were in the same ward in the same period (see also chapter 'Hospital infections'). They can be viewed from an 'Isolation series' with the contextual menu functions 'Possibly caused by' and 'Possibly transmitted to'.

Glims also offers a trigger that runs each time a cross infection is created.  This cross infection trigger can be specified in the general microbiology options (Start -> System management -> Customize -> Glims microbiology).  The defined MISPL expression must have "CrossInfection" as base table.  The main advantage that the cross infection trigger has over the "Infection trigger" is that more information is available: from patient, to patient, via ward, via room etc.

Remarks

For isolation series (infections), the field 'Hospital acquired', is automatically set to Yes or No according to the parameter 'Maximum number of days between admission and hospital acquired' (see screenshot above (5)) in the microbiology specific settings and/or in the microorganism 'Epidemiology' settings.  The field is set by the function 'Determine infection'.

Example: if the Maximum number of days between admission and ' hospital acquired' is set to "3" and a patient enters on 01/01/2008 then an isolation detected on 02/01/2008 will NOT be considered as "hospital acquired".

Only isolation series of which the option "hospital acquired" is enabled are considered during the detection of hospital infections.
