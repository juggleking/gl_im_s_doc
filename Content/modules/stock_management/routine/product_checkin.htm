Stock management: Product check-in

Contents: Introduction - Description - Check-in unused products - Check-in to substock - Processing

Introduction

The product check-in screen is used to check in ordered or unsolicited products for a specified work place and stock. It is also used as an editor on delivery, which allows full tracing.

Description

The product check-in screen can be accessed from the main menu via Start -> Stock management -> Check in.

Ordered or unsolicited

When starting the check-in program, you will first be asked if you want to check-in ordered or unsolicited products (e.g. products that have not been ordered but that should be registered in the stock). You can create pre-configured menu options for each choice. The actual check-in screen will slightly vary according to the chosen option.

Screen example

1. Enter delivery information

By specifying the following fields, you will provide GLIMS with more tracing details about the product deliveries.

Courier

An organization acting as a courier for the delivery.

Shipment no

A courier-specific reference number for the delivery. The combination of these two fields is unique; if one is specified, the other should also be.

Comment

Editor to supply a delivery comment.

Received at

Displays the current date and time (only specified in delivery editor).

Received by

Displays the current user (only specified in delivery editor).

2. Specify general check-in info

Ordered products

In the Check-in page, select a Work place, Stock and Purchase order to indicate that you want to check-in products ordered through that purchase order.  The Supplier and comments will be shown.

Unsolicited products

In the Check-in page, select the Work place and Stock for which you want to check-in products.

3. Indicate the type of product you will check-in

In the Items browser, you should now indicate which kind of products you will check-in by selecting a product specification.  As the browser always shows the quantities in number of units per packing, the values are only displayed correctly if complete packages are scanned (see below). So, the last column Status will give more information by displaying Some products if incomplete packages are delivered and Some packages if complete ones are delivered.

Ordered products

The browser shows all product specifications applicable to the selected purchase order.  Select the one for which you want to scan products.

Unsolicited products

Insert new product specification in the browser.  You can also use the "Insert" key.
Remove product specifications from the browser. You can also use the "Delete" key.

Add all the product specifications in use for the selected stock and work place.

See Check-in unused products.

4. Scan the products to be checked in

Once you have selected a product specification, you can start scanning the products to be checked in.

Fields

Scan by

Determines the number of products that will get the same external id.

Product

Each product will need to be scanned and gets a different external id. The browser shows each separate product.

Package

One scan action assigns the external id of a number of products equal to the number of units per packing/number of units per consumption. The browser only shows one of the products of a packing. Switching the option to Product causes the browser to show all products again, so individual products can still be modified.

Scanned and order quantity

The scan screen has an update-able field Scanned quantity and a read-only field Ordered quantity.  Default values:
  • Scanned quantity = ordered quantity - already available quantity (in the delivery editor this will be the delivered quantity)
  • Ordered quantity = purchase item ordered quantity (0 for unsolicited check-in)

Scan unit

Depending on the Scan by option, the screen shows the consumption unit name or packing unit name.

Unique external id

The uniqueness of the product external id will be checked if this option is set.  If another product/package of the same specification with the same external id is available, the user gets a warning, but it is still possible to select it.  If this option is enabled, the scanned quantity is set to 1 and read-only.

Auto confirm

If set, the barcode will be confirmed automatically.

Production date, expiration date and lot

Must be specified before scanning the product

Usage

  1. Enter a lot, production date and expiration date.

  2. The bar code can be scanned; i.e. after scanning, the right record in the product browser will be updated with the data specified in the field barcode, lot no, production date and expiration date.  The barcode field is cleared after each scan, but the other fields keep their value, so updating subsequent records is easy.  If a lot was specified during the creation of the purchase order, that lot will be specified by default.

  3. In case of unsolicited check-in, the records are created by scanning.  After the first scan, a number of products equal to packing unit/number of units per consumption will be created. Subsequent scanning will cause these records to be updated. If they are all marked as checked, then an additional scan will create a number of products equal to packing unit/number of units per consumption again.
Notes

Notes

  • The default expiration is calculated using the storage period defined in the product specification. It can be overridden manually or by choosing a lot with expiration date.  Mandatory for product specifications for which the option with "perishable" is enabled. Note that you still need to define a command to set product to status Expired when the expiration date has elapsed.
  • Each scan option can be undone; all products with the same external id as the focused one will be reset. Modifying products in the product browser is not allowed.

Check in trigger

If a site function is specified in the Check in trigger field of the Stock management general settings, it will be executed.

5. Specify invoice and payment information

For each purchase order that is selected for check in, a new record is added in the embedded Billing browser of the check in screen. The invoice and payment information (invoice no, invoice date, payment no, payment date, cost location) can be filled in this browser. Additionally, billing information of previous deliveries is displayed (if available), but grayed out and in read-only mode.

When clicking on the Check in button, the purchase order invoice records are created.

Note

Initially, the Checked option of the Purchase order invoice record is set to No. The Check function on Purchase order can be used to mark the purchase order invoices and the purchase order as Checked.

6. Check-in the scanned products

  1. When the Check in button is pressed, all the check-in actions (including product label printing) are performed and afterwards the check-in screen is refreshed with the delivered item info:
    1. Completely checked in items will disappear from the product specification browser.
    2. The number of delivered items will be copied to the number of available items and the number of delivered items will be set to 0.
    3. The checked in products will be set grayed-out and insensitive.  
  2. As long as the check-in screen is not closed, the products will be linked to the current delivery. The current delivery (courier, shipment no, comment, received at and received by) is updated on each check-in action.

Check-in unused products

GLIMS allows checking-in products that are not used in a specified workplace/stock:

  1. The product specification query has an option Unused only to restrict the product specifications to those that have no product usage.
  2. If the general option Support unused product specifications (Start -> System management -> Customize -> GLIMS stock management) is set, the product check-in screen will offer an extra button for checking-in products that are not in use.
  3. When clicking this button, the product specification browser with option Unused only will be opened. Further processing is identical to processing of product specifications with usage.

Check-in to substock

"Product usage" field "Substock"

A field Substock is available for Product usage.  The field is displayed in the product usage browser and is supported as product usage query parameter. A query selection screen will be displayed before execution of the product usage query.

Product usage browser

The product usage browser displays the sum of the quantities of products (scheduled, ordered, delivered) in both stock and substock.

Menu function "Set usage" on "Product specification"

The menu function "Set usage" on "Product specification" allows to set the substock.  When double-clicking in the fields to select a stock and substock, a stock overview will be shown as tree view (instead of the regular stock browser).

Menu function "Update" on "Product usage"

The menu function Update on Product usage allows to set the substock for existing product usage records.

"Product check-in" screen

Products will be checked in to the substock of the selected stock if its usage record has a substock specified. If no substock is specified, they will be checked in to the stock, as before.  The destination stock is displayed in the embedded product browser in the check in screen.  

Menu function "Order products" on "Product usage"

The menu function Order products on Product usage takes both products in the stock and substock into account in order to calculate the available, ordered and scheduled quantities.

Transferring products from one stock to another

No warning will be issued while transferring products from a stock to a substock.

Processing

For all purchase orders the quantities of all items are added together. If the ordered quantity matches the already available quantity plus the delivered quantity, the purchase order's status is set to Complete, else the status is set to Partial. In this latter case the order can be set to completion with another delivery.

For ordered items with a complete delivery the status of the associated products is set to "Delivered"; for unsolicited items, new product records in status "Delivered" are created. The number of created products equals: 

  • Purchase item quantity (number of units per packing) * number of units per packing / number of units per consumption.

Stock, work place, external id, etc... are set for each delivered product. The effective price, which may be different from  the specification's price, is stored with each product.

For each delivered product where the check box Print labels is set for its specification, a label is printed. The content of this label is determined by a dynamic text on base table "Product", to be specified in the product specification table.  It should preferable evaluate to a string that is unique for each product. Click here for more information on the printer that will be used.
