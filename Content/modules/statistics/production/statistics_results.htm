Statistics -> Result statistics

Result statistics give an overview on what has really been done in the lab. It allows to answer questions like:

  • How many analyses were performed on a certain station, or for a certain doctor, by a certain external lab, ...?
  • How many QC-results have been performed?
  • What was the average time lag between the sampling and the validation of the result ?
  • ...

Note that the production numbers might give one an indication on how much has been expended (hence they are sometimes called "Expenditure statistics"), but it does not return a price value linked to the bills. For real price information the financial statistics are more appropriate. What production statistics can do is calculate a kind of total cost based upon statistical weights per property that were assigned by the user.

Textual value

This will work identical as specifying a user-defined function of type 'String', containing "RETURN .RawValue;". So it will do statistics on the raw measured values of the results.   

Property weight (numeric)

This is the "statistical weight" of the property this result refers to. It can be used to store e.g. the cost or price of performing one analysis of this property. Its value can be configured in the property weight table where you can specify a start date and end date.  The validity period is the first period in which the 'availability date' of the result (date at which the value was entered) is contained (start date <= availability date <= end date).
 

Turnaround time (conf/val)

This is the time lag between the time the specimen was taken from the patient and the time the result was confirmed/validated; it can of course only be calculated for confirmed/validated results.  The result is textual.
 

Turnaround time (conf/val) in min

This is the time lag between the time the specimen was taken from the patient and the time the result was confirmed/validated; it can of course only be calculated for confirmed/validated results.  The result is a decimal.  A different calculation than the above turnaround options is used which may lead to a slightly different rounding in the end result.

The selection screen

  • Which time reference should be used when specifying a time range ?
    • Produced: the date on which the results were produced (measured, given a value).
    • Requested: the order's receipt date.  Allows to compare the the result statistics with the corresponding order statistics

This option also influences the meaning of all time-related options in the first screen (for rows/columns/...) => which date will be used to group the results in rows and column ?

If you choose Produced (default value), the statistics will consider the date on which the results have been given a value (have been produced in the lab).
If you choose Requested, the statistics will consider the date on which the orders have been received, so the date on which the results have been requested.  

When using the Requested time option, the user can include results that have not been produced yet, i.e. results with status Initial or Expected... (see Status from - to field. )  

Note

When using the Requested option,  QC results will automatically be excluded (since they were not ordered).
