Request definitions

Contents: Introduction - Fields - Types - Options - Property.Replace

Introduction

Request definitions (Requestable table) determine what will exactly be requested when entering its request code in order entry.  This can be a single property, a material, a panel, etc.  

Each request definition refers to a request code which is used to request it during order entry.  

Tip

When creating a property, material, etc  GLIMS will automatically create a default request definition and corresponding request code.

Note

A request definition may depend on the issuer and on the time. This allows to link several request definitions to a single request code.  When entering a request code in order entry, GLIMS will lookup the appropriate request definition (linked to the request code), taking into account the issuer and time specifications of the request definition.

Fields

Request definitions are available from the main menu via Start -> Configuration -> Request definitions -> Request definitions.

General

Request code

Select the request code used to request this request definition during order entry.

Mnemonic

Short, unique identifier. Cannot contain spaces, quotes nor backward slashes.

Discriminator

Please check the chapter Requesting tests with a discriminator.

Start date, End date

The request definition is only valid between these 2 dates. 

Issuer

If an issuer is specified, this request definition will only be used for orders with this issuer.

Is default request definition

Automatically set by GLIMS.  

When creating a property, material, etc  GLIMS will automatically create a default request definition (and corresponding request code) and will mark it as being the Default request definition.

Minimal urgency

If specified, this minimal urgency will be applied to all requests created with this request definition. This implies that the urgency of a request is calculated as the maximum of the urgency specified in the order entry screen, and the urgency specified for the request definition.

Note

If the request definition is a panel, its urgency is inherited by its members, unless they have a higher urgency by themselves (maximum of both).

Maximal urgency

Indicates that any request, created with this request definition, should not exceed the specified urgency. When a higher urgency is requested (e.g. at order entry time) it will silently be lowered to the specified limit.  

Note

If the request definition is a panel, all panel members will inherit the lowered limit, unless they have a lower urgency limit by themselves.

Example

The following urgencies are defined in GLIMS:  N,P,C,S,V

A request definition is defined with Minimal urgency P and Maximal urgency S

  • The order is requested with urgency N => the request will have urgency P (minimal urgency)
  • The order is requested with urgency V => the request will have urgency S (maximal urgency)

Types

A request definition can be of type Property, Material, Panel, Procedure, Blood, Microbiology, Pathology, or Genetics. Depending on the type, additional fields will be shown (see below).

Property

Property that will be requested with this request definition.

Material

Material that will be requested with this request definition.

Panel

A panel allows to group a number of request definitions.  When requesting a panel in order entry, all request definitions within that panel will be requested. 

Fields

Description

Free text description.

Confirm

If enabled, GLIMS will show a confirmation screen with all panel members when requesting the panel.

Add / Remove panel members

Click a panel member to add or remove it to / from the panel.

Tip

  • By default, all panel members are included. This can be changed by enabling the Optional field of a panel member.
  • The selected panel members are not requested until the user presses OK.

Discriminator

The panel member's discriminator can be modified using the standard syntax.

Note

After clicking OK in the confirmation screen, GLIMS does not remember that the requests were actually part of a panel. panel tampering

Accept shared members

Is used to determine whether a panel is complete or not.

If a panel has a fixed price, GLIMS has to determine whether or not the panel is complete before applying the panel price. Only complete panels can be tariffed.

This option is mainly used in case of overlapping panels in the same order.  When this option is enabled, the panel completeness check will accept that a panel is not fully completed due to the fact that some members have been requested through another panel.

Charge

Members only

Only the panel members will be evaluated for tariffication.

Panel only

Only the panel itself will be evaluated for tariffication, not its members.

Panel and members

Both the panel itself and its members will be evaluated for tariffication.

Members or panel

The panel itself will only be tariffed if all panel members are not to be tariffed because they have not been requested with a billing mark that is included in the Tariff member billing marks field of the panel.

Discount rate

Please check the chapter Discounts.

Tariff member billing marks

Allow to specify a list of billing marks (no separator).  Only requests that have been requested with a billing mark from this list will be tariffed.

Example

Tariff member billing marks = ABC

When requesting the panel and specifying D as Billing mark at order entry, the panel members will not be charged.

Note

This option is only applied when the panel members are to be charged, i.e. the option Charge is set to Members only or Panels and members.

Adding members to a panel

To add members to a panel

  1. Select the request definition and choose the contextual menu / ribbon item Panel members.
  2. Use the Insert key to add a new panel member.

Sequence number

Determines the sequence of the members within the panel. They will be requested in this order.

We strongly recommend to always specify a sequence number in order to determine in which sequence panel members must be requested / visualized.

Note

If no sequence number is specified, the panel members are sorted/requested according to the ID of the request code linked to the panel member.

Member mnemonic

Request code that will be added when this panel member is requested.

Discriminator

Please check the chapter Requesting tests with a discriminator.

Billing marks

Allows to specify a list of billing marks (no separator).  If specified, the panel member will only be requested if its billing mark (specified at order entry) is included in this list. 

Optional

If enabled, the panel member will not be requested automatically when requesting the panel. You will have to manually add the panel member in the panel confirmation screen which is shown when the Panel option Confirm is enabled (see above).

Microbiology

Procedure mnemonic, Procedure material

In microbiology, you actually request a microbiology procedure.

These fields allows to select a microbiology procedure, by first selecting the Material of the microbiology procedure.

Add microscopy

If disabled, the Microscopy property defined in the microbiology procedure will not be requested. This can be used to skip the microscopy phase of the microbiology examination.

Pathology

Procedure mnemonic, Procedure material

In Pathology, you actually request a pathology procedure.

These fields allow to select a pathology procedure, by first selecting the Material of the pathology procedure.

Genetics

Genetic procedure

Genetic procedure that is requested along with the request definition, leading to the creation of a genetic exam.

Options

The Options button allows to define which request definitions should be displayed in the request definition browser.  

Tip

The same options are available when looking up a request code during order entry.

Microbiology labs can e.g. specify that they only want to see request codes with microbiology request definitions.

Types

Only show request definitions of this type.

Description

Only show request definitions with this description.

Issuer, Check

GLIMS allows to define issuer-specific request definitions.  The Issuer and Check fields allow to take this characteristic into account when looking up a request definition.  

You can specify the issuer and the relation between the issuer and the request definition:

None

The issuer is not taken into account. All request definitions are shown.

Available

Only the request definitions which can be requested for the specified issuer are shown.

Specific

Only the request definitions explicitly referring to this issuer are shown.

Enabled

Yes

Only show request definitions of which the request code is enabled.

No

Only show request definitions of which the request code is disabled.

?

Show all request definitions, without checking if the request code is enabled or not.

Explicit

Yes

Only show request definitions of which the request code is marked as Explicit.

No

Only show request definitions of which the request code is not marked as Explicit.

?

Show all request definitions, without checking if the request code is marked as Explicit.

Urgency from, to

Only show request definitions with a minimal / maximal urgency between this range.

Property.Replace

The goal of this function is to replace all request definitions referring to the original property with request definitions referring to the newly chosen property. For more information, please check the chapter Properties - Replace.
