Introduction

In medical environments persons are embodied by patients.

The Persons browsers can be opened via Start -> Correspondents -> Persons.

Fields

The Person editor looks as follows:

Main page

Demographic data

The main page contains general demographic data about the person such as the names, title, birth date, twin, sex, language, spouse name, address information and phone.

Display last name

Last name used by the person

Legal name

Person's name as indicated in national registers

Note

To search for a person record on the basis of its Display last name and/or Legal name,

  • add these fields to the person externalization MISPL,

  • use the order consultation query, which can look in the externalization MISPL (cf Externalization field).

Spouse name

Name of the person's spouse

Age

Shows the age of a person once his birth date is known. The age is either expressed in years, months, weeks or days, whichever is most convenient. The reverse is also possible: specifying the age of the person in years, months, weeks or days will automatically update the person's birth date.

Name query

Allows doing a remote query based on the name and first name filled in in the corresponding fields.

Birth order

Only accepts values between 1 - 9. The birth order can also be specified during order entry.

Sex

Gender of the person as written on administrative documents such as identity cards. May be different from the biological sex.

Internal id

Internal identification of a person. For more information, please check the chapter Person internal id.

PIN / Assigned by

A person can have several identifications (PINs) from different identification sources (PIN providers). The identification source shown is determined as follows:

  1. the user preference Default identification source
  2. The identification source with the lowest sequence number in the PIN provider group specified in the general settings

  3. The person identification source is specified on SpecificSite level (Start -> System management -> Customize -> Glims general)

Allow automated creation of person identifications (or PINs). When both provider and code are specified and upon committing the screen, the corresponding Identification record will be created if it does not exist yet. If only the PIN provider is given, and this PIN provider has an automatic identification code function (see Correspondent.IdentificationCode), this site function is executed in order to generate a PIN code for this person.

PIN query

When a patient administration system (e.g. a HIS) is connected and active, this button allows querying the remote system for the patient data using the current value of the PIN field. If the patient is found, the data is created or updated in the Glims database and is shown in the editor.

This button offers the same functionality as in the order entry screen. Also, just as in the order entry screen, pressing F8 when located in the PIN field has the same effect as pressing the query button. The query button can be used both in insert mode or in update mode.

Note that any uncommitted changes to the patient record still present in the editor are discarded when the query is executed.

Phone, Phone 2, Mobile phone, Fax, Fax 2, Modem number, E-mail, Web site

Refer to the corresponding fields of the Correspondent table.

Template

See object templates.

Comment

Can be used to store clinical remarks for a patient.

Payment agreement browser

The person editor also contains a payment agreement browser, showing the existing payment agreements of the person (if any). The existing agreements are sorted by:

  • Active first
  • Primary agreement - third payer agreement
  • Start date

Agreements which are no longer valid today are shown with a red start-end validity period.

There are two ways to insert, update or delete agreements in the person editor:

Using the payment agreement browser

The payment agreement browser can be used to create or update simple payment agreements without the need of using the full payment agreement editor. It allows entering the fund mnemonic or name, the matriculation and the policy name. Hitting the Insert key will create a new empty row in which the necessary data can be entered. Hitting the Delete key will mark the record for deletion.

Notes

  • The records are only created, updated or deleted when committing the entire screen (i.e. when hitting the OK button or the F2 key).
  • When creating a new agreement, the program may propose to close existing open payment agreements of the same person.
  • When the program finds an existing agreement for the same person, fund, policy name and matriculation, it will be reused instead of creating a new agreement.
  • When the creation of the new agreement fails (e.g. due to the matriculation number check), the user is offered the option to try it again with the PaymentAgreement.Checked field disabled. This will make sure the matriculation check is skipped, but this will of course yield a payment agreement that will fail the checks during the actual billing process.

Using the icons next to the browser

These icons will open the full payment agreement editor in order to create/update more complex payment agreements.

Extra page

External id

Identification of a person given outside the lab.

Department

GLIMS fills the Person.Department field upon creation of the Person record to indicate to which department (legal entity) the person is linked. This information is useful for e.g. shielding purposes. The Person.Department can be updated by the user. If a Person record is created:

Manually

The Department field in the Person editor is filled with the department for which the active user is logged in.

Electronically

Via electronic order entry. The Department field in the Person editor is filled with the department of the batch user linked to the service.

The person externalization can display the Person.Department:

Default externalization

The default person externalization also contains the code of the department stored in the Person record. If the Code of the Department is not specified, it will not be included in the externalization.

User-defined externalization

If a site function is used in the general options (Start > System management > Customize > GLIMS general) to calculate the person externalization and it should also include the person's department, then it is required to update this site function.

Family doctor

Family doctor normally consulted by the selected person.

When this field is filled in, a button will be available in the Order Entry screen after first having selected the Person in order to allow easy selection of the family doctor as Issuer of the order.

Employer

The current employer of this person

Mother

Specifies the person's mother. This is used

  • in medical laboratories for the billing of assessments on infants.
  • in the context of genetic tests on a foetus to refer to the mother of the Person of type Foetus.

Date of birth status

Contains an indication to the completeness of a person's birth date.

Standard

The birth date is complete.

Year known

Only the year specification of the birth date is valid, so the birth date must be interpreted as 00/00/yyyy. In order for GLIMS to have a correct birth date, it should be set to 1/1/yyyy.

Year and month known

Indicates a 00/mm/yyyy date: the day is unknown.

Unknown

Indicates that no birth date is known.

Decease time

Specifies the time of death of the selected person.

Real identity

Optionally refers to a person which contains the real identity of an anonymous person. This link should be protected against unauthorized access by defining a privilege on the Person.RealIdentity function. For more information, check the chapter Patient anonymity.

Keyword

This can be any important word that has something to do with this person. You can use it to your own needs. The purpose is that you can search for (a) person(s) having a certain keyword. To do this, use the browser Persons by keyword from the main menu.

Foetus on creation

A person can be marked as a 'foetus' (an unborn person). This can be useful when tests are being performed on a pregnant mother or her foetus. If this option is enabled, the field Before birth is hidden. For more information, please consult the following chapters: Foetus (old system for Genetics), Prenatal context (Genetics module).

Before birth

Allows to make a reference to the person record which represents the same person but as a foetus. You can only select persons of which the field Foetus on creation is enabled. For more information, please read the Foetus chapter (old system for Genetics).
In the Genetics module, this field is obsolete.

Foetus page

This tab page displays the following fields:

Pregnancy no.

Identification of the pregnancy

Genotypic gender

Twin type

Placenta

Discriminator

Digit, letter or name used to differentiate the foetus from the other foetus(es) of the pregnancy.

The values of these fields can be updated

  • if in the Extra tab page, the Mother is indicated and the option Foetus on creation is activated.

  • using the contextual menu/ribbon item Update foetus information.

    The fields of the Update foetus information window are usually initially filled in when a foetal specimen is created. The Pregnancy information fields can only be filled in if the mother is the Object (patient) of an order.

Protected person records

In hospital environments, the Correspondent.ProtectImportedData option is often enabled. When this option is active, Person or Correspondent data imported via electronic communication with this correspondent is protected against manual updates. This allows creating an organization with one single master system (e.g. the HIS) from which all patient data is administered.

System managers are able to update protected person records. When a user of type System manager opens the person editor on a 'protected' person, an Unlock button will appear in the upper right corner of the editor. When pressing the button, all fields are enabled and the person can be updated. Pressing the button only removes the restriction for the current record.

When protecting person data, the identifications will also be protected. In order to allow system managers to delete such protected identifications, a menu function Remove PIN is available in the Identification contextual menu. Only users of type System manager are allowed to execute this function.
