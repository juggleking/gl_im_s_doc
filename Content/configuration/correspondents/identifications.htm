Identifications

Contents: Introduction - Concept - Identification lookup/creation - Identification fields - Identification types - Usage - Iceland - Remarks

Introduction

In GLIMS each correspondent type can have an identification assigned by another correspondent type.  

The most obvious example is a PIN: each patient (person in GLIMS) has an identification assigned by the hospital information system (institution in GLIMS).  However, identifications could also be assigned to HC providers, wards etc.  Identifications can be created automatically through communication with external systems (e.g. HIS communication) or can be assigned manually in GLIMS.

Concept

The figure below shows the concept of identification in GLIMS:

An identification is always assigned by a correspondent (identification source) to another correspondent (identification target). If a correspondent is used as identification source, the following Correspondent fields can be used:

Identification check

Allows specifying a site function that verifies if an identification assigned by this correspondent is valid (e.g. complies with a given format).

Identification code

Both the Person editor and the Person series creation program allow automated creation of person identifications (or PIN's). When specifying a PIN provider without specifying the actual identification itself, the site function specified in the Identification code field is executed in order to generate a PIN code for this person.

Translator

In the order entry screen, you can lookup a patient by identification.  In order to do so, you must also specify the PIN provider (which is a correspondent). To establish the communication with this PIN provider, the Translator specified in his Correspondent record is used.

National patient identification numbers

GLIMS allows specifying the PIN provider that distributes the national patient identification numbers. The field National Pin Provider in the general settings (Start -> System management -> Customize -> GLIMS) allows indicating which organization is the national PIN provider.

  1. In the Person editor, the national identification number will be shown if the patient has a valid identification assigned by the National PIN provider.
  2. The Station option Patient identification can be set to National number, which means that the National PIN provider will be used.

Priority for PIN providers

Wherever a patient identification is to be shown in GLIMS (order entry, person details) or used in analyzer communication, and the patient has multiple identifications from different PIN providers, GLIMS has to know which identification should be shown/used.

To solve this , it is possible to specify a priority for PIN providers (identification sources).

  1. From the main menu, click Start -> System management -> Customize -> GLIMS general.
  2. The PIN provider group field allows to select a correspondent group.
  3. The selected correspondent group determines the priority among identification sources.

Note

  1. The PIN provider group is restrictive.  If a patient has an identification assigned by "ProviderX", but "ProviderX" is not included in the correspondent group, the identification will never be used.
  2. In the order entry screen, the order entry option Person identif. source (or the user preference Default identification source) has priority over the PIN provider group.
  3. In the person editor, the user preference Default identification source has priority over the PIN provider group.

Identification lookup/creation

As explained above, identifications are often created automatically through communication with external systems.  However, you can also manually assign identifications in GLIMS.  There are several ways to lookup / create an identification for a correspondent.

From the main menu

From the main menu, click Start -> Correspondents -> Identifications.  

Query

Assigned by

Search all identifications assigned by this correspondent (internal id or name).

Assigned to

Search all identifications assigned to this correspondent (internal id or name).

Target type

Search all identifications assigned to correspondents of this subtype.

Valid on

Search all identifications valid on the specified date.  If unknown, today will be assumed.

Automatic

Search all identifications which have been created automatically or manually.  Check the identification field "Automatic" for more information.

Browser

A list of all matching identifications will be shown.

This identification browser is also used when hitting Ctrl-F in each field where an identification can be selected (e.g. the order entry screen). The order entry program and the order series creation program also pass the target type and the source to this browser, so the identification browsers in these programs only show identifications for correspondents of the desired type.

From the correspondent contextual menu

From each correspondent, you can select the contextual menu/ribbon item Identifications and then:

Assigned by

Shows a list of all identifications assigned by this correspondent.

Assigned to

Shows a list of all identifications assigned to this correspondent.

In the Person editor

If a person has identifications from multiple providers, the PIN shown in the Person editor is:

  1. The PIN assigned by the user preference Default identification source.
  2. The identification source with the lowest sequence number in the PIN provider group specified in the general settings (see above).

Using MISPL

Correspondent.Identification(<SourceInternalId>)

Returns the identification code of the first Identification given to this correspondent by <SourceInternalId>.

Correspondent.IdentificationList(<SourceInternalId>, <RefDate>, <format>)

The MISPL function Correspondent.IdentificationList returns a comma-separated list of identification codes assigned to the correspondent by <SourceInternalId> . When <SourceInternalId> is not specified, then ALL the identifications assigned to this correspondent are returned in the format "provider/code,provider/code,...". Only identifications active on <RefDate> are returned. When unknown, TODAY() is assumed.

The <Format> parameter is not yet supported.

Note

As these MISPL functions use the internal id of a correspondent, you have to be aware of the fact that this function will no longer work if you change the internal id of the concerned correspondent !

Identification fields

Assigned to

Correspondent to whom this identification was assigned.

Assigned by

Correspondent by whom this identification was assigned.

Code

The actual identification code.

Valid from- to

Period during which the identification code is valid.

Automatic

Identification types

Introduction

When exchanging patient information, a patient is identified by one or several identifications each assigned by an identification provider (a Correspondent in GLIMS). In communication messages, both the actual identification and identification provider are mentioned.

However, several countries have also implemented a national health care program which introduces a national number. 

Examples

  • The Netherlands: BSN number
  • Belgium : Rijksregisternummer
  • France: INS number

For each identification provider (correspondent), GLIMS is able to indicate which identification type is used in its communication messages.

The HL7 protocol explicitly supports this in the field PID-3, CX-5 : "Identifier type code".

Example

"NNNLD" for BSN numbers in The Netherlands.

Defining identification types

To define the different identification types, click Start -> Correspondents -> Identifications -> Identification types.

Mnemonic

Unique code for the identification type that is used in communication messages. E.g. "NNNLD" for BSN number in The Netherlands.

Name

Name that is used to display the identification type in GLIMS.

Description

Multilingual description of the identification code.

Assigning an identification type to an identification provider

For each identification provider (correspondent), the identification type can be specified in the field Default identification type.

Tip

If an identification provider supports several identification types, you will have to define a different Correspondent record for each identification type.

Communication messages

In communication messages, GLIMS will identify a patient by the actual identification number, the identification provider and the identification type.

Example

[1234^^^HIS_A^PI~5678^^^HIS_B^NNNLD]

where

  • 1234 is the identification assigned by provider "HIS_A" with identification type "PI"
  • 5678 is the identification assigned by provider "HIS_B" with identification type "NNNLD"

Note

Whether or not the identification type information is processed depends on the driver (version) and the external application.

Usage

In communication with external systems

Identifications are often used by communication programs (in combination with the identification provider) to uniquely identify a correspondent. 

Lookup in order entry

in the order entry screen, you can lookup a patient by identification.  In order to do so, you must also specify the PIN provider.  In the order entry options, you can define a default Identification source as well as the Identification source access. 

Search criterion in order consultation

In the order consultation program, you can use the PIN as query criteria for the orders that you would like to view.

Iceland

In Iceland, each patient has a unique, nationwide number, consisting of his birth date, a sequence number and a check digit. Due to the widespread use of this number, explicit support is provided to allow using this number as Person.InternalId.

To this effect, the following functionality is available:

  • The field SpecificSite.PINType allows to indicate which special type of person identification is used on the site. The currently supported values are Standard and Icelandic. 

    If set to Icelandic, the following happens:

    • The InternalId field is made sensitive, allowing the manual entry of an internal id. When entering an internal id manually, all characters are converted to uppercase.  
    • If an internal id is entered, its validity as an Icelandic National Number is verified.  
    • When valid, the person birth date (when still unknown) is filled automatically.  
    • If SpecificSite.PINProvider is specified, an identification record for this person, issued by the specific provider, will be created or updated automatically. This will allow using the national number wherever a PIN is required.  
    • Any identification created for the default pin provider specified in SpecificSite.PINProvider will be subject to the same checks as the person internal id. If necessary, this identification code is copied to the Person.InternalId field.

    • The function Person.AssignInternalId() has no effect. The internal id will not change.

    Note

    Icelandic national identification numbers starting with '8' or '9' are accepted without any further verification. Moreover, these numbers do not contain the individual's date of birth, therefore, GLIMS does not automatically set the birth date.

  • The SpecificSite.PINProvider allows to specify a default PIN provider for the site. Most programs dealing with person identifications will use this as a default provider (although a PIN provider specified in the user's order entry options has precedence).

Remarks

  • After each update of a person identification record, the corresponding person's externalization will be recalculated. This allows using the person's PIN code in his externalization.  
  • GLIMS allows a correspondent having multiple identifications with the same code in a different time period. Typical example: a HC Provider has been assigned a code in the past, got a new code later, and finally got the original code back.  So, a correspondent can have multiple identifications with same Source and Code, providing the Start date differs.
