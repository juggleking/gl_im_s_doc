Double patient detection

Introduction - Query options - Double patients browser - Automated generation of double patient list - Performance

Introduction

GLIMS allows to produce a listing of double patients by means of the DoublePersons query on the Person table.

The query is incorporated in two browsers:

  • A main browser gp_Site.DoublePersons, accessible from the main menu via Start -> Correspondents -> Persons -> Lookalikes that allows verifying all Person records created within a specified period.  For all selected Person records, the database is searched for look-alikes. If any are found, the triggering person along with all its look-alikes are shown in the browser.   
  • A Lookalikes child browser of the Person table that shows all look-alikes of a given person. Note that, contrary to the usual child browsers, the original person record itself is also included in the browser in order to facilitate merging.

From these browsers, you can use the Merge function to merge the look-alikes.

Query options

The principle of the double patient search program is quite simple: you specify the criteria to be used for comparison, along with the allowed number of mismatches. If that threshold isn't reached, two persons will still be considered as identical, although they don't match 100%.

Comparison criteria

The comparison criteria can be used to determine whether or not two persons are identical. Select the options that are applicable to the situation in your organization.

Note

Most of the selectable attributes are local fields of either the person or its related correspondent table. These attributes are simply compared.

For PIN, Payment agreement and Encounter, the situation is somewhat different. For these three options, two persons will be considered equal if they have respectively at least one common PIN, matriculation or encounter external id.

Soundex

If you want an attribute to be assumed equal when sounding similar, active the Soundex option.

Knock-out

This flag indicates that a mismatch is fatal. Regardless of any other option specified, a mismatch on a knock-out attribute will cause the comparison to be aborted and the persons to be considered as different.

Comparator function

The Comparator function option allows for an additional comparison criterion by means of the SpecificSite.PersonComparator function.

Created since, Created until

Choose an absolute or relative date.

For all Person records created within this date range, the database will be searched for look-alikes. For the Lookalikes function on the Person table, these fields are disabled because there is only one record to be checked.

Allowed differences

Allowed number of mismatches. When specifying '0' (zero), all knock-out toggle-boxes will be disabled, because this means that each of the selected options is a knock-out option.

Ranking

This Person-based, integer-valued function returns the 'level of reliability' of the person record (e.g. records created by user "hl7batch" are more reliable than manually created records). 

Identical records will be displayed in ranking order (descending, i.e. highest ranking first), and the record with the highest ranking will be passed as the 'good' record to the Person.Merge() function. When no ranking function is specified, or when two records yield the same ranking, the most recently created record has preference.

Double patients browser

After having specified the query options, the double patients browser will be displayed.

This browser offers the following features:

Direct communication with the Person.Merge() function

Identical person records are automatically pre-selected. The pre-selection starts at the first block of look-alikes and continues one by one until all blocks are processed.

This pre-selection is intended to work together with the two buttons Merge and Skip. The Merge button passes the group of selected records to the Person.Merge() function, while the Skip button simply selects the next block of identical persons without any further processing.

Alternating background color

Blocks of look-alikes are displayed with an alternating background color to allow easy recognition. The color is hard-coded and hence cannot be changed.

Extra info

In the lower left corner of the browser window, extra person information is displayed: 

  • PIN's: all PIN's assigned to the person are shown. The standard PIN is always displayed as the first, followed by an alphabetically sorted list of the others. Any non-standard PIN is also followed by the internal id of its provider (between "()"). 
  • Encounter external id's 
  • Payment agreements

When the information doesn't fit in the area, the full content is displayed in the tool tip.

Progress indicator

During selection of look-alikes, a progress indicator is shown. Instead of using fixed intervals, it is attempted to update the indicator every 3 seconds.  This may yield varying increments.

Performance

The program will perform best when no mismatches are allowed. In that case, the database can be queried straightforward, and the search can be aborted on the first mismatching criterion. In the other case, all criteria are to be verified until a final decision can be taken.

Some combinations of options will not be accepted because it would take too long to complete. This may occur in the rare case that none of last name, spouse last name or birth date is selected.

It is advised to start with a zero-tolerance, and increase the allowed number of mismatches until it satisfies your situation. Also, start with a small time period to reduce the initial result list.
