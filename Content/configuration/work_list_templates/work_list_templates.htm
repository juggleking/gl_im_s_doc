Work list templates

In general

Work lists are ordered sets of procedure realizations, or actions. Work list templates are models for work lists, determining the basic criteria for inclusion and sorting of actions in work lists, as well as some layout directives of the work list. Work list templates are empty until they contain one or more work list template entries, identifying procedures to include realizations from. Note that a work list template, grouping only procedures without inputs and with material type outputs, is a sampling list template. Work lists are commonly used as a medium for listing the work to be done by a particular work unit (person or station) in the laboratory.

When Request.InternalComment or Result.InternalComment is available, this will be marked with a '*' next to the result.

Type

Determines if actions that already appear on a previous version are added to the new version.

Complete: allows each generation of the same template to retrieve all the actions of the considered state (from...to...), even if they are already present on a previous work list.

Only new: for the same work list template, GLIMS will not return the actions already present on a previous list, whatever the date of its edition (the numbering of the actions starts at '1' for each list).

Continuation: same procedure as above, with a variation in the numbering of actions from one list to another (GLIMS takes into account work lists published on the same day). For example: if the last action in the previous list is No. 6, the first action in the next list will be No. 7.

Continuation - reset: same as 'Continuation' but the numbering of the actions starts again from '1' each time the list is edited.

Include criterion

Action-based, Logical-valued MISPL expression that is evaluated at work list generation time for each action meeting the work list's procedure and status criteria. Determines whether or not it will eventually be included in the work list.

Sort criterion

Action-based, String-valued MISPL expression. It is evaluated at work list generation time for each included action and determines its relative position among the other actions in the work list. If not specified, the field WorkListTemplateEntry.SeqNo (or, when empty, Procedure.SeqNo) is used during the sorting phase of the work list generation program. Note that the WorkListTemplateEntry.SeqNo field (or, when empty, Procedure.SeqNo) also applies for scanned work lists (i.e. when the specimens which are to be put on the list are explicitly scanned).

Layout type - Grid view

In general

In case of 'grid' work lists, the following fields allow to customize the work list layout:

In Grid view, the work list is visualized as a two-dimensional grid of cells, organized in rows and columns. Each row corresponds with a work list entry, each column corresponds with an action output (either specimen or result). Grid view is most often used to edit work lists containing assessment actions. It is implemented with a generic grid control: in each cell, the result value can be viewed or modified.

There are three types of grid views:

  • Standard
  • Tiled
  • Dynamic

Page

Grid page header

References a WorkList-based text, defining the fixed contents of the page header for printed work lists. It will be used for the second and following pages.

Grid trailer

References a WorkList-based text, defining the fixed contents of the trailer for printed work lists.

Grid header separator

Specifies the character to be used for the separating line between the header and the first row (default is "-", blank is allowed).

Grid screen width

It is possible to specify the width of the entire grid on screen. This field allows a predictable layout and when set equal to the paper width, should result in exactly the same layout on screen as on paper.

Grid add row separator

When the option is set, the actions in the printed version of the work list will be separated by a line (consisting of the same character as the 'header separator').

Grid skip empty lines

When the option is set, the printed version of the work list will not contain lines that have no fields (i.e. tests). This allows reducing the amount of pages printed.

Note that the row label and the action header/trailer are printed completely. This means that a row having only one line with tests but 10 lines in the row label still takes 10 lines on paper.

Layout type - Outline view

In general

In Outline view, the work list is represented as a tree-like hierarchy with object, action and specimen/result output levels in increasing order of detail. Outline view is most often used to print sampling lists. Of particular importance in outline view are the template's text attributes ObjectSection, ActionSection, SpecimenOutputSection and ResultOutputSection. They describe the layout and contents of sections to be generated whenever the outline generator encounters a new record of the corresponding type.

Note

The tree-like hierarchy only appears when work lists are printed. On screens, work lists are always shown with grid layout.

Headers

First page header

References a WorkList-based text, defining the fixed contents of the first page header for printed work lists that have an outline layout. If not specified, the page header for the outline layout will be used.

Page header

References a WorkList-based text, defining the fixed contents of the page header for printed work lists that have an outline layout. It will be used for the second and following pages.

Trailer

References a WorkList-based text, defining the fixed contents of the trailer for printed work lists that have an outline layout.

Group criterion

References an Action-based, String-valued expression, evaluated when a work list is printed in outline view. For example, you want to create a new group for each new order number, ...

Each time the value of the group criterion changes, a form feed is emitted, and a group header appears as variable part of the page header.

Page group header

References an action-based text which is used in work list outline view only.

Group trailer

Allows to specify a Action-based text to appear after each group (as specified by the 'group criterion'). It is emitted immediately after the group (not at the bottom of the page).

Sections

Object section

References an Object-based text. This text is emitted for each new object appearing on a work list, printed in outline view.

References a Specimen-based text which is printed each time the input specimen of the action changes. It can be useful for generating lists for sending specimens to other laboratories.

Action section

References an action-based text.

Specimen output section

References a Specimen-based text. This text is emitted for each new specimen output appearing on a work list, printed in outline view.

Result output section

References a Result-based text. This text is emitted for each new result output appearing on a work list, printed in outline view.

Isolation test section

References an IsolationTest-based text. This text is emitted for each new isolation test output appearing on a work list, printed in outline view.

Add auto prompt

When set, procedure outputs marked as 'Auto prompt' will be added to the result section of an outline work list.

Remarks

  • If work list template entries are defined, only the corresponding outputs will be emitted. Only their existence is important for outline work lists. All other attributes are ignored.
  • Existing action outputs will always be emitted, regardless whether the procedure output is enabled or not. Auto-prompt outputs however will be ignored when disabled.
  • Contrary to grid work lists, ProcedureData nor PropertyInput records are automatically added to an outline work list.

How to put procedures on a work list template

Adding procedures to a work list template

Each work list template entry is a procedure. On the work lists, the procedures will be organized by their sequence number. For each procedure, the outputs will appear on the work list again as specified by their sequence number. So, you can manually add procedures by choosing the contextual menu function Entry on the work list template.

There is, however, another way to create a work list template which might be useful in some cases. Open the Station browser and select the contextual menu function Generate work list template. A work list template will be created containing all the procedures linked to the selected station.

Using work list template items

It is also possible to specify exactly which inputs and outputs of a procedure should appear on a specific work list, and in which row and column they should appear. This can be established by using the WorkListTemplateItem table.

For a given Work List Template and Work List Template Entry, this specifies an output, input or procedure data item of the Work List Template Entry's procedure, together with a line and column number. Note that you can specify a line and column NUMBER (e.g. 1,2,3,...) , not an actual position (e.g. 10, 45, ...).

When AT LEAST one work list template item has been specified, this has the following consequences:

  • When generating a work list, only actions having at least one output belonging to the work list template items will be selected.
  • When editing or printing a work list, only inputs, outputs and data having a corresponding work list template item will be shown. The corresponding fields will appear in the row and column as specified by the work list template item. Note that this positioning will only function when not using the 'Dynamic' grid mode.

How to add extra information or historical results

By specifying procedure data for a procedure, you can add more information (properties which output of the procedure) or historical results on your work lists. Use the contextual menu function 'Procedure data' on the selected procedure to add one or more procedure data records.

Seq no.

The sequence number will determine where the procedure data is displayed on the work list. In the example above, the previous result (history index = 1) for the property 'Chol' (sequence = 36) will be displayed right after the most recent 'Chol' assessment (sequence = 35).

Property

The property field indicates the property of which you want the (previous) result to be displayed on the work list.

History index

The history index can be used to display a historical value. Procedure data with history index 0 (meaning: the current value of this property) are also shown when the related result still has status 'Initial'. This allows to show a placeholder (and later the value) for a calculated result of the same order.

Show date

If the Show date option is set, the object date will be shown next to the result value.

Patient history on work lists

It is possible to show previous results of patients on work lists without having to add procedure data to the procedure definitions. Simply specify the number of historical actions in the 'Grid History Count' field in the work list template definition.

To search for a 'historical' order, the result history of the property of the FIRST COLUMN of the work list is used. Thus: a previous order is only added as history action when it contains a request for the property of the first column of the work list.

When found, all results of the thus found order which are configured on this work list, are gathered in a temporary 'virtual' action and added after the original action. The cells of the virtual action are disabled and the action's row label does NOT have a sequence number.

Important: To distinguish between a virtual history action and a real action within MISPL, the virtual action's Status field is set to -<virtual action number>.
