Encounters

Contents: Introduction - Fields - Query - Encounter type - Close old encounters - Failed deletion of encounters

Introduction

Encounters represent the contact between a person and an institution for observation or treatment.   It can be used as an administrative grouping level of hospital laboratory results. Each encounter can have a number of stays.

In general, encounters are automatically created via a connection with the Hospital Information System (e.g. HL7 ADT, HPRIM ADM messages). 

Tip

  • Although possible, we recommend not to create or update encounters manually in GLIMS.  This might lead to problems in the HIS communication.
  • For one patient, encounters should not overlap.

Fields

Person

Person involved in this encounter.

Institution

Institution where the person has the encounter.

Note

When the Institution of an Encounter changes, the electronic reports of all orders of the encounter of the last month will be marked with Needs checking (unless the option Only result changes is activated). This will cause the electronic reports to be resent.

External id

External identification of the encounter.

Note

When the External id of an Encounter changes, the electronic reports of all orders of the encounter of the last month will be marked with Needs checking (unless the option Only result changes is activated). This will cause the electronic reports to be resent.

Type

Refers to an encounter type (hospitalized, ambulatory)

Note

When the Type of an Encounter changes, the electronic reports of all orders of the encounter of the last month will be marked with Needs checking (unless the option Only result changes is activated). This will cause the electronic reports to be resent.

External info

Physician

Start - End

The start / end date and time.

End date and time calculation

The encounter's end date and time will be (re)calculated:

  1. when an encounter is created
  2. when the start date / time changes
  3. when the type of the encounter changes

Note

Once the end date and time is explicitly specified, it will never be recalculated.

Automatic

Indicates whether the encounter has been created automatically.

Encounter type

Via its Type field, each encounter refers to an Encounter type. Encounter types can be accessed from the main menu via Start -> Configuration - Encounter types.

Mnemonic

Description

Residence type

Hospitalized or ambulatory.

Tip

The browser which shows all patients currently stationed in a specific ward completely ignores encounters with an ambulatory residence type.  This implies that a patient will be considered as present in a ward, even when a more recent non-hospitalized encounter has already been created. 

The same logic also applies to the Person.Stay MISPL function and the Stays of Ward browser.

Close old encounters

Introduction

GLIMS offers a tool for closing old encounters.  It is available from the main menu via Start -> Routine -> Encounters -> Close old encounters and allows closing encounters with a start date older than a specified date and which still have no end date.

This program is restricted system managers. A password confirmation is required. 

Which encounters should be closed

Residence type

Only close encounters with the specified residence type.

Encounter type

Only close encounters with the specified encounter type.

Institution

Only close encounters from the specified institution.

Started before

Only close encounters with a start date older than the specified date.

Setting the end date when closing the encounter

Set end date <x> days after start date

Specify the number of days between the new end date of the encounter and its start date .

Set end date <x> days after last order

Specify the number of days between the new end date of the encounter and the date of the latest order referring to this encounter.

Failed deletion of encounters

HIS systems can send a message to GLIMS indicating that a given encounter must be deleted.  If so, GLIMS will first try to delete all stays referring to that encounter.  If one of these stays cannot be deleted, this will be traced in the stay.  This also implies GLIMS cannot delete the encounter.

If GLIMS cannot delete the encounter, the following actions will be taken:

  1. The encounter's End time is set to its Start time.  If the start time is unknown, the event time is used to fill both the start time and end time.  If no event time is specified in the message, then the current date/time is assumed.
  2. The following site attributes are created, set and filled:
    • Cancellation error (_AnnulationError): reason why the encounter could not be deleted
    • Cancellation time (_AnnulationTime): time at which deletion was performed
    • Cancellation source (_AnnulationSource): translator name which sent the message
    • Cancellation user (_AnnulationUser): user by which the concerned communication broker is running
