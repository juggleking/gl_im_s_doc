Property classifications

Introduction

Definition and usage

Property classifications allow to classify properties in a hierarchical structure. This structure will be used to visualize properties and results in

  • Result consultation and validation screens
  • Result reporting

Property classifications can be accessed from the main menu via Start -> Configuration -> Properties -> Property classifications.

Default property classification (used for result consultation and validation screens)

GLIMS allows to define several property classifications for different purposes, but there must be one Default property classification that contains ALL properties. If not, not all properties will e.g. be displayed in the validation screen.

To mark a property classification as Default, select it and choose the contextual menu / ribbon item Set as default. The first column in the property classification browser indicates if it is the default.

Tip

GLIMS also allows to define a department-specific property classification.

When a user logs in with a specific Department, GLIMS will check if a property classification is specified for that Department. If so, the department-specific classification will be used instead of the default classification.

Moreover, if the Department option Class. is fixed is enabled, users will not be able to change the property classification in the consultation and validation screens.

Property classifications for reporting

Property classifications are also used to define the structure and content of result reports. The Property classification can be specified in the applicable Report template.

Editing a property classification

To define the content of a property classification:

  1. Select a property classification (Start -> Configuration -> Properties -> Property classifications).
  2. Choose the contextual menu / ribbon item Edit

The property classification will be shown as a tree view.

A property classification consists of different Property classification nodes.

Tip

Titles can be defined by creating a property classification node with a Label but without Property.

Property classification nodes - Fields

General

Label

Used to define a title which can be printed on reports. Allows to use a dynamic text that will be expanded on the table specified in the Base table field.

Examples

Title node

Order info: {.ReportInformation}

With Base table = Order.

Property node

Salmonella {< "(" + .Specimen.Material.Mnemonic + ")"}

Property

Allows to select the actual property. If the Label is not specified, the property's Description will be displayed on the report

Content

Allows to define a MISPL expression that will replace the actual result value. Its base table is specified in the Base table field.

Note

Formatting for text reports.

The content function has to take care of any formatting that has to occur. For instance, if the text is too long to fit on one line, one may use the FitText MISPL function to break up the text in lines which do fit.

The column overflow protection mechanism which automatically cuts off any line which does not seem to fit on the page (replacing the removed text with an "@"-sign) is not active for content nodes.

The text of the content function is put on the report without any change.

Note

This field is also supported for electronic reports.  However, in case of microbiology results a distinction must be made between structured and non-structured microbiology results (as set in the report template).

Non-structured microbiology results

The result value will be replaced by the return value of the content MISPL (Base table: Result)

Structured microbiology results

Result Property type <> Antibiogram/Culture

The result value will be replaced by the return value of the content MISPL (Base table: Result)

Result Property type = Culture (Culture Result Property)

The Isolation Appraisal Mnemonic/Description/Text will be replaced by the return value of the content MISPL (Base table: Isolation)

Result Property type = Antibiogram (Antibiogram Result Property)

The Antibiotic MIC value will be replaced by the return value of the content MISPL (Base table: AntibioticResult)

Example

If a Result-based site function of data type String is defined for a property classification node that refers to a property (defining this site function for a non-property classification node does not have any effect) and the site function returns the unknown value ("?"), then the result will not be reported. This also applies for electronic reports.

For instance: RETURN IF .Status <= ResultStatus["Confirmed"] THEN ? ELSE .RawValue;

Base table

Base table for the dynamic text / site function defined in the Label and Content fields.

Note

Supported tables are: Result, Specimen, Order, Object, Report, PropertyClassificationNode and Property. Any other table will be ignored and the Result table will assumed. If no table is selected, again the Result table will be assumed.

Reportability

Reportable

Indicates the conditions that need to be fulfilled before this node is displayed on reports. See also: Visibility and reportability of properties.

When requested

As soon as the property has been requested.

When complete

The results linked to this node (and, in case of a title, all its sub-nodes) will only be reported when complete.

Note

If this option is set to When complete for Title nodes, and not all underlying nodes are complete, the properties will be included in the report, but ALL result values will be hidden (ReportTemplate.EmptyResultText will be shown).

In other words, the underlying result values will only be reported if ALL of them are complete.  This applies to both text and electronic reporting.

Tip

A result is considered to be Complete when its status reaches the (Microbiology) Completeness status as defined in the Default report configuration.

Always

The node is always displayed on reports, regardless whether a result is requested and regardless of its status.

Report condition

This field can be used if the node should only be reported in specific conditions. It allows to specify a Logical, Result-based Site function. If the latter returns True, the node will be reported.

Tip

The Reportable field must be set to Always, else the Report condition will not be checked.

Example

GLIMS allows to define different request codes for the same property. Depending on the request code used, you can report the property in a different chapter.

Repeat per specimen

If enabled, this node has an exclusive row per specimen. Example: for microbiology reporting, where the same property might appear several times in an order, but for different specimens.

Note

This function is only taken into account in the case of

  • text reports.
  • online reports, if the specimens have material variables or discriminators.

Previous results

GLIMS allows indicating which previous results should be printed on cumulative reports using the Default report option Previous results.  However, this option is always applied to all chapters of the report.

To refine this feature, it is possible to override the default report settings for specific chapters (property classification nodes) using this option.

?

GLIMS will check the Default report option Previous results.

None

Previous results will never be reported for properties within this node.

By chapter

Previous results are only printed if the current order has a result for a property within this node.

By property

Previous results are only printed if the current order contains the same property.

  1. The Default reporting option None always has precedence over the classification node settings !
  2. If not specified, a classification node inherits the Previous result settings from its parent node.

Flush comments after

If enabled, comments will printed immediately after the current node (without foot note reference when it concerns a non-cumulative result).

GLIMS will not use the general footnote numbering method for properties marked with 'Flush comments after'.  Instead, a reference like (*), (**), ... is used.  This has the advantage that such immediately flushed comments do not influence the numbering scheme for the foot notes which are printed at the end of the chapter or the end of the report. Moreover,  it reduces the danger that the reader interprets such a numbered foot note reference as a result value.

Tip

It is possible to have such immediately flushed comments printed in the result value column, as opposed to printing them at the left margin of the report. To do this, enable the Report template option Comment in value. 

When this option is enabled for a Title node, the comments of all the results in the corresponding (sub) chapter are buffered. As soon as the chapter corresponding with the marked node is finished, the comments belonging to this chapter are flushed (with a footnote reference).

Layout

Page break before, Page break after, Keep lines together, Keep with next

These options are used to decide which paragraphs are to be printed together on the same page and where a page break is to be inserted.

Page break before

If enabled, the classification node will be preceded by a page break.

Page break after

This option indicates whether a page break should be inserted before the next node of the same level.

Keep lines together

  • If set, there will be no page breaks within the paragraph. If the paragraph doesn't fit entirely on the page, it will be put on the following page.  If this option is set for a 'title' node, the program will try to keep the title and all its sub nodes together.
  • When a result, for which Keep lines together is enabled, is followed by a comment paragraph (triggered via Flush comments after option on the property classification node), a Method or an SOP paragraph, the program will try to keep the result and these extra paragraphs together.

    • Keep with next

    If set, there will be no page breaks between this paragraph and the following one.  If this option is set for a 'title' node, the program will try to keep the title and all its sub nodes together.

    Moreover, if the next chapter of the same level in the classification is actually present in the report, the program will try to keep this chapter and the next chapter together.  

    On ANY node, this option is only respected when the next node of the same level (and the same parent) of the property classification is actually present in the report. 

    Tips

    • The global idea is to let the Keep with next and Keep lines together options be interpreted on a chapter or paragraph level. In other words, Keep lines together on a chapter title states 'Keep the lines of this chapter together', while Keep with next on a chapter title states 'Keep this chapter on the same page as the start of the next chapter when this chapter is present in the report'.
    • The options have a slightly different behavior for 'property' nodes and 'title' nodes.  The lines of a title node are never separated from the first property node below this title (even if the first property is separated from this title by a subtitle). This implies that it is not necessary to specify KeepLinesTogether and KeepWithNext on a title node to achieve this effect.
    • To ease the configuration of these options, you can use the contextual menu /ribbon item Reset keep with next and keep lines together on a property classification.  This function will reset the options Keep with next and Keep lines together of all property classification nodes within the selected property classification.

    • When activating the Detail in the tree editor, the nodes will be marked with appropriate codes like KLT, KWN, PBB, PBA when the KeepLinesTogether, KeepWithNext, PageBreakBefore or PageBreakAfter options have been set.

    Additionally, to facilitate examining the actual behavior on reports, the report generator will add the same codes (but translated to the paragraph level) to the generated output when the header of the report contains the string "!DEBUG!". 

    For instance, a classification defined as follows:

    Widow / Orphan control

    If enabled, the first line of the paragraph will not be printed by itself at the bottom of the page (Orphan) and the last line will not be printed by itself at the top of the page (Widow).

    Physical line count

    This option can e.g. be used to indicate that a paragraph takes no (0) lines because it is printed on the right hand side of the sheet; or to indicate the number of lines a graphic chart takes.
