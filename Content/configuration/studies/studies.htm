Studies

It is possible to assign an order to a study, by choosing a study in the order entry program or by using the function Order.OrdersOfStudy. A study is a type of correspondent.  The study browser can be accessed through the main menu via Start -> Studies.

MISPL

Study.GetNextStudyObject

The MISPL function Study.GetNextStudyObject allows retrieval of the different members of a study, while Study.GetNextOrder allows retrieval of the different orders within the study.

Order.SetStudyEpisode("<Value>")

The MISPL function Order.SetStudyEpisode("<Value>") allows to set the study episode of an order to the specified value.  The MISPL function returns a logical value to indicate if the function succeeded.

The MISPL function performs the following checks:

  1. If there is no study attached to the order -> Study episode can not be set.
  2. If the status of the order is closed or discontinued -> No changes allowed, so study episode cannot be set.
  3. If the entered value > 10 chars -> Study episode cannot be set.

Study examples

  • For a liver donor screening: Study where Study.Type = Donor and Study.Object = acceptor. 
  • For a genetics study: Study where Study.Type = Genetics and Study.Subject = the disorder we are looking for.  If there is a start patient, the Study.FirstOrder and Study.Object can refer to it. The StudyObject.SeqNo and StudyObject.StudyObjectRelation should be filled to identify the relation between each study member and the study patient. Optionally, StudyObject.Role can indicate the role of the object within the study (e.g. Carrier or Control).
  • When an order for the testing on a transmittable disease is positive, often one wants to create sub-orders for all the persons the original person came in contact with.  Via the use of Order.StudyOrders() function, a study can be created on the fly, immediately allowing browsing or inserting of other orders related via the same study. MISPL navigation between these orders is accomplished via the above-mentioned MISPL function.

During order entry

When a study is chosen in the order entry program, then the request code of the study is copied to the request entry field. The order issuer (not the study issuer!!) specified in the study will be proposed as issuer for the order.

Setting the request code and the corresponding issuer is also done when automatically choosing a study while selecting a patient whose previous order was a study order (supposing the 'Default Study' option in the order entry options is set to 'Previous').

When a study is specified for an order, and the study correspondent has issuer-specific request definitions (where Requestable.Issuer = Study.Correspondent), then the study specific request definitions will be used instead of the order issuer specific request definitions (where Requestable.Issuer = Order.Issuer).  In other words, when a study is specified and there is at least one active study-specific request definition for this study, then the issuer-specific request definitions will be ignored.

The study episode field is used to differentiate the different orders for the same patient for a specific study. Although it is a character field, it will typically hold the visit number of the patient within the context of the study.

Order.OrdersOfStudy

The function Order.OrdersOfStudy offers the following functionality when called:

  1. If the Order does not yet refer to a study, the user is asked if he wants the study to be created automatically. If so, a study is created where Study.Object = Order.Object and Study.FirstOrder = Order.Id.
  2. If a study was created this way, the user is offered the possibility to complete the study definition before continuing. He might e.g. want to set the correct study type or specify the study subject.
  3. Next, a browser is called containing the orders already connected to the study. If the study is newly created in the previous step, the original order will be the only one visible in the browser.
  4. The user can browse though the orders of the study and insert new orders in the standard way. When inserting new orders, they are automatically connected to the study.

Issuer combinations

When specifying a study in the order entry or the order series creation program, the same IssuerCombination mechanism as for the issuer is used, providing no agent has been specified so far.  This implies that if there is exactly one Issuer Combination where IssuerCombination.Issuer = the study, then IssuerCombination.Agent is used as default agent. When the study has more than 1 Issuer Combination, a selection list automatically pops up.

When choosing an agent (either manually of via the previous mechanism), this agent is automatically promoted to Identification source for the patient and the issuer identification field, providing this Identification Provider is not yet set and providing this agent is the source of at least one identification.  If the last condition is not met, but the Study itself is an identification provider, then the Study is promoted to Identification Provider.

Reporting

When an order is assigned to a study, this has important effects:

  1. This order will no not be added as cumulative column on a report belonging to an order which is not of the same study. This implies that such an order will not be added to normal object, encounter or year reports.
  2. When scheduling reports for this order, a decision is made on which default reports have to be examined:
  • If Order.Study.Correspondent has at least 1 own default report, only the default reports of this correspondent will be evaluated. This is what we recommend as method of defining the reports for studies.
  • If not, we use the same mechanism as for normal orders:
  • If Order.Issuer has at least 1 default report with 'Usage=Standard', only the default reports of Order.Issuer will be evaluated.
  • If not, the OwnOrganization.Correspondent default reports, plus any 'Usage=StandardCopy' default reports of Order.Issuer are evaluated.
  • When Order.Study <> ?, the Order.Encounter.Institution.Correspondent default reports are NOT evaluated (as opposed to normal orders).

Note

A Study.Correspondent can also define his own default reports with 'Usage=Screen' for use as screen report !

  • A Report target type Study is available.  This indicates that the Study.Correspondent will be the target of the report. Such  default reports are always ignored when Order.Study = ?.
  • A Report Scope Study is available.  Such a scope will create a single Report Group for all the report pages of the same study, regardless of the patient. For each order, a separate Report Page will be generated (like for an Order-Scoped report). This Study report scope can be interesting when you want to gather all the reports of the same study in one single report group. We do not suggest that you MUST use this scope. A report for a study order may well be accomplished via a normal order-scoped report, taking into account that no orders not belonging to the same study and patient will be shown in the cumulative columns of the same report!

Billing

  • A ReimbuserType/PayerType 'Study' is available. This implies that the Study.Correspondent will be the reimburser/payer of the invoice. The same holds true for the Order Set Party Type: this can also be set to 'Study'.
    This allows creating order sets, invoices and invoice summaries specific for this study. Note that the Study.Correspondent.Payer field still allows to finally send the invoice summary to a paying organization in stead of to the Study.Correspondent.Address itself.
  • A Study.Correspondent can have its own Payment Agreements, in which you can freely use the ReimburserType = Study setting. The Order Set Determination MISPL can easily be adapted to select these payment agreements when Order.Study <> ?.

  • When the following conditions are met:

    • The order is linked to a study.
    • The study type is "Donor".
    • The study mnemonic is identical to the encounter number of the study object.

    then the study mnemonic will be sent as encounter number (as opposed to the encounter number of the order), and the study object related PIN number will be sent during billing export (FinancialShipment.Send). This functionality is currently supported by the Infohos, TripleP and ACC protocols.
