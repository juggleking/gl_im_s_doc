Ambiguity issue for multi-material procedures

When a procedure has more than 1 input material, the following ambiguity issues have to be tackled:

  • The Result.Specimen reference can refer to only one of the input specimens
  • Printing 'the' input specimen bar code: only a single specimen input is chosen
  • Deciding if a result can be derived from a root material: each input material is allowed as root specimen material
  • The combination of the latter 2 problem could yield a third problem in a way that the program erroneously decides NOT to reuse existing results or specimens in the same order because the 2 decisions above lead to a different choice of root material

In order to avoid such problems, the following decisions are taken:

  1. When searching 'the' input material of a procedure or 'the' input specimen of an action, only the input with the lowest ProcedureInput.SeqNo is considered and the other inputs are ignored. This makes the choice of 'primary' input configurable and predictable.   
  2. When checking if a result can be derived from a specific root material, the program only considers the lowest ProcedureInput.SeqNo inputs in its traversing of the procedure tree (in order to find a road from the material to the property).

Note that this only applies to ambiguous situations in which 1 input material must be selected from several (e.g. by means of MISPL functions that the select 'the' input specimen of an action or 'the' specimen of a result).  When e.g. printing labels in order entry, all labels will off course be printed.
