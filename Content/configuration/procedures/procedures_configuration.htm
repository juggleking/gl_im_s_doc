Introduction

Procedures are models of basic laboratory activity, i.e. they store models for actions that can be performed in the laboratory. A procedure is typically defined by its inputs and its outputs, each either material or property. It is however also possible to define procedures without input or output.

Examples

The procedures are the rectangles. To the left of the procedure are its inputs, to the right are its outputs. Note that different symbols are used for material and property channels.

Procedures are available from the main menu via Start -> Configuration -> Procedures.

General fields

Name

Used to visualize a procedure in GLIMS.

Code

Alternative identification used where few space is available.

Station

Station where the procedure is to be executed.

Available

Must be enabled in order to activate the procedure.

Seq no

Used for sorting procedures and derived actions.

Examples

  • Sorting of actions when Editing results and specimens of an Order.
  • Sorting actions on Work lists.
  • Sorting actions in Specimen scan programs.
  • Sorting procedures in Station - Edit procedures
  • Sorting procedures in the pathology work screen

Main

Department

Determines the department responsible for carrying out the procedure. This department will be stored with all results obtained by this procedure. Among other this is used to:

  • To display the validator signature on reports.

  • To determine the executing class in the billing module

Specimen scope

When requesting several tests that can be executed on the same specimen, you can choose whether the action scheduler should create a new specimen record rather than reusing another specimen of the same order (but which for instance will be used on another station).

For more information, please check the Order entry - Specimens.

Note

It is not recommended that procedures with an isolation test as output have a specimen scope. If they do and the Procedure.SpecimenScope differs from the Specimen.Scope, the procedure will never be scheduled.

Responsible

Indicates the user that is responsible for this procedure. This will e.g. be the person who is responsible for auto-validated results.

Turnaround time

Time (expressed as a day fraction) required to execute the procedure. Can be used in the result statistics module.

Destructive

Should be set when the procedure destroys its material inputs, so that they are no longer available as input for any other procedure.

Auto-start

Can be set to automatically promote derived actions from status Pending to status Outputting.

Realization trigger

Allows to specify an Action-based, Logical-valued site function which is executed when all outputs of an action are realized.

SOP

Reference to an external document describing the associated Standard Operating Procedure.

For more information, please check chapter Hyperlinks.

Allow replacement

Allow to move action outputs to another specimen being queried by an instrument.

Comment

Allows specifying any additional comment about the procedure.

Cell counter

Counted property / Count cyle / Count limit / Decimals

Please check the chapter Cell counter.

Microbiology

Don't confirm/validate with microbiology

Please check the chapter Microbiology - Confirmation and validation.

Genetics

Type

Indicates in which phase of an approach plan the procedure is used: Pre-analytical, Analytical, Post-analytical or Validation.

Used for retesting

Indicates whether the procedure is used to retest CNVs or sequence variants.
If the procedure is used to retest variants, make sure it has output(s) configured for that purpose.

Selection

QC only

Should be enabled when this procedure is restricted to QC purposes.

Object template

Used during action scheduling: when an object template is specified in the procedure and an order's object also has an object template, then these must match.

Note

If an order for an object without template is entered, then procedures with an object template can be selected.

Eligibility

Allows specifying a Request-based, Logical-valued site function. It is evaluated during action scheduling to determine whether the procedure is eligible to satisfy the request.

Inputs/outputs

Shows all procedure inputs and procedure outputs linked to this procedure.

Data

Shows all procedure data linked to this procedure.

Sorting targets

Shows all sorting targets linked to this procedure.
