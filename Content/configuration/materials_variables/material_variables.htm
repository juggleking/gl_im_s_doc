Material variables

Introduction

A material variable provides more information on the material or on the sampling conditions. It is for instance possible to make a distinction between urine-midstream and urine-catheter. In both cases, the material is urine, but the sampling conditions differ. The same material can be sampled at different places, e.g. from the left knee or right ear.

GLIMS allows to define one or several variables for each material. Thanks to these variables, you can limit the number of materials to be created, for instance urine, pus etc. For each material, a variable can be created; for instance 'origin' for urine or 'sampling' for pus.

Fields

Select the material for which you want to create variables and use the contextual menu/ribbon item Variables.

Name

Used to display the variable.

Enabled

If a variable is not enabled, it will not be available during order entry.

Auto prompt

If enabled, the user will be actively asked to specify a value during manual order entry.

Mandatory

When requesting a material with a mandatory variable during manual order entry, the user MUST enter a value.

The name of mandatory material variables are followed by an asterisk: Color*.

Sequence number

Sequence in which the variables are shown during order entry.

Material variable choices

The material variable screen has an embedded browser with material variable choices. For each material variable you can define a number of possible choices from which you will be able to choose one as value for the variable during order entry.

Mnemonic

Short, unique identifier. Cannot contain spaces, quotes nor backward slashes.

Value

Shown in the list of available values for a material variable.

Enabled

Only enabled choices will be available when selecting a value for the material variable during order entry.

Seq no

Sequence in which the choices are presented to the user.

Auto prompt on extra info

During order entry, specimen variables can be assigned automatically to specimens by using request definition variables. This requires no manual intervention from the user (see below).

If the Auto prompt on extra info option is enabled, and this material variable and material variable choice are automatically added via a request definition variable, the user will be asked to specify extra information. This extra information is then saved in the Extra info field of the specimen variable.

Order entry

Manually entering values for material variables

When a specimen is explicitly requested during order entry, GLIMS will check if the material has material variables.

  • For Mandatory variables, the user will be obliged to enter a value.
  • For Auto prompt variables, GLIMS will actively ask for a value.

Note

The user is only prompted for material variables of explicitly requested specimens (including an explicitly requested microbiology procedure's material). GLIMS will never prompt for material variables of implicitly added specimens.

The specified value will be stored as Specimen variable.

Using request definition variables

For each request definition, you can define a number of request definition variables. These variables can automatically be added to the corresponding specimen as specimen variables.

Using this functionality, you no longer have to specify the material variables manually during order entry.

Select a Request definition and use contextual menu/ribbon item Variables.

Note

During order entry, when checking if the request definition has any request definition variables to be added to the specimen as specimen variables, both the original request definition (which may be a panel!) and the low level request definition are examined.

Any variables of both request definitions are added to the requested specimen, but the panel request definition has priority.

In case of a panel request definition with variables, the latter will only be applied if the panel also contains the material request definition.

Note

When the option MicrobiologyRequestable.AddMicroscopy is enabled, only the low level base request definition is examined (the panel request definition does not have such a field).

If a panel contains different microbiology request definitions (referring to different microbiology procedures), GLIMS will assume that these are to be executed on the same material. If a panel contains the same microbiology procedure twice, GLIMS will assume that these procedures are to be executed on two different samples.

About duplicate requests / specimens

General principle

Some rules and checks that are applied to specimen discriminators are also applied to material variables. In a way, material variables are considered as a more elaborate system of specimen discriminators.

Definition of "duplicate requests"

When a request is added to an order (either manually or electronically), and the request definition contains material variables, then the request is not always rejected if the order already contains a request with the same request definition.

GLIMS compares the material variables of the new request with the material variables of the existing request:

  1. If for at least 1 material variable there is a difference in value (a "mismatch": this means that both requests have a value and those values are different), then the new request IS allowed to be added, because it is considered to be different from the request that existed in the order.

    Example

    • Order contains request "Ur-GME" with variable "Method=Midstream".
    • User tries to add a request "Ur-GME" with variable "Method=Catheter".
    • Consequence: GLIMS will allow this.

  2. If there is no real mismatch between the variables of the existing request and the new request, then the new request is refused.

    Example

    • Order contains request "Ur-GME" with variable "Method=Midstream".
    • User tries to add a request "Ur-GME" without variable "Method".
    • Consequence: GLIMS will refuse this (message "Duplicate request").

    Example

    • Order contains request "Ur-GME" without variable "Method" (Method=?).
    • User tries to add a request "Ur-GME" with variable "Method=Catheter".
    • Consequence: GLIMS will refuse this (message "Duplicate request").

Note

This principle is applied to each of the material variables, if there is more than one.

Action scheduling

When scheduling a new request, and the order already contains one or more specimens, then GLIMS also looks at the material variables of the specimen(s) and those of the new request definitions.

A request will not be scheduled on a specimen for which there is a "mismatch" or "conflict" in one of the material variables. The definition of "conflict" is the same as above.

Example

  • Order contains a specimen "Ur" with variable "Method=Midstream".
  • User adds a request "Ur-GME" with variable "Method=Midstream".
  • Consequence: GLIMS will schedule this request on the existing specimen.

Example

  • Order contains a specimen "Ur" with variable "Method=?".
  • User adds a request "Ur-GME" with variable "Method= Catheter".
  • Consequence: GLIMS will schedule this request on the existing specimen.

Example

  • Order contains a specimen "Ur" with variable "Method=Midstream".
  • User adds a request "Ur-GME" with variable "Method=?".
  • Consequence: GLIMS will schedule this request on the existing specimen.

Example

  • Order contains a specimen "Ur" with variable "Method=Midstream" and a specimen "Ur" with variable "Method=Catheter".
  • User adds a request "Ur-GME" with variable "Method=Catheter".
  • Consequence: GLIMS will not schedule this request on the first specimen (with variable "Midstream"), but on the second specimen (with variable "Catheter")

Example

  • Order contains a specimen "Ur" with variable "Method=Midstream".
  • User adds a request "Ur-GME" with variable "Method= Catheter".
  • Consequence: GLIMS will not schedule this request on the existing specimen, but it will schedule a new specimen "Ur" with variable "Method=Catheter".

Notes

  • If the user adds a request EXPLICITLY on a certain specimen, and both the specimen and the new request have material variables, then GLIMS will check if there is no "conflict" between these variables. If there is a conflict, the new request is refused.
  • If a property is requested, for which the option "Use available specimen" is enabled, then GLIMS will check if the property can be scheduled on a specimen that exists in the order AND for which there are no conflicting material variables between specimen and request definition.

Selecting specimens for reuse

When a new order is entered, the order entry program automatically searches for specimens that exist in older orders (for the same patient of course) and that can be reused for the new requests (of course, if the configuration allows this).

This automatic search looks at:

  • The material
  • The discriminator
  • The specimen scope

GLIMS also compares the material variables between the old specimen and the new request. However this comparison is more strict than the comparisons described above, which were inside the same order:

If - for a certain material variable - the old specimen has no value (e.g. the old order has a specimen "Ur" with "Method=?"), and the new request has a value for it (e.g. "Method=Midstream"), then GLIMS will NOT try to reuse that old specimen, because it cannot guarantee that the old specimen fulfills the requirement of the new request. In the example: we do not know for sure if the old specimen was a midstream urine.

The same principle applies to specimen discriminators and specimen scopes.
