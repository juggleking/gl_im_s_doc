Automation - Commands & Tasks

Content: Introduction - Commands - Tasks - Task schedulers - Interactively starting a command - Cancelling a scheduled task

Introduction

GLIMS allows to automate the execution of workflows such as report generation, printing invoicing etc.

Basic concepts:

  • A Command defines which records should be selected and which function should be executed on these records.
  • A command can be linked to a Task that defines when the command is to be executed.
  • Task schedulers are services responsible for the actual executing of the tasks.

Commands

Introduction

The Command browser can be accessed from the main menu via Start -> System management -> Commands. 

A command consists of two main parts:

Query

Selects the records that need to be processed by the function.

Function

Specifies the action that will be executed for the records selected by the query.

Fields

Table

Database table from which you would like to select records.

Description

Free-text description.

Query

Select a predefined query based on the chosen table.

Query parameter set

Allows to configure the query that you selected:

  1. Double-click to open the Actual query parameter set browser.
  2. Use the Insert key to create a new query parameter set and provide a Short description.
  3. Select the created query parameter set in the browser and choose the contextual menu / ribbon item Setup.
  4. Configure the query parameters and click OK.

Skip query setup

If enabled, the query setup screen will not be shown when executing the command interactively.  In batch, the setup screen is never shown.  

Sort function

Allows to specify a site function to sort the results returned by the query.

Function

Function that will be executed for the records selected by the query.

Function parameter set

Allows configuring the function that you selected.

  1. Double-click to open the Actual function parameter set browser.
  2. Use the Insert key to create a new function parameter set and provide a Short description.
  3. Select the created function parameter set in the browser and choose the contextual menu / ribbon item Setup.
  4. Configure the function parameters and click OK.

Skip function setup

If enabled, the function setup screen will not be shown when executing the command interactively. In batch, the setup screen is never shown.  

Notes

The pre-configuration of new function or query parameters added via a new functional GLIMS release will always be supported when the original parameters were pre-configurable.

This is accomplished by storing these parameters in a field HiddenParameters of the Actual function/query parameter set record as opposed to the Actual function/query parameter table where the originally defined parameters are stored.

Tasks

Introduction

Link a command to a task to define when the command should be executed:

  1. Select the command.
  2. Choose the contextual menu / ribbon item Show all -> Tasks.
  3. Use the Insert key to define / assign a new task.

Fields

Name  

Uniquely identifies a task record among the task records for the same command (i.e. the combination command + name should be unique).  This name would typically contain something like 'Evening shift', or 'Weekly'.  

Note

For the automatically generated one-shot task records created by the Do via task scheduler function, the name User, followed by the user's login name is used, followed by a sequence number if required.

Enabled

Please check the section Cancelling scheduled tasks below.

From

The date on which the task should be started.

Until

The date after which the task should be stopped.

Next run

Set automatically. Indicates when the next execution of this task will take place.

Tip

By manually entering a Next run, you can for instance delay the executing of a task. If you want to delay the execution of a task because you need to modify its configuration, it is better to deactivate the Enabled option.

Latest run

Set automatically. Indicates when the command was last executed.

Schedule status

Displays the TaskStatus.ScheduleStatus.

Executing

Set automatically. Indicates whether the command is currently running.

Executing service, service pid, service host

GLIMS saves the ID and Process ID of the executing task scheduler in the task record while it is being executed (including server name).

This has several advantages:

  1. For a specific task scheduler, GLIMS can look up which task it is executing.

  2. Better crash recovery: if a task scheduler stopped while executing a task, it is easier to recover tasks by checking whether there are tasks still being executed.  If so, GLIMS can also verify whether the executing task scheduler still has the same Process ID as when the task was launched. If not, the task can be recovered and marked as "Executing = No".

  3. For a task being executed, GLIMS can verify if the process id of the executing task scheduler is still identical to the one recorded when starting the task.  If not, it can be marked as "Executing = No".

Recurrence

Determines the periodicity of the task execution. Possible values:

Instantly

Task to be executed at a certain scheduled minute interval, on certain days, between the schedule start and end date. Select the appropriate day by marking its toggle box (see below).

Example

Schedule a task each Monday, every forty-five minutes between 06h00 and 18h00.

Daily

Task to be executed once every day at a pre-defined moment, starting from the specified start date.

Example

Schedule a task every day at 09h00.

Weekly

Task to be executed at a scheduled week interval, on certain days of the week at a certain time, starting from the specified schedule start. Select the appropriate day by marking its toggle box (see below).

Example

Schedule a task every two weeks, each Monday, Wednesday and Friday at 12h00.

Monthly

Task to be executed at a certain scheduled month interval. The day within the month is determined by the Day type and Day type in month (see Recurrence pattern).

Example

Schedule a task every two months, every second Friday of the month. The Schedule start field defines the start time.

Yearly

Task to be executed at a certain scheduled year interval on a specified month. The day within the month is determined by the Day type and Day type in month (see Recurrence pattern).

Example

Schedule a task every two years, in January, every second Friday of the month.

On holiday

Indicates what should be done when the command is to run on a holiday:

Ignore

Ignore the holidays and execute the task as scheduled.

Skip

The task will not be executed on holidays. The schedule will be recalculated.

Shift backward

Task is to be executed the day before the holiday.

Shift forward

Task is to be executed the day after the holiday.

Recurrence pattern

Start

Time within the scheduled start date when the scheduling task should be started.

Until

Time within the scheduled end date after which the task should no longer be executed.

Every x ...

Determines the schedule interval, i.e. the time lapse between consecutive runs of the task.

Note

The unit of this interval depends on the Recurrence. It is minute for instantly tasks, day for daily tasks, weeks for weekly tasks, month for monthly tasks, year for yearly tasks.

Example: if the Recurrence is weekly and the schedule interval is 2, then the task will be executed every two weeks.

Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, Sunday

Allows to select a specific day (or several days) in the week on which the task should be executed.

Task schedulers

Introduction

Tasks define when a command is executed. The actual execution of a command is performed by a task scheduler.

In GLIMS, a task scheduler is defined as a service of type Task scheduler.

Which task scheduler executes which command

GLIMS allows defining several task schedulers. When you define only one single scheduler, you define only a single queue for task execution. This implies that tasks may effectively execute later than their scheduled time.

Without scheduler assignments (not recommended)

If you have not defined any scheduler assignments (see below), tasks can be executed by any of the defined task schedulers. Any free task scheduler will pick up the next task to be executed.

Note

Without any scheduler assignments you have no control over the execution of the defined commands. A heavy command (e.g. result statistics) may delay other important commands such as report generating, financial exports, ...

We strongly advise to use scheduler assignments (see below).

With scheduler assignments (recommended)

To have more control, you can define a link between a command and one or more task schedulers. This implies that the command will always be executed by one of the assigned task schedulers.

  1. Select a command.
  2. Select the contextual menu / ribbon item Scheduler assignments.
  3. Use the Insert key to create a new scheduler assignment.
  4. In the Scheduler assignment window, select a task scheduler service in the Task scheduler field.

Tip

Assume that Command A has a scheduler assignment with Task scheduler 1 and that Command B does not have any scheduler assignments. In that case, Command B may still delay Command A as it can be executed by any task scheduler, including Task scheduler 1.

To avoid such situations, you can enable the general option Strict scheduling (Start -> System management -> Customize -> MATE). If this option is enabled, commands without scheduler assignment will never be executed! All commands to be executed should have scheduler assignments.

Task scheduler fields

In GLIMS, a task scheduler is defined as a service of type Task scheduler. Important fields:

Type

Must be set to Task scheduler

Startup options

A service of type Task scheduler does not require any startup options.

Port

An unoccupied port of the server

Host

Specify the host field if the command should be executed on another computer than the GLIMS server. In that case, you also have to specify the Application root folder, the folder on the host where GLIMS is installed.

For detailed information on all fields, please check the chapter Services.

Interactively starting a command

Commando.Do

The Command-based function Do allows to execute a command from within the GLIMS session.

Note

This function does not use the task scheduler ! The command will be executed within the current GLIMS session.

Command.DoViaTaskScheduler

The Command-based function Do via task scheduler allows to launch a command via the task scheduler (on the server).

Note

This function will actually create a Task for this command with

  • Recurrence set to Once
  • Next run set to the current time

The (possibly modified) query and function parameters are stored within the task in the Task.QueryParameters and Task.FunctionParameters fields.  When a task scheduler finds such a task, the parameters stored in the task are used for the execution of the command. When the task is completed, it is deleted.

The task scheduler executing a task that was scheduled with Do via task scheduler will log by which user and when the task was created.

gp_Site.DoCommand

This function allows to launch a command from the main menu or application-wide ribbon. It has two parameters:

Command

Allows selecting the command to be executed (by selecting the Start table and then the actual Command).

Execution method

Indicates how the command should be executed:

Local

Will execute the command from within the current GLIMS session.

Via task scheduler

Will execute the command via the task scheduler (on the server).

Via remote server

Not supported.

Notes

Notes

  • A user can only Do a command, if (s)he has the required privilege for the corresponding function. This does not apply for tasks, which are run by a user with Session type set to Batch, having a role of type System manager.
  • To protect the system against tasks which would make the task scheduler fail and stop unexpectedly, a task which is already marked as 'running' at the moment the task scheduler restarts, will only be recovered once. At the moment of recovery start, the Task.NextRun is cleared, so that if the task scheduler fails again, the task will not be recovered again upon next restart.

Cancelling a scheduled task

The automatic execution of a scheduled task can be halted.

Disabling a task

It suffices to deactivate the option Enabled on the Task record. This allows disabling a task temporarily in order to modify a part of its configuration, without changing the task's planned execution time and recurrence pattern.

Interrupting a task

It is possible to interrupt the ongoing execution of a task, using the context function Interrupt that is available on the Task table.

This works in the following way: the task scheduler checks if a task has been interrupted

  • before the execution of the task
  • while selecting the records based on the query
  • while going through the selected records to execute the function

If the task has been interrupted, the scheduler safely halts the execution of the task and moves on to the next available task.

After clicking on Interrupt, the user can also disable the task using the option of the Interrupt function, Disable the task after it has been interrupted.
