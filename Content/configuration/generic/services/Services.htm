Services

Contents: Introduction - Fields - Service log files

Introduction

Services are background processes with the ability to access the database. Services are available from the main menu via Start -> System management -> Services. 

Fields

Main fields

Name

Should not contain any characters other than letters, digits and underscores. The first character must be a letter or an underscore.

Internal language

Should be matched by the internal language of any translator communicating with the service. This should always be set to URL, except for services with Type set to Service controller which uses XML.

Type

Translator server

Translator servers are used to exchange information between the database and one or more translators.

Task scheduler

Task schedulers are used to execute scheduled tasks. 

Service controller

Used for remote start/stop of services and translators (e.g. via MIPS Watchdog) and is especially useful in multi-server environments.

Report builder

Please check the chapter Report builder service.

Message queue

Please check the chapter Message queue.

Java communication engine

Services of type Java communication engine act as a container for Java translators.

Note

About the logging options for a Java communication engine:

  • To database is not supported.
  • File refresh on max size is auto-enabled.
Notes

Note

The Service editor performs some built-in checks to prevent configuration errors:

  • When the type is set to Task scheduler, the internal language is fixed to URL.
  • When the type is set to Service controller, the internal language is fixed to XML.
  • When the type is set to Report builder, the internal language is fixed to XML and GLIMS will always log to file instead of database.

Auto start

If enabled, the service will be automatically started by the Startup services function. On UNIX systems, use the parameter -a: e.g. mam glims start -a -n.

Seq no

Determines the sequence in which services are started and stopped by the Startup services and Shutdown services functions.

Specific options

Are specified as a list with format { -<name> [ <value> ] }.

Tip

Instead of using service specific options, MIPS advises to use translator specific options (depending on the driver).

Port

Any unoccupied TCP/IP port on the server. Do not use port numbers smaller then 1024 as these are reserved for system processes. We recommend using numbers between 3100 and 4000.

Tip

To avoid double usage of port numbers, it is recommended to define the chosen port number in the operating system's Services file as well.

Service group

The service group to which this service belongs.

Tip

To easily move multiple Services from one Service group to another, use the Service-based function Change service group.

Description Page

Description

Free text description.

Logging Page

Verbosity

Defines the amount of logging detail. Info, warning and error messages are always logged, independent from the log verbosity.

Low

TCP/IP communication is not logged.

Medium (recommended)

TCP/IP communication is fully logged.

High

Byte per byte logging.

Note

Use this for debugging purposes only !

To database (not recommended)

If enabled, logging will be stored in the database instead of exported to a file.

Note

This is not recommended as it may degrade performance !

File refresh, File refresh unit

Period after which open log files should be closed and a new log file should be opened.

File keep, File keep unit

Period after which log files should be automatically deleted. If not specified, old log files have to be deleted manually. 

Note

You cannot delete a log file of a running service.

File max size

The maximal size in MB of the log file.

File refresh on max size

Determines what happens if the File max size is reached.

  • If enabled, a new log file is created.
  • If disabled, the service exits to prevent disk overflow.

Allowed to start

For more information, see Shutdown services.

Process id

Automatically set when the service is started.

Restart failures

For future use.

More page

Login user

Reference to the user on account of which the service will run. Only users with Session type set to Batch can be selected.

Application root folder

Root folder, on the Host (as defined in the service group), where the application is installed.

Note

Only for Services with Type set to Java communication engine, an Application root folder can be defined on Service level. The reason is that you cannot have multiple Java communication engines in the same root folder.

Configuration for 2 Java communication engines:

  • The first JCE can use the default folder [$APPL\JCE]

  • For the second JCE:

    • Create a new folder [$APPL\JCE2]

    • Copy the default JCE folder into the new folder [$APPL\JCE2\JCE]

    • In the Service for the 2nd JCE, specify the Application root folder [$APPL\JCE2]

Service log files

Service log files are located in automatically created folders.

Default location / file name

<log root>/svc/<service name>/<service name>_<time stamp>.log where <log root> is <application root>/log by default.

Custom location

<log root> is <application root>/log by default, but can be overridden by

  1. the Log directory of the service group to which the service belongs
  2. the Log folder in the general options (Start -> System management -> Customize -> MATE).

Log file names with time stamp

Due to automatic refreshing, multiple log files may be generated. The actual log file names will be different for each new version by suffixing them with a creation time stamp.

Log file names without time stamp

There is also a single log file without a time stamp suffix in its name.

This file captures any messages generated either before the logging environment of the process is fully set up or by default low-level error handling.

Tip

The presence of such a log file may be an indication of unexpected errors.
