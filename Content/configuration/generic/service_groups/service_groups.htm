Service groups

Introduction

In order to streamline the management of services and translators in a multi-server environment, GLIMS allows to define Service groups.

Service groups serve as centralized configuration of Host and Log directory information, and optionally allow setting an Administrator role which users will need to have in order to Start/Stop/Register/Unregister the Services and Translators which are a member of a Service group.

Fields

Mnemonic, Description

These are standard fields that allow you to clearly define a unique name and the goal of the individual Service group records.

Host (mandatory)

Host on which the services/translators in this group are running. As it is not allowed to directly modify this field after creation, a function Change host is available.

Tips

Tips

  • It is possible to configure multiple Service groups for the same Host. This allows for more logical grouping of Services and Translators in function of their goal within the communication network.
  • It is possible to configure a different Service group for the Service and its underlying Translators. This makes it clearer that the Translator processes are running on a different server than the Service process through which they communicate with the application.

Host aliases

Can be used in a cluster environment. Allows to store a comma-separated list of the cluster instance node names.

Example

Assuming there is a cluster defined with two instance nodes:

  • cluster name: mipscluster
  • instance node 1: mipsinstance1
  • instance node 2: mipsinstance2

Then you should configure the Service group as follows:

  • Host = mipscluster
  • Host aliases = mipsinstance1,mipsinstance2

Administrator role (optional)

When you set this Administrator role, users will only be able to Start/Stop/Register/Unregister the Services and Translators in this Service group, if they are logged in with that specific Role.

By default, Administrator role is set to the unknown value ? which means that no such restrictions are applied and that users are able to administer the Services and Translators within that group, as long as they have the required privileges.

Notes

Note:

This is an important feature for multi-site environments, where each site has its own IT-staff and these employees should not accidentally interrupt the routine on another site.

Application root folder

Root folder, on the Host, where the application is installed. It will be applied to all services and translators in the Service group.

Log directory (optional)

This field allows you to set a specific "root" log directory for the Services and Translators which are a member of this Service group.

By default, Log directory is set to the unknown value '?' which means that the existing 'root' log directory is applied (as specified in the general options).

Free comment

Assign services and translators to a service group

There are two ways to indicate that a service or translator belongs to a specific service group:

  1. Specify the service group in the Service group field of the individual service or translator.
  2. To easily move multiple Services and Translators from one Service group to another, use the Service or Translator-based function Change service group.
