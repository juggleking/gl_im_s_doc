Introduction

When an order is entered, GLIMS verifies which reports should be created for this order. This is called the report scheduling.  

During report scheduling, GLIMS verifies which default reports are applicable for the created order. For each applicable default report, a report will be scheduled.

Linking default reports to your own organization

First you have to specify the Own organization in GLIMS. Click Start -> System Management -> Customize -> GLIMS general. Enter you organization in the Own organization field.

The default report browser (accessible from the main menu via Start -> Configuration -> Reports -> Default reports) displays all default reports for your own organization.  Hit Insert to add a new default report.  The newly created default report will automatically be linked to your own organization.

Issuer reports

By default, reports are linked to your own organization. However, it is also possible to link a report to a specific issuer, which can overrule the reports linked to your own organization.

  1. Select an issuer (e.g. HC Provider) and choose the contextual menu / ribbon item Correspondents-> Issuer -> Reports -> Default reports
  2. Use the Insert key to add default reports for the selected issuer.

Scheduling of issuer reports

  1. During report scheduling, GLIMS first checks if the issuer of the order has default reports.  
  2. If all of the issuer's default reports are Standard copy reports, then GLIMS will also evaluate the default reports linked to the organization.  
  3. However, if the issuer of the order has at least one default report which is NOT a Standard copy report, then the default reports linked to the organization will no longer be evaluated.

Study reports

Similar to issuer reports (see above), it is also possible to link a report to a study.  These reports will overrule the ones linked to the own organization.  

  1. Select a study and choose the contextual menu / ribbon item Correspondents-> Issuer -> Reports -> Default reports.
  2. Use the Insert key to add default reports for the selected study.

For more information, please check the chapter on studies.

Default report fields

Main page

Code

Non-unique code that identifies the default report.

Scope

Imposes object time criteria on the results to be included in the report. The scope indicates WHAT should appear on the report.

Order

All results of a particular order will be displayed on the report. This is the standard report type.

Day

All results obtained for a patient during one day will be reported.

Encounter

All results obtained during the encounter of a patient will be reported. These results will be divided into different segments (columns). Placing all the reports in a row, will provide a complete overview.

Period

No longer supported

Object

Extended year reports. All results for a patient will appear on the report, without any time-related criteria.

Exclusive

Are special order reports, ruling out the creation of other reports. They offer the possibility to create special reports for specific orders suppressing the normal reports.

Specimen

All results for one particular specimen will be put on one report. E.g. An order has three different specimen => each specimen will be put on a separate report. Results which are not linked to a specimen (e.g. weight of the patient) will be included on each report.

Year

All results obtained for a patient during an entire year will be reported.

Study

Used for study reports

Ward

All results of all patients residing in a ward.

Medical record

Correspondent

Shows to which correspondent the default report is linked: own organization, issuer or study (see above).

Issuer group

If (a correspondent group) is specified, GLIMS will only schedule reports for orders that have an issuer belonging to the specified group. For cumulative reports, only the results of previous orders that have an issuer belonging to the specified group will be added.

In other words, when the issuer of an order is not a member of the specified issuer group, this default report will not be evaluated for the order and no new report will be created nor will existing reports be extended.

Please check the chapter "Default reports: use cases for some practical examples.

Active

Indicates if this default report is active. If not, it is not taken into account during the automatic report scheduling for a new order.  This allows deactivation of no longer needed Default Report records without having to write a MISPL eligibility returning FALSE.

Target type

Determines if and how the report is targeted indirectly to its recipient correspondent. If the target type is Explicit, you must also specify the actual correspondent in the Explicit target field. 

Issuer

Report will be sent to the issuer of the order.

Default

Same as issuer

Location

Definition of the target of a location report

  • For Order, Specimen, Encounter and Study reports (see "Scope" field), the target is defined as the last known location of the patient within the encounter of the report record.  
  • For Year, Object, Day reports, the target is defined as the current location of the patient as defined for the Person.Stay() MISPL function. Note that for these reports the encounter of 'the' order has no influence!

About ambulatory encounters

Some customers do not wish to take ambulatory encounters into account when determining the 'current location' of a person in order to know where to send a location report. This is typically the case when keeping ambulatory encounters open for a long time instead of closing them properly.

To cope with this situation, the general option Location report when ambulatory (Start -> System management _> Customize -> GLIMS general) allows indicating that ambulatory encounters should be completely ignored when determining the new target for a location report.

What happens after stay updates

Whenever a stay record is updated (Start time, End time or Ward), created or deleted. the existing Location reports of the patient may be retargeted:

  • The new target is defined as described above  
  • If the new target is unknown, the report target is not changed

Reports affected:

  • All Order, Specimen, Study or Encounter reports referring to the encounter of the modified stay  
  • All Object, Year (where year = current year) and Day (where Date >= Yesterday) reports of the patient.

It will make no difference if the stay is modified manually or via a hospital connection.

Notes

  • The Report.CheckStatus function will also correct the target of a location report when necessary
  • When scheduling reports, GLIMS will try to make sure that there is only one location report per report page. This implies that if a report page is already targeted to another correspondent, it will be reused and retargeted to the new ward rather than creating an additional location report for the new ward.

Family doctor

Allows sending a report to the person's family doctor.  When the object of an order is not a person or no family doctor is specified for this person, the report is not created (no need to check this via the Eligibility MISPL).

Explicit target

Must be specified only if Target type is Explicit. It directly references the correspondent who is to receive the reports.

Eligibility

References an Order-based, Logical-valued MISPL expression that is evaluated at order entry to determine whether or not actual reports must be derived from this default report. You could for example create correspondent groups with several issuers and then check on these groups in your MISPL expression.

Report template

Specifies the template that will be used for this default report.

Exclusive

If set, then as soon as one exclusive report is selected for scheduling, all non-exclusive reports are ignored. This can be helpful for sending special reports under specific circumstances.

Target-specific

Indicates if a separate report should be created for each target (e.g. issuer).  A default report for which this option is disabled is not allowed for default reports with scope Order, Specimen or Study.

For instance, for issuer-targeted object reports:

  • When not set, there will be only one object report per patient and per report template.  Actually, a report record will be created for each issuer, but only the report for the issuer of the last order in the object report will be active, i.e. its Need checking option will be enabled.  So depending on who is the issuer of the last order in the report, a different report will be activated (Needs checking option will be enabled).

  • When set, there will be a separate report record per patient per template and per issuer.  Each of these reports will be activated ("Needs checking" option will be enabled).

    Also, when this field is set, an issuer-targeted report will only contain pages which contain at least one result requested by this issuer

Add missing pages

As it is possible to send a separate copy of the report to each recipient, it is also possible to add a copy of all missing pages to a (new) recipient's report list.

When this option is enabled, upon creation of a new report targeted for a certain recipient, the previous pages of the same report group are checked, and a new report record is created for each page for which no report was sent to this recipient. This allows sending a copy of the patient's file pages that were added since a given correspondent last received a page from this report.

Note

If the option Target-specific is not enabled, then this field will not be taken into account (as this could lead to unwanted report printouts).

Usage

Determines for what type of purpose this default report is used, and when it will be eligible by the report scheduler.

Standard

This is the 'normal' case. It indicates a report that is automatically scheduled during order entry, as an original. Also, when an order for a given issuer is entered, his private Default Report configuration will only be used when at least one active Default Report with Usage = Standard is found for this issuer (see above, Linking default reports to an issuer).

Standard copy

Indicates a report that is automatically scheduled during order entry, but should be considered as a copy rather than an original. Also, after scheduling of the site's organization default reports (thus skipping the issuer's default reports), any of the issuer's default reports where Usage = Standard copy are added to the already scheduled reports.

Screen

Default report is only eligible when a screen report is generated.  

Manual copy

This indicates a default report which is only taken into account when reports are being scheduled for Order.ReportCopyReceiver (or being added manually or via MISPL).

Remember that you can specify such a report copy receiver in the order entry screen. In this case, after scheduling of the original reports, the order is rescheduled as if the copy receiver was the issuer.

In this case, the default reports marked as Manual Copy will also be eligible for scheduling. Moreover, all reports scheduled during such an operation will be marked with Usage = Manual Copy, regardless of the original default report's Usage setting.

Manual

It will not be used by the report scheduler at all.  It indicates a report that is typically added manually or via MISPL.

Electronic signature

See Digitally signing PDF reports via carte CPS for more information.

Settings page

Previous results

Determines which previous results are printed on the report.

None

No previous results are printed

All

All results of previous results within the current Property Classification are printed.

By chapter

A previous result is only printed if the current order has a result for a property from the same property classification chapter (i.e.: only print previous hematology results if the current order also contains hematology)

By property

A previous result is only printed if the current order contains the same property.

Tip

This option is always applied to all chapters of the report.  To refine this feature, it is also possible to override these default report settings for specific chapters using the property classification node option Report previous results.

Recapitulative  

If set, properties for which no previous results exist, will not appear on the report.

Note

This option interacts with the option Previous results:

  • If the Previous results option is set to By Property, and the report is marked as Recapitulative then the property node will not be shown on the report if no previous results exist for that property.
  • If the Previous results option is set to By Chapter, and the report is marked as Recapitulative then the property node will not be shown on the report if no previous results exist for any of the properties in the same chapter (the definition of "chapter" is same as non-recapitulative).
  • If the Previous results option is set to None, and the report is marked as Recapitulative then the property node will never be shown on the report.
  • If the Previous results option is set to All, and the report is marked as Recapitulative then the property node will always be shown on the report, regardless whether there are previous results or not.

Issuer restricted

Determines whether or not results requested by other issuers can appear on cumulative report.  This only applies to

  1. Default reports with scope Order
  2. Default reports with an issuer group specified, regardless of the scope.

Note

The field HCProvider.Practice is used to specify that a HC provider is connected with a specific health care practice (typically together with a number of other HC providers).  

This will automatically have the effect that HC providers belonging to the same practice can have shared cumulative order reports, even when the Issuer restricted option is set.

Material restricted

Indicates that each row in a report should only contain results referring to a root specimen of the same material (and primary material variable when applicable).

If this option is enabled, each different root material (and material variable) gets its own report row and no cumulative results are shown next to a result obtained on a primary result referring to a different material (or material variable).

Start date / additional period:

When specified, this start date will be used as start date of newly created object reports, as opposed to the default (Order.ObjectTime - DefaultReport.AdditionalPeriod)

Trigger status

If a result reaches this status, the report becomes eligible for further inspection by enabling its Needs checking field.

GLIMS implicitly distinguishes two types of report:

Status reports

Reports with a Trigger status < Available are considered to be Status reports.  This implies that each status change makes the report eligible for generation.

Value reports

Reports with a Trigger status >= Available are considered to be Value reports.  This implies that only actual result value changes or external comment changes makes the report eligible for generation.  Result status changes (except the moment that the trigger status is exceeded) will not be considered when checking if the report has to be generated.  

So a value report can only be triggered:

  • when the trigger status is exceeded (e.g. first report generation)
  • the result value or external comment has changed (and naturally the status has to exceed the trigger status)

Note

There is one exception: a Value report will also be triggered upon each result status increase when it is a Communication or Electronic report (i.e. Report.Medium = Communication). 

This differentiation between Status and Value reports also has an influence on the interpretation of the New results option in the report generation screen.

Minimal result status

Results with a lower status than the one indicated will be hidden.

Completeness status/ Microbiology Completeness status

The status in which a 'normal' result and a 'culture' result are considered to be 'complete'.  

  • This allows to query all complete or incomplete reports during report generation
  • It can be used in the {ReportStatus} variable.
  • The completeness status is also checked when the Property classification node option Reportable is set to When complete

Completeness status

When at least one of the results in the report has a status lower than the Completeness status, the report is considered as incomplete.

Microbiology completeness status

When the microbiology action of a culture result has a status lower than the Microbiology completeness status, this result (and both its culture result and the report) is considered as incomplete.

Note

When not specified, the following defaults are applied:

Status Any Validated Closed
Value Validated Validated Closed
Value Confirmed Confirmed Ready
Value Available, Initial Available ready

Note

Caution when setting the New results option in the report generation screen to Only new results or New result required:

Example 1: 

Diff panel, Reportable = WhenComplete

Eosino = 120, validated
Baso = ?, expected
Blast = ?, expected

reported with Mark output enabled and Only new results / New result required WILL NOT export a report (No new results). This is expected behavior because not all diff members are complete, so we cannot report 'Baso' nor 'Blast' (not complete) and thus we cannot report 'Eosino' despite its completeness ! 

Example 2 :

Diff panel, reportable = WhenComplete

Eosino = 120, validated
Baso = ?, expected
Blast = ?, expected

Natrium = 130, validated

reported with Mark output enabled and Only new results will export a report with only the 'Natrium' result value specified. If 'Baso' and 'Blast' become complete, all tests will be exported and the report will become complete.

Maximum print count

Maximum number of times that this report can be generated. As soon as the field Report.PrintVersion reaches this maximum, the report is deactivated (Report.Active=NO and Report.NeedsChecking=NO).

This is helpful in those cases where the number of versions of the report sent to the report target should be limited (e.g. only send once). 

Please also check the chapter Report copies.

Note

Setting this field to 0 will lead to the creation of inactive reports!

Hide cumulative comments

If enabled, previous external comments on cumulative order reports will be suppressed. 

Report discontinued results

Indicates whether or not discontinued results should be reported.

Notes

  • For Medicard, Medidoc, EDI1.0, Medigest, Imed, HealthOne and MEDVRI1.0 reports, the value of this option is always disabled, i.e. discontinued results will never be put on the report (as these reports cannot transfer result status information).

  • Discontinued results are considered as being new results, so that a report is triggered for generation even if the New results field in the Report generation screen is set to New results required.   

    However, if the latter is set to Only new results (and the Report discontinued results option is disabled) and the only new result is a discontinued result, no report will be sent.
