Stations

Introduction

A station can either be a place where tests are executed manually, an analyzer, a concentrator, ...

General fields

Mnemonic

The default visible identification of the station.

Name

The name of the instrument, place of survey, ...

Type

Analyzer

An analyzer can belong to a download group so that it can download actions scheduled on other stations with the same download group. When the inquiring station queries for actions, the outputs of the action scheduled on the other station should have at least an assessment method with the same property configured on the inquiring station. An analyzer station can be defined as client of a LAS or concentrator.

Concentrator

A concentrator station serves one or more client stations (stations of type Analyzer). The client stations normally have no translator, as all communication (result upload, patient download…) is handled by the concentrator station. Procedures with in- and outputs are defined on the client stations. GLIMS will use the in- and outbound id's of the client station's assessment methods.

Result messages from the concentrator should be received with an accompanying instrument id. GLIMS will try to identify the concentrator client station with that instrument id (station remote id).

LAS

A LAS station serves one or more client stations but will only download actions to the station.  Procedures with in- and outputs are defined on the client stations. GLIMS will use the in- and outbound id's of the LAS station to identify the assessment methods.

Results are always transmitted via the client station. The client stations must therefore have their own translator.

Counter

Stations of type Counter are used for the cell counter program in GLIMS.  A counter station can be recognized by a procedure with a cell count limit greater than zero.  Check the chapter "Cell counter" for more information.

Scan

Refers to a station for scanning specimens. A scan station does not have a translator nor a procedure.  Can for instance be used in the specimen archive module.

Manual

Represent a manual work place.

Calculator

Station used for calculated results (i.e. results where the PropertyOutput.Value contains a formula AND where the action has at least one input result).  For calculation procedures, the Value field of the property output should be specified.

Code

The station code (which is shorter than the name).  The code can be:

  • Used in the communication programs: download to analyzer
  • Used by the Specimen scope functionality

Workplace

The work place to which the station belongs.  A laboratory can be divided into different subspecialties called workplaces. For example: Biochemistry, Hematology, Immunology, etc.  A workplace can contain different stations.  An example could also be a workplace containing all Advia stations.  Workplaces can, among other, be used in:

  • Confirm by action: allows to confirm all actions performed in a specified workplace
  • Result statistics: include all results assessed in a specific workplace
  • ...

Download group

Allows grouping multiple stations so that actions of a related station could be downloaded to the inquiring station. Note that download groups are only used for stations of type Analyzer.

Scheduling availability

If not enabled, this station will not be taken into account during the scheduling phase of order entry.

Color

Allows either typing in a color in the <backgroundcolor>,<foregroundcolor> syntax, or selecting a color from a color map via ctrl-f or mouse double click.  Is currently used by the specimen scan program (Show stations option).

Run capacity

The maximum number of samples per run.  Indicates the number of samples that fit into the instrument's tray.  Among other, this is used in work list generation: the work list for a station will not contain more entries than the run capacity.

Data size

Allows the internal service to download name fields (like patient first and last name, patient address, municipality) translated or non translated.  This functionality has mainly been added for analyzers that do not support special characters.

When set to 7 bits, the name fields will be downloaded and converted to values with characters in the range a-z, A-Z and 0-9. Characters that are left unchanged are space and the '-' character.  Characters that have not a meaningful translation (such as ©, ¶, ...) are eliminated.

When set to 8 bits, name fields are downloaded as they are stored in GLIMS.

Data mapping

Coding system

Determines the search flow to find micro-organism and antibiotic codes received from a station (result messages) and the flow for download of isolated micro-organisms codes to a station

From station to GLIMS:

Micro-organisms:  If a coding system is defined, the micro-organism is looked up in the micro-organism code set with a matching code. If no coding system is defined on the station, it is looked up in the Micro-organism table with a matching mnemonic.

Antibiotics:  If a coding system is defined, the antibiotic is looked up in the antibiotic code set with a matching code. If no coding system is defined on the station, it is looked up in the Antibiotic table with a matching mnemonic.

From GLIMS to station

Micro-organisms: if a coding system is defined, search for an instance in the micro-organism code set for the given micro-organism and transmit the code value to the station. If no coding system is defined on the station, the micro-organism's mnemonic is sent.

Urgency levels

Specifies the mapping of urgency levels with those on the instrument.

Example:

Example: R,S1,S2,S3,U

Urgency level 1 will be transmitted to the translator as 'R', urgency level 2 will be transmitted as 'S1', etc.

If the urgency level is not within the list or if no list is specified, the urgency code will be transmitted to the translator as 'R' for level 1 and 'S' for any level above 1.

Identification page

Sample

Bar code offset /Bar code length

These fields are combined to determine which part of the barcode received from the analyzer is actually the specimen identification.  This is used in one specimen queries, specimen range queries and result upload (not for ALL-queries).

The bar code received from the analyzer may contain leading zeros, not belonging to the specimen identification. The Bar code offset field indicates the offset at which the actual specimen id starts (no offset = 0).

The Bar code length field indicates the actual length of the specimen identification. If unknown, the entire bar code is assumed.

Example:

Assume that the barcode received from the analyzer is "1234567" and we only want to extract "3456" as specimen identification, the bar code offset would be "3" and the bar code length would be "4".

Identification type / Use external id / Use internal id / Search via order / Material / Create specimen if necessary / Create order if necessary:

All these fields are used to lookup a specimen in GLIMS when a barcode is received from the instrument.  For a full description of this topic, please check the chapter Specimen and order search via barcode.

Synchronize backward / forward

In GLIMS, it possible to indicate in e.g. the specimen receipt program that the object time of an order and its entire contents (results, specimens, ...) should be synchronized with the current time.

To allow this mechanism to be used automatically when a specimen is confirmed via e.g. a LAS, you can use the following fields:

  • Station.SynchronizeBackward and Station.SynchronizedForward indicate that if a specimen is confirmed via scanning for this station, synchronization is to take place.
  • The field Order.Synchronized indicates if an order has already been synchronized or not.
  • An order can only be synchronized once.

Note:

The synchronization options of the specimen scan and specimen receipt programs will have preference over the settings of the selected station in these programs. Please consult the Mark as synchronized option in the Order entry documentation to disable unwanted backward / forward synchronization and to check the Synchronized flag of the order.

Object

Patient identification

Determines how the patient should be identified in analyzer communication.

Person identification code

PIN provider determined as follows: Station -> Translator -> Service -> Batch user -> User preferences -> Default identification source.

National number

The national number is sent in the work order message's patient id field (MIPS ASTM P.3.1.1). It also instructs GLIMS to search by national number when a station queries for a patient by id.  GLIMS will also send the national number in the work order message as the patient's universal number (MIPS ASTM P.5.1.1).

Download QC as patient

Download QC as patient samples to station, meaning that samples in the download message to the station will always be sent as if it were a patient sample.

Specimen variables

Specimen variables

This option determines which specimen variables should be downloaded to the analyzer.

Only source body site

Only the specimen variable known as the specimen source body site (E.g.: left arm, left hand, right lung) is downloaded. The source body site variable is identified by the material variable for which the name equals the one specified in the Source body site field (see below).  In case this field is not specified GLIMS will assume that the body site variable is the one with the lowest sequence number.

All

All specimen variables are downloaded.  If the specimen source body site is identified in the Source body site field (see below), GLIMS will download this specimen variable as source body site.  All other specimen variables are downloaded as non specimen source body site variables.  If no specimen source is identified as being the source body site,  GLIMS assumes there is no source body site and will download all specimen variables as non specimen source body site variables.

Source body site  

This field is used to indicate which specimen variable should be considered as the source body site.  This field is used in combination with the Specimen variables field (see above).

Order

Alternate order data function

Allows specifying an order-based MISPL function returning a string value.  When specified, GLIMS will download tagged values from this string value to the station.  If no tags are present, GLIMS will download the default values.  The table below shows the supported tags and the corresponding MIPS ASTM fields:

Tag Field Default when no tag is available
WARD1 P.26 Order current encounter -> current Stay -> Ward
WARD2 O.28.1

If requested in download query:

Order.Agent->Correspondent.InternalId (if ward) or

Order.Issuer->Correspondent.InternalId (if ward)

WARD3 O.28.2

If requested in download query:

Order.Encounter -> current Stay -> Ward

ISSUER O.17 Order.Issuer->Correspondent.InternalId

Example 

STRING List;
STRING IssuerId;
STRING WardId;
IssuerId := .Issuer.Identification("ISRPROV");
WardId := "DEP100"; /* Default ward */
List := PutTag("Issuer", IssuerId, List, "\");
List := PutTag("Ward1", WardId, List, "\");
RETURN List;

Work order page

General

Test selection

Check downloaded actions

Please check the chapter Analyzer download status.

Mark actions by download group

Please check the chapter Analyzer download status.

Send complete selection

If set, the complete test selection will always be sent to the station. When a request is added to an already downloaded specimen, all result outputs (the ones already sent and new ones) are selected for download.

Maximal result download status

Sets the maximal result status for which results can be selected for download to the station.

Minimal action status

This option determines the minimal status that actions should have before they will be downloaded in batch (ALL-query).

  • When set to Inputting, specimens should not be confirmed in order to be downloaded.
  • When set to Pending, specimens should first be confirmed.

Minimal specimen status

This option allows to only select the actions of specimens in status Expected (and Available) during a batch download (i.e. ALL-query).

This way it can be guaranteed that all the actions a station receives from GLIMS upon reply of an ALL-query are in fact to be actually executed in the near future, when the samples are confirmed available. The default value for this field is Initial, which means that all specimens are taken into account. This option is only editable when the Minimal action status is Inputting.

Download finished tests

GLIMS is able to download confirmed/validated and/or discontinued tests in a download message. The finished/discontinued tests are downloaded together with new requested tests.  This functionality is activated if following conditions are met:

  • To download confirmed/validated tests: the station receives a test selection query with an indication 'FinishedTests' (MIPS ASTM field Q.15.x)
  • To download discontinued tests: the station receives a test selection query with an indication 'CancelledTests' (MIPS ASTM field Q.15.x)
  • The station for which finished tests should be downloaded, should have the field Download finished tests enabled.  Note that the field has only effect on the requests scheduled for that station.

Notes

When receiving the first ALL query from an analyzer (e.g. after a service restart or a station configuration reload), GLIMS will only search for tests finished / discontinued since the last query for finished / discontinued tests, with a maximum of 24 hours in the past (for performance reasons). GLIMS will keep track of the time when the last query occurred in the hidden Station fields TimeLastAllQryFinTst and TimeLastAllQryCncTst. For each new ALL query, GLIMS will only look back until the previous query to download finished / discontinued tests.

The following Station-based MISPL expression returns the value of the hidden fields:

RETURN "Last cancelled test query: " + IfKnownString(DateTimeToString(.TimeLastAllQryCncTst,"%d/%m/%Y - %H:%M:%S") )+ "~nLast finished test query: " + IfKnownString(DateTimeToString(.TimeLastAllQryFinTst,"%d/%m/%Y - %H:%M:%S"));

When a finished result is downloaded to the analyzer, it is registered in the download status of the result output.

Send modification method

If set, GLIMS will download the result repeat or dilute information to the station.

Past / future object time

These fields have been introduced in order to optimize the performance in work order download in relation to Action.ObjectTime.

  • Past object time: when specified, GLIMS will not process actions for which the action's object time is older than the past object time limit. The limit is expressed in days.
  • Future object time: when specified, GLIMS will not process actions for which the action's object time is more recent than the future object time limit. The limit is expressed in days.

The optimization mainly has effect for labs which are planning orders in the future which should not be downloaded to an instrument today.  

Example:

Suppose actions scheduled from 48 hours ago until now are to be selected, the fields should be set to "2" for past object time and "0" for future object time.

Result page

Result comments

The following options allow to control the upload of comments received from analyzers.  To understand these options, one should know that analyzers can send comments in coded form (keywords), in textual format (one or more full sentences) or in both forms at the same time.

Comment storage

Determines which types of result comments (coded, textual) received from the station are to be stored.

Code or text

Coded comment or textual comment is stored, dependent of which of the two types are received. If both are received then only coded comment is stored.

Text or code

Coded comment or textual comment is stored, dependent of which of the two types are received. If both are received then only textual comment is stored.

Coded only

Only coded comment is stored.

Text only

Only textual comment is stored.

Comment as text reference

If enabled, result comment codes received from the station are stored as text reference. This means that the code is prefixed with "{<" and suffixed with "}". Note that these text references are not automatically expanded.

Multiple comment lines

If set, comments are separated with a new line character. If not set, comments are concatenated with 1 blank character

Store comments as

Determines how GLIMS should store result comments sent by the analyzer: as internal or as external result comment.  Note that the "intern"/"extern" tags specified in the result message have priority.

QC

Certain instruments can send additional QC population-related data to GLIMS. GLIMS can update the QC population based on the target and deviation values received from the instrument. The options on this tab page allow to configure how these incoming values should be interpreted. They can be applied to influence the precision and/or the accuracy of a QC population.

For precision, it is possible to save the:

  1. Target mean
  2. Target deviation

For accuracy, it is possible to:

  1. Save the target value

  2. Calculate the accuracy high and low limit

    Note

    When using the incoming deviation value for calculating the accuracy limits (high/low) of the QC population, the following formulas are used:

    1. Accuracy low limit = TargetValue - <QC Deviation Count> x DeviationValue
    2. Accuracy high limit = TargetValue + <QC Deviation Count> x DeviationValue

    <QC Deviation Count> = QC norm deviation count of the Assessment method OR QC norm deviation count in the GLIMS QC settings OR default value (= 3).

Note

When setting the accuracy, the Origin info field of the QC population is updated to "instrument:YYYY-MM-DD hh:mm:ss" to indicate that the instrument is responsible for an updated target value. However, this only happens when there is a change (i.e. the QC population will not be updated again if a result message contains the exact same values as before).

The (QC) result value which accompanies the target and deviation values sent by the instrument can be converted:

  1. Via a conversion MISPL on the assessment method.
  2. Via a unit conversion as the assessment method can have a different unit than the unit of the property.

However, the target and deviation values are not converted. They should be sent in the unit of the property in GLIMS.

Other

Verify uploaded material

When the field is disabled, the value in O.16.1.1 will not be used to lookup a matching material in GLIMS. Only the specimen id will be used to identify the matching specimen. This implies that result messages will not be rejected if the specimen could not be found due to a non-matching material.

When the field is enabled, the value of O.16.1.1 is used for the specimen identification in GLIMS. The material of the specimen in GLIMS must match the material in the result message. If the station of the result message sending translator has a coding system with a code set on Material, then GLIMS will search for a code that matches the field value in O.16.1.1. If no coding system or no code set is defined on Material, GLIMS will search for a material with a mnemonic matching the field value in O.16.1.1. If the material could not be identified, then GLIMS will log a warning message (service log file).

Store operator id, Store verifier id

Allows to indicate whether the operator/verifier sent in a result message should be stored in GLIMS as station user.

The operator id will also appear in the result audit trail: e.g. 11/09/03 10:51:54 Downloaded by the-operator-id [10.20].

Note

When a result is uploaded by an analyzer, and the operator / verifier is specified in the message, this info can be consulted in the result detail screen.

Operator, Verifier

Both fields refer to a Station user.

Upload per specimen 

Indicates if the station (or the translator) sends all the results of the same specimen in one action.  If not set, then the QC result upload will create an action with a result output for each possible QC population for this QC material, in order to make sure that the next QC result of the same control material is stored in the same action (otherwise an action would be created for each separate result of the same control material, leading to problems in the confirmation browser).

Result update status limit

Allows to limit the status in which results can be updated by an instrument. This field offers the following options:

Available (default value)

If the result in GLIMS is confirmed or validated, the update is rejected. Available results may be updated.

Confirmed

If the result in GLIMS is validated, the update is rejected. Available and confirmed results may be updated.

Validated

The update is always allowed, even for validated results.

Results

Antibiotic panel mode

GLIMS allows automatic antibiogram creation.  Possible values:

None

No panel creation, allow any result. All antibiotic results transmitted by the instrument are stored.

Reject non members

An antibiogram will be generated automatically at the moment the first result arrives. The results of non antibiotic panel members that are transmitted by the instrument will be rejected.

Non members not reportable

An antibiogram will be generated automatically at the moment the first result arrives. The results of non antibiotic panel members that are transmitted by the instrument will be set to not reportable.

Note:

When the 2nd or the 3rd option is active, beware that results will also be rejected or will also be set to not reportable when no antibiotic panel is generated for the isolation concerned.  

Auto-confirm isolations

If set, isolations uploaded from the instrument will be automatically confirmed

Auto confirm isolation tests

If enabled, isolation tests uploaded by an instrument will be confirmed automatically.

Delete antibiotic results

If enabled, and at least one valid antibiotic result is received for an isolation, GLIMS will delete the existing antibiotic results.

Example

  1. A microbiology analyzer sends a full antibiogram to GLIMS.
  2. Upon a second reading, a number of individual antibiotic results are suppressed on the analyzer or middleware solution.
  3. When the analyzer sends an update to GLIMS, these suppressed antibiotic results are not automatically deleted in GLIMS.
  4. If, in this case, the antibiotic results should be deleted in GLIMS, enable this option.

Notes

  • This action is logged in the service log file.
  • GLIMS tracks the deletion of antibiotic results. This can be consulted in the audit trail of the isolation and microbiology action.
  • The Station option Isolation update status limit remains valid. If this limit is reached, GLIMS will not accept any updates and therefore not delete existing antibiotic results either.

Auto-confirm antibiotic results

If set, antibiotic results uploaded from the instrument will be automatically confirmed.

Check antibiotic result status

Avoids that confirmed antibiotic results are overwritten by new results.

If the option is enabled

antibiotic results that are already confirmed will not be overwritten by new results received from the station.

If the option is disabled

confirmed antibiotic results may be overwritten.
