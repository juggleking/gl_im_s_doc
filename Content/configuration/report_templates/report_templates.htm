Report templates

Contents: Introduction - Main - Layout - Report Builder - Texts - Columns - Rows - Electronic - Microbiology

Introduction

Report templates are models for result reports that will be sent to order issuers or other correspondents. They determine both the content and the appearance of the documents.

The report templates are directly accessible via Start -> Configuration -> Reports -> Report templates.

Main page

General fields

Mnemonic

Unique identification of this template

Description

Allows to enter a general description of the report template.

Generator

Allows indicating the type of generator which should be used to produce the report.

Notes

  • HTML: is only supported for screen reports and email reports, NOT for printed reports. Issues not related to GLIMS have been detected for automatic printing of HTML reports.
  • HL-7: will produce a fixed HL7 format and cannot be used in combination with a translator. When using a translator to generate HL7 reports (e.g. "houl"), you need to set the Generator field to Online.

    • File name extension

    This field only applies to report templates with generator type "HTML" or "Text", and NOT to electronic reports.

    For text reports it will by default be ".txt" and for HTML reports ".htm". When generating reports (with this template) to a file (by using the report generation option Output to file), this file extension will automatically be used. This also applies to screen reports. By defining a correct file extension in the report template, the screen report will be opened with the correct application (as defined by your Windows system).

    Property classification

    Allows to enter the property classification which determines the report's property content and hierarchical organization. Only properties occurring in this classification will appear on the report. For more details on property classifications, please check the dedicated chapter.

    Warning

    Once a report template has been used to create reports, you MUST NOT change its property classification. Otherwise, new reports will be printed for all orders which have been reported with this template in the past.

    Default printer

    Allows to specify the default printer to which the reports will be sent.

    Non-electronic reports

    Show predecessor

    If set, a repeated result, which is not yet available, will show as value the value of the predecessor (originally discontinued) result, together with its rejection reason. A repeated result is a result of which the previous result has been discontinued.

    Suppress duplicates

    Allows to suppress the generation of report duplicates triggered by Report.Target.ReportCopyCount. If this field is set to TRUE, the copies are not generated.

    Page width

    The page width determines the maximum width of the page. It is expressed in characters.

    Note

    The Page width field is not used to try to fit the report headers, footer and trailer to a width of PageWidth characters, for two reasons:

    • The headers and footers are typically already configured in a way that they will fit on the report page.
    • These text may contain control sequences which may lead to an incorrect decision of the page fitting mechanism.

    The PageWidth is only used for full width or text results, foot notes, some special rows (e.g. SOP row) and title rows. Note that in these cases actually the maximum of the sum of the different column widths and the page width is used. This implies that you might have to lower the value set for the ReportTemplate.PageWidth field to the actual physical page width.

    Page line count

    Allows indicating the number of lines on a page. When the page line count is specified, the report file is always closed by a form feed.

    Archive mode

    Generated reports associated with an order, can be stored as attachment files to the order.

    • Archive Document: a plain .prn file will be stored as attachment file.

    • Archive As PDF Document: the .prn file will first be converted to a PDF document before it is stored as attachment file. The actual PDF conversion must be handled by third party software.

      Note

      Each customer is free to choose the conversion tool to be used. License costs for these tools are not included in the GLIMS license and are at the expense of the customer.

      MIPS has evaluated the following applications:

      LINCPDF (Windows only)

      Website: http://www.lincolnco.com/products/pclpdfconverter.htm

      Install LINCPDF and copy LPDFCtrl.dll to %GLIMS%\ocx "ocx\lpdfctrl.dll"

      Register this dll.

      PCL2PDF (Unix and Windows)

      Website: http://www.visual.co.uk/

      On Windows:

      Install PCL2PDF and copy Pcl2pdf32.dll to %GLIMS%\BIN "bin\pcl2pdf32.dll"

      On Unix

      Put the executable "pcl2pdf" in the GLIMS bin directory (or anywhere within $Path).

    About attachments

    • Note that you need a separate license to use the attachment functionality in GLIMS.

    • Reports generated for a report template with a value of Archive Document or Archive As PDF Document for this field, will have their generated documents saved as attachment to the order (if and only if Mark Output was checked when generating the reports).

      For more information on where the attachment files can be stored, please check the chapter Attachments. An example would be to define an Order-based site function called AttachmentFolder to determine the folder where the attachments are saved. Example:

      RETURN "\\THALAMUS\pdis\Glims\v9\att\Order\"
      + DateToString(DateTimeToDate(.CreationTime), "%Y%m%d");

      This stores all attachments of a specific order in an order-related directory and these order-related directories are grouped by day (every day a new directory with subdirectory for each order created that day).

      The attachment file name is of the form:

      <Orderinternalid>_<ReportTemplateMnemonic>_<ReportTargetInternalid>_<PrintVersion>

    Save as PDF

    This option is only supported when the report template's Generator field is set to Text and it is used as attached report of an online report. If enabled, attached reports will be saved as PDF document before being sent along with the primary report.

    The actual PDF generation is realized via an external PDF converter (see description above of field Archive mode). The original attached report should be configured as a PCL report.

    Note

    When using this feature, make sure that the option File append in the report generation screen (Electronic page) is disabled. PDF reports cannot be appended.

    Layout page

    Group results in columns by

    Determines how columns are set up on the report. Possible values are:

    • Object time: if an order contains different object times, a column will be created for each object time. However, a report column always refers to a single order. This implies that two orders with identical object times will be displayed in two columns.
    • Order: one column per order.
    • Order date: one column per order per object date. An order that is spread out over several days will be subdivided. For each day, a separate column will be created. However, there is one exception: dynamic tests spread out over more than one day will be grouped on the day of the first test.
    • Specimen: one column per specimen.

    Number of result value columns

    Determines the number of result value columns (orders) on a report. Setting this field to "0" will make sure that the normal report body (i.e. the result columns) is not printed. Thus, for each patient, the program only generates the defined first page header (which is report-based).

    Number of "previous value" columns

    Allows specifying how many previous value/date (PV/PD) columns you want on the report. For more information, check the documentation of the Result section field below.

    Print empty columns

    Decides whether empty (unused) columns on the report are suppressed or not.

    Reverse column sequence (oldest at the left)

    Enabling this field results in columns being classified chronologically from left to right. In the opposite case, the most recent results are in the leftmost result column.

    Report layout template (Windows / Unix)

    File name of a Jasper layout file, only used when the generator is Report Builder.

    Suppress wrapping and Suppress clipping

    When users try to create an html report, they often are confronted with the effect that the report program will clip some fields when they are too long to fit in the column (e.g. unit column), or will wrap the text around in order to fit it in the column (e.g. value column).

    To allow the user to switch off these effects, the following fields are available: Suppress wrapping and Suppress clipping. Next to experimenting with the column widths and the "FootNote=After" setting, these fields should allow generating a valid html report.

    Result section

    The result section layout fill-in determines which columns will appear in the body section of the report, containing the results. It also determines the relative sequence of these columns and their respective width and alignment.

    ResultSectionLayout : ColumnLayoutList [ '(' ColumnSeparator ')' ]
    ;
    ColumnSeparator : Character
    ;
    ColumnLayoutList : ColumnLayout { ';' ColumnLayout }
    ;
    ColumnLayout : Content [ ':' Width [ ':' Alignment [ ':' Char width ] ] ]
    ;
    Content : P | U | M | N | V | F | SV | SP | B | A | S | PV | PD | MD
    ;
    Width : column width in pixels (e.g. 150)
    ;
    Alignment : L | R | C | D 
    ;

     

    Content codes

    Code Description Remarks
    P Property description
    U Unit description When all the results of a numeric property (data type numeric or titer) shown on
    a single row in a text report have a non-numeric value (e.g. "No sample"), then the unit for this row is suppressed.
    N Normal range When all the results of a numeric property (data type numeric or titer) shown on a single row in a text report have a non-numeric value (e.g. "No sample"), and the norm of this property is numeric, then the norm will not be shown. A numeric part followed by a non-numeric part (e.g. '1.23 Verified') is also considered as numeric in this context.

    Note

    Norms making use of the Norm.ReportText feature (and thus bypassing the automatic numeric norm formatting mechanism) are NOT considered to be numeric
    M Material mnemonic This is the material defined in the property editor and NOT the procedure input material.
    MD Material description This is the material defined in the property editor and NOT the procedure input material.
    V Value The value columns are always adjacent and are together represented in the layout descriptor by a single entry.
    F Result mark Use character 'F' when a separated result mark column is wanted. Note that the result mark column becomes obsolete when the report uses more than 1 result value column. The result marks are then incorporated in the result value column as it was for reports without result mark columns. This only functions for single column reports.
    PV Previous value

    Allows adding a 'previous' value column on the report, showing the previous value for each primary result on the report.

    The previous value column on text report takes DefaultReport.IssuerRestricted into account: when set, the previous value shown has to be requested by the same physician Report.Order.Issuer. It also takes the dynamic time discriminator (dynamic panels) of a result into account when searching for a previous value for this result.

    The field ReportTemplate.PVColumnCount allows specifying how many previous value/date (PV/PD) columns you want on the report.
    PD Previous date The previous date column will contain the date indication of the previous value printed in the 'PV' column. The format is "DD/MM/YY HH:MM". In order to allow for a narrower column, this is automatically changed to "DD/MM/YY" when the previous result is not of the same date as the original report. Note that the date of the previous result is actually compared with the Report.EndTime field.
    S SOP
    A Action start time
    B Validator initials B stands for 'Biologist'.
    SV Short value Both the SV and SP columns must be used as end columns.
    SP Short property Both the SV and SP columns must be used as end columns.

    Alignment

    The alignment alternatives denote Left, Right, Centered and Decimal. The latter (Decimal) implies centering around the decimal point or comma of the numeric results. With this alignment, one must also specify how far from the left side of the column the decimal point is to be printed. Therefore, the syntax of the new alignment is 'Dn' with 'n' an integer value. For instance, 'V:10:D5' will result in a value column with width 10 and alignment D5 and will look like this:

    |0123456789|
|          |
|   12.3   |
|  234.56  |
|    3.141@|   truncated decimal part
|   12     |
|1234567.8 |   too many digits before the decimal -> left alignment
|negative  |   only numeric results are aligned on the decimal point

    The alignment of the right-hand part of 2-column results is always 'Left', regardless of the report template settings.

    Result mark

    Allows specifying a result-based, string-valued expression which will be used as prefix for entries in the value columns. Note that the result mark is not applied for properties with report format Full width, except if the report template field Generator is set to HTML.

    Tip

    In order to allow making a distinction between 'primary' and 'previous' results during the expansion of the result mark, a parameter {IsPrimary} is passed to the dynamic text, indicating if the current result is a primary result (it belongs to the original order of an order report), or a previous result value (it belongs to a previous order). Moreover, if the result contains a new value (i.e. a value which has not yet been printed on this report), this is indicated via the {IsNewValue} parameter. The result mark dynamic text could then contain something like: 

    IF {IsPrimary}
 THEN ((IF {IsNewValue} THEN "." ELSE " ") +
       (IF .Severity <> 0 THEN "!" ELSE " "))
  ELSE " "   

    Physical width of the result mark

    Indicates the ACTUAL (physical) width of the generated result mark, expressed in characters. This is useful if you want to use result marks wider than 2 characters, or if the physical width of the result mark is smaller than the generated character sequence (e.g. a gray box generated by a printed command sequence).

    Footnote type

    Determines how footnote references are used. Possible values:

    • None: do not use footnotes at all and put all values and comments in the cell.
    • Before: use the GLIMS footnote mechanism, taking the PropertyClassificationNode.FlushCommentsAfter into account and add the footnote reference to the left of the result value. This option applies to text reports.
    • After: same as above, but the footnote reference is added to the right of the value.
    • Native: no longer supported.

    Expand result in "Short value" column

    Makes it possible to put the expanded result value and the results comments in the short value column. If set, the short value column will contain the same text as the (most recent) result value column.

    Footnotes of the short value column will be placed on the right side of the report. The right side of a report starts with the short property column or the short value column (whatever comes first).

    Example: 

    Property   |    Result   | Norm    |Short Prop |Short Value|
------------------------------------------------------------
Natrium    | 10 (1)      |         |Na         | 10 (1)    |
    (1)  Na external comment            (1) Na external
                                            comment

    Comment in value column

    Allows having immediately flushed comments printed in the result value column, as opposed to printing them at the left margin of the report.

    Show external comment dates

    If set, external comments will be preceded with their registration date.

    Note

    If the field Isolation comments is specified in the report template, this option will not be applied.

    Title filler

    Allows to fill the character while centering the title.

    Long value column span

    No longer supported.

    Comment in separate row

    If the Property option Report format is set to Columnar, and the result's external comment is too long to fit in the column, the text will be wrapped, which may result in a visually unpleasing report. If this option is set, result comments will be printed in a separate row.

    Placeholders

    The following two placeholders are supported in the four metadata documented above (PdfAuthor, PdfKeywords, ...):

    • PAGECOUNT: the total number of pages of the generated report
    • PAGERANGE: either "1" if the number of pages is 1, or "1-N" where N is the number of pages of the generated report

    Thus a PdfKeywords text definition of

    Pages <PAGERANGE> ; nombre de pages <PAGECOUNT>

    would be evaluated in ReportBuilder as follows

    • for a single page report as: Pages 1 ; nombre de pages 1
    • for a five page report as: Pages 1-5 ; nombre de pages 5

    Texts

    First page header

    Appears at the head of the first page.

    Header

    Appears at the head of the second and following pages.

    Body

    Is a Result-based site dynamic text which is expanded for each result by some report generators (see variable reports) to generate the body of the report.

    Footer

    Appears at the bottom of each page.

    Trailer

    Appears at the end of the report.

    Empty result

    Is a Result-based dynamic text, used as default value when the result itself is not yet available. It can contain something like "<Follows>" or "{<NoResultText}", where NoResultText would be the mnemonic of a result-based text module.

    Note:

    In case of electronic reporting, this field also allows to define a comma-separated list of entries, one for each result status. A result can have the following statuses: Initial, Expected, Available, Confirmed, Validated or Discontinued. So, the first entry text will be displayed if the result status is 'Initial', the second entry text if the result status is 'Expected', etc.

    Example 1:

    • The 'Empty result text' is 'Follows-StatusInitial,Follows-StatusExpected,Follows-StatusAvailable'.
    • The report template's minimal result status is 'Confirmed'.
    • The result status is 'Available' -> the actual result value cannot yet be shown.

    The reported result value will be 'Follows-StatusAvailable'

    Example 2:

    • The 'Empty result text' is 'Follows-StatusInitial,Follows-StatusExpected'.
    • The report template's minimal result status is 'Confirmed'.
    • The result status is 'Available' -> the actual result value cannot yet be shown.

    The reported result value will be 'Follows-StatusExpected' as there is no entry for status 'Available' (in this case, the preceding entry is used)

    Example 3:

    • The 'Empty result text' is 'Follows'.
    • The report template's minimal result status is 'Confirmed'.
    • The result status is 'Available' -> the actual result value cannot yet be shown.

    The reported result value will be 'Follows' as there is no entry for status Available, nor for Expected (in this case, the first preceding existing entry will be used)

    Remark:

    If the empty result text must contain a ',' character as data, precede it with a '~' character.

    Tip

    Each of these text fields can contain dynamic text modules. Check out some header text examples.

    Variables

    This chapter explains variables which are available during the expansion of any dynamic text, e.g. header, footer, ...

    {ReportToArchive}

    Contains 'yes' or 'no', indicating whether this is a report which has to be archived. This offers an alternative to the '&ReportArchiveMessage' symbol.

    {Section}

    Contains the section number of the current report section (equal to the '&DocSecNo' symbol).

    {SubSection}

    Contains the sub-section number of the current report subsection. Subsections of a report are created when the report page contains more columns than allowed by ReportTemplate.ColumnCount. This normally only occurs when adding extra history to a report by activating the Add history toggle box in the report generation function parameter screen.

    {SubSectionCount}

    Contains the number of subsections in this section of the report.

    {ReportTemplate}

    Will be translated into the ReportTemplate.Mnemonic. This variable can be used in the standard way.

    Example

    {= IF "{ReportTemplate}" = "LayoutA" THEN ... ELSE ... }

    {ReportCopyList}

    Upon text report printing, the report program will build a list of the names of correspondents who will receive a copy report for the current order. Note that this only applies to standard reports and not copy reports.

    Example

    {= IfKnownString("Copies sent to: " + "{ReportCopyList}" }

    {ReportStatus}

    It is possible to indicate whether the report is an interim report, a complete report, a corrected report or a duplicate report. The Report.CompletenessTime field stores the date and time upon which a report was first generated as a complete report.

    For object, encounter and year reports, Report.CompletenessTime is interpreted as 'The time at which the first complete version of the report with the current columns is generated'. Each time a new column (or order) is added to the report, the Report.CompletenessTime is cleared, so that it again becomes filled in at the moment the 'new' report becomes complete.

    During text report printing, this field aids in deciding the report status:

    • Interim: an incomplete result was detected

    • Complete: the decision whether a report is considered to be complete, depends on two default report fields:
    • DefaultReport.CompletenessStatus: when one of the results has a status lower than the CompletenessStatus, this result (and the report) is considered as incomplete.
    • DefaultReport.MBCompletenessStatus: when the microbiology action of a culture result has a status lower than the MBCompletenessStatus, this result (and both its culture result and the report) is considered as incomplete.

    When these values are not specified, the following defaults apply :

    • For 'status' reports, CompletenessStatus is set to 'Validated' and MBCompletenessStatus is set to 'Closed'.
    • For 'value' reports with MinimalResultStatus = 'Validated', CompletenessStatus is set to 'Validated' and MBCompletenessStatus is set to 'Closed'.
    • For 'value' reports with MinimalResultStatus = 'Confirmed', CompletenessStatus is set to 'Confirmed' and MBCompletenessStatus is set to 'Ready'.
    • For 'value' reports with MinimalResultStatus = 'Available', CompletenessStatus is set to 'Available' and MBCompletenessStatus is set to 'Ready'.

    • Duplicate: complete and Report.CompletenessTime was already filled in.

    • Modified: complete and Report.CompletenessTime filled in and a new result was detected.

    This status keyword is passed as variable with name "ReportStatus" to the expansion of the report headers and footers. There, you can print it directly on the report, or examine it via MISPL.

    Example

    Status:{= ENTRY(LOOKUP("{ReportStatus}", "Interim,Complete,Duplicate,Modified", ","), "Partial,Complete,Copy,Changed report", ",")}

    will transpose the keyword in whatever you like to print on the report.

    Note

    The Report.CompletenessTime field is only filled when a complete report is generated with the Mark output option active.

    Symbols

    This chapter explains special symbols, not expanded via the dynamic text program, but replaced by the document output program. Note that these symbols can not be interpreted during the regular expansion of the dynamic text. They are only interpreted by the program which actually does the page formatting of the report.

    &DocSecNo

    Will display the section number in case of encounter reports,day reports,...

    &PageNo

    Will display the current page number.

    &TotalPages

    Will display the total page count.

    Example

    'Page &PageNo of &TotalPages' could be translated into 'Page 2 of 4'.

    &BatchPageNo

    Can be used in the report header or footer to print the running page number in the current batch. This batch page number is reset upon each call of the document generator (e.g. the report generator) and keeps increasing until the end of the current batch. It can also be used in the 'Target header page' that can be specified in the report generation function parameter screen and in the invoice headers when printing invoices together with the report. This batch page number can assist in checking whether all pages that were generated by the program have physically been printed.

    Variable used for report archiving.

    Masks

    Masks are used to add layout to the different report elements. For paper report you could for instance add printer codes to print the value, norm, etc. in bold, italic, underlined,... A mask must always contain the text '&1', typically surrounded by PCL printer code.

    Examples

    • Norm mask: (s0p16h1s0b16901T&1(s0p12h10v0s0b16901T

    • Footnote mask: (s3b1S~&a-40V&~&a+40V(s0b0S

      This example uses & instead of &1. GLIMS will therefore automatically add () for the footnote The PCL code will lift and then lower the pen. The text is printed in bold and italic.

    • Unit mask: (s0b1S&1(s0S

      Bold is deactivated to avoid that the unit is printed in bold because the result is printed in bold. The norm is printed in italic.

    There are three special masks:

    Value / Short value

    In the value and short value mask, you can also use a Result-based, character-valued site function instead of the above explained '&1' format. The field should then contain the name of the site function to be used. This can for instance be used to set another font for abnormal results on paper reports.

    Warning

    When using a MISPL expression in these fields, the new result mask will not be used, allowing the site function to have complete control over the way this result gets printed. You will have to cover this aspect in your site function.

    When printing the result, the corresponding actual value for the mask will be calculated via MISPL, based on the current result (or an empty result if the cell does not contain a result). During reporting, the following fields are available (next to the rest of the result fields which are a simple copy of the actual result's fields):

    • Result.FirstReportTime: is filled with the last update time of the result if and only if the result has already been printed on this report. When the result is new on this report, the field has the unknown value.
    • Result.LastReportTime: is filled with the last output time of this report. This allows to check whether a 'new' result is printed on the first printout of a report or not. Many customers prefer not to mark a result as new on the first printout of a report.

    Example

    RETURN

    (IF .RawValue <> ? AND .FirstReportTime = ? THEN "\Boldon" ELSE "") +

    (IF .Severity > 10 THEN "\UnderlineOn" ELSE "") +

    "&1" + "\BoldOff\UnderlineOff"; }

    ---------------------------------------------

    IF .FirstReportTime = ? THEN "^N&1^n" ELSE "&1";

    New results will be marked as such even on the first report printout.

    ---------------------------------------------

    IF .FirstReportTime = ? AND .LastReportTime <> ? THEN "^N&1^n" ELSE "&1"

    New results will be marked as such but not on the first report printout.

    New result

    This mask is used to customize the layout of new results.

    Warning

    This field is not used when either the value or short value mask contains a site function.

    Many customers prefer not to use the New result mask on the first printout of a report, since on such copies all results are new and hence all results are marked. To cope with this, the New result mask is not used the first time the report gets printed (i.e. the Report.LastOutputTime has not yet been filled in). To make the same decision possible using the MISPL Value/Short value mask method (see above), the fictitious Result.LastReportTime field is filled with the value of Report.LastOutputTime (see practical examples above).

    Rows

    Row labels

    SOP

    Indicates if a SOP row has to be generated when the SOP of the test changes and what its label should be.

    Validator

    Indicates if a validator row has to be generated at the end of the report. If enabled, each column of this row will be filled with the initials of the users who validated results in this column.

    Method

    Allows to add Result.Method (or AssessmentMethod.Method, see note below) on a text report as a new row following the result row. The method is added as soon as row label is specified in this field or the field ReportTemplate.MethodRowMask is filled and the current result's method has a name. The method name after a test will be suppressed when the next test on the report has the same method name.

    Note

    The Result.Method field stores the value of AssessmentMethod.Method at the time of result entry. The report program which can put the method field on the report uses Result.Method as soon as the result is available and Result.Method is known. If not, AssessmentMethod.Method is used.

    This allows storing the history of the method field in the result table itself, thus making this field reproducible on the report, even when the assessment method has been changed.

    The function AssessmentMethod.SetResultMethod() allows to explicitly fill this Result.Method field with either the value from the assessment method record, or a specific value. This may be useful for storing the method name in old result records.

    Row masks

    Title

    Allows to create a mask for the chapter titles (property classification nodes). When the title mask is specified, it is applied before the title filler and the alignment is applied.

    Example

    - Title = Chemistry
    - Mask = <b>&1</b>
    - Filler = "-"
    => resulting title = "--------------<b>Chemistry</b>--------------"
    - Filler = ?
    => resulting title = "<b>Chemistry</b>"

    Also, when a postamble is defined in a Property classification node for a title, the postamble is appended to the actual end of the title (and its filler and mask) and not to the end of the line.

    SOP

    If a SOP row is generated, then this field allows to specify a mask for the SOP row.

    Method

    Allows specifying a mask for the assessment method row.
