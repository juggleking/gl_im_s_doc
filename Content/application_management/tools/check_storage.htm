Check storage

Introduction

The people of our helpdesk often have to deal with hard disks running out of space, sometimes leading to one or two days of 'corrective' actions rebuilding the database and cleaning up.

We would like to emphasize the importance of a correct system administration and point out that it is the customer's responsibility to keep an eye on the disk usage. Everybody will agree that it is better to prevent than to cure, so let's live to it.

Automated checks

The 'Check storage' function

The Check storage function allows monitoring both the disk space and the overflow extent usage. This function is available from the main menu via Start -> System management -> Database -> Check storage. This function can be automated through a command and task. As of version 8.1, this command is automatically added. By default, the task will run every night at 3 a.m.

This function will also issue a warning when the last before image extent has become larger than 1GB, suggesting to either add a new extent or truncate the database.

Backup maximal days passed

If the last backup has been taken more than X days ago, a warning will be will be sent to users of type System manager and Developer.

Disk minimal free megabytes

Minimal amount of free disk space that should be available. Below this level, warnings will be sent to users of type System manager and Developer.

Disk maximal percentage used

Maximal amount of disk space (expressed as a percentage of disk capacity) that should be used. Above this level, warnings will be sent to users of type System manager and Developer.

Area minimal free megabytes

Area maximal percentage used

Check overflow extent

If set, the application will verify whether the database overflow extents are in use. If so, warnings will be sent to users of type System manager and Developer. For more information check the chapter Adding database extents'.

Note

This option is not supported for Oracle databases.

Support email address

Any warnings will also be forwarded to this email address.

Automated checks on Unix machines

On Unix servers, a functionality is available which warns about the disk usage. The start procedure "mam glims start" will issue warnings when some non-fixed database extents are on disks with less than 200 MB free OR on disks with 80% or more used space. If users are about to write on the overflow extents of the databases, a warning will be issued via a high priority mail with message subject "Database watch".

Note

Only users of type 'System manager' and 'Developer' will receive this kind of mail.

Example

You are writing in the overflow extent of database 'genro', area 'Schema Area' Overflow extent size is 131005

You are writing in the overflow extent of database 'genrw', area 'Schema Area' Overflow extent size is 28925

You are writing in the overflow extent of database 'glims', area 'Schema Area' Overflow extent size is 137952

Adding database extents

Each Progress database consists of several database extents. Typically, this will be a number of fixed-size extents and one overflow extent which will be used when the fixed-size extents are full.

The overflow extent has a variable size and slowly grows according to the needs. This causes overhead and fragmentation which both lead to a bad performance. Moreover, such an extent can have a maximum size of 2GB. If this limit is reached, the database will stop.

To avoid writing in the overflow extent, the above-documented checks will issue warnings. If this is the case, new database extents have to be created. This can be done as follows:

  1. Get a list of the currently existing database extents: prostrct list glims. This will create a glims.st file with all extents of database 'glims'. In this file, you can check which is the last database extent, e.g. "glims.d99".

  2. You can now create a new add.st file to add new extents, for instance:

    d "Schema Area":6,64 /data2/glimsdbf/glims.d100 f 512000
    d "Schema Area":6,64 /data3/glimsdbf/glims.d101 f 512000
    d "Schema Area":6,64 /data5/glimsdbf/glims.d102 f 512000
    d "Schema Area":6,64 /data2/glimsdbf/glims.d103 f 512000
    d "Schema Area":6,64 /data3/glimsdbf/glims.d104 f 512000
    d "Schema Area":6,64 /data5/glimsdbf/glims.d105 f 512000
    d "Schema Area":6,64 /data2/glimsdbf/glims.d106 f 512000
    d "Schema Area":6,64 /data3/glimsdbf/glims.d107

    f = 'fixed size' and 512000-> creates extents of 512 MB. In this example we will add 7 extents of 512MB = 3GB. The overflow extent will now be "glims.d107". The previous overflow extent "glims.d99" will automatically be converted into a fixed-size extent.

    To add the extents: prostrct add glims add.st

    Progress will format the new extents (30sec/500MB). The extents will be put on different disks in order to divide the working set of data (the most recent data) over several disks.
