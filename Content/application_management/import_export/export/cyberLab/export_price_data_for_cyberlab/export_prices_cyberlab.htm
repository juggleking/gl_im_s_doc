Export of prices for import in CyberLab

Content: Introduction - Starting the export - Exported data - Non-exported data

Introduction

GLIMS offers a function to export price information in CSV format to CyberLab: Export price list, based on the gp_site table. The generated .csv file can then be imported in CyberLab to offer a price indication during order entry.

Warnings

Warnings

  • This export function requires a separate license: GLIMS CyberLab synchronization.
  • The full price export functionality as described here is only supported for the communication between GLIMS 9.6.3 (and higher) and CyberLab 9.12 (and higher).

Starting the export

Manually

From the main menu, click Start -> Billing -> Export price lists.

The fields of this window are described below. The data specified in these fields is saved and will be automatically displayed when the window is opened again later on.

Directory

Directory where the exported file will be saved.

File name

Name of the exported file.

HC code provider

The price export function allows the export of both price list issuers and request definition issuers to CyberLab.The system works as follows:

  • If the HC code provider is filled in, GLIMS looks for the HC provider identification:
    • if a valid identification is found, it is added to the price export file.

    • if a non-valid identification is found, none of the request definition data is exported and a log entry is created under System Management -> Logging -> Logs -> Log type = Export prices for CyberLab.

      Note

      Note

      From this log entry, you can easily get a list of all the identifications assigned to the HC provider:

      • right-click on the log entry -> Open subject.
      • right-click in the Issuer field -> Go to -> Issuer
      • right-click anywhere in the Issuer editor -> Identifications -> Assigned to.
  • If the HC code provider field is left empty, the HC provider mnemonic is exported instead of the identification.

Automated

In most cases, the export will be run at a regular basis through a command.

Exported data

The following list indicates which information elements are present in the .csv file and in which order they currently appear.

  1. request definition mnemonic
  2. Nomenclature code
  3. Billing code
  4. Billing code description
  5. Reimbursement class mnemonic
  6. Price list mnemonic
  7. Price code
  8. Base Value code
  9. Base Value
  10. Multiplier price formula (only if extension price list is used)
  11. Fixed Amount
  12. List Price (Calculated Amount)
  13. Discount (is always 0)
  14. Charged Price (multiplied with extension factor)
  15. To charge field of request definition
  16. To charge field of tariff
  17. Start date tariff
  18. End date tariff
  19. HC provider identification/mnemonic of price list
  20. Issuer of request definition
  21. Type of request definition issuer

As illustrated in the example below, these fields are all mentioned in the header of the .csv file. That way, CyberLab and GLIMS can add fields or change the order of the fields as necessary, regardless of the version on which they are running.

Note

Note

When exporting price information, the validity of the request definition is checked: the export time must lie between the start and end date indicated in the request definition editor.

Request definition mnemonic

The request definition that is to be billed is indicated by its mnemonic in field 1 of the .csv file. Two questions arise:

  1. For a request definition to be billed, it needs to be linked with a billing code. What happens if the request definition to be billed does not have any billing code assignment?
  2. Request definitions of type Panel are themselves composed of several request definitions that need to be billed. Which price related data is exported to CyberLab, then?

Request definition without billing code assignment

If the request definition to be billed does not have a billing code assignement, the following mechanism is used:

Request definition of type Panel

Request definition of type Panel (called "panel request definitions" below) are themselves composed of several request definitions, which can have their own price data. Thus, in case of a panel request definition, GLIMS first needs to determine which pieces of information should be exported to CyberLab. GLIMS does this on the basis of the value indicated in the field Charge of the panel request definition editor. This field can have 5 different values:

  • Members only: the data of the panel request definition is not exported, only the data of its members are. This means that the price indicated in CyberLab will be the sum of the prices of the individual panel request definition members.

  • Panel only: only the data of the panel request definition is exported. The price indicated in CyberLab will be that of the panel request definition.
  • Panel and members: The data of both the panel request definition and its members is exported. The price indicated in CyberLab will be that of the panel request definition + the sum of the prices of the individual panel request definition members.
  • Members or panel: this value is not supported by CyberLab.
  • ?: This value cannot be chosen if the option To charge of the panel request definition is activated.
Notes

Notes

In case of Members only and Panel and members:

  • If a member is itself a panel request definition, the value of its Charge field is also checked:
    • Members only: the data of the subpanel members is exported, the data of the subpanel itself is not.
    • Panel only: the data of the subpanel is exported, that of its members is not.
    • Panel and members: the data of both the subpanel and its members is exported.
  • If a panel request definition member does not have a valid tariff record for a certain price list, the price information for the panel request definition / panel request definition member is not exported to CyberLab. In this case, a log entry mentionning the related billing code is created under System Management > Logging > Logs > Log type = Export prices for CyberLab.
  • The price of a panel member is the sum of all its billing codes. Therefore, if a panel member has several billing code assignements, then all its billing codes are exported.
Examples

Example 1

Panel request definition R is composed of

  • request definition Cl
  • panel request definition B, itself composed of
    • request definition Na
    • request definition K

Value of the Charge field for panel request definition R: Members only

Value of the Charge field for panel request definition B: Panel only

--> The export looks as follows:

request definition mnemonic Nomenclature code Billing code Price
Req_R 1111 BC_Cl 10
Req_R 222 BC_NA+K 20

Example 2

Same as in Example 1 but the value of the Charge field for panel request definition B is Members only.

--> The export looks as follows:

request definition mnemonic Nomenclature code Billing code Price
Req_R 1111 BC_Cl 10
Req_R 22211 BC_NA 5
Req_R 22212 BC_K 5
Note

Note

The resulting price does not solely depend on the value of the Charge fields. MISPL functions, for instance, can play a role in its calculation.

Reimbursement class mnemonic

The billing code (and thus the request definition record) can be linked with a reimbursement class. Field 5 of the .csv file allows the export of the reimbursement class mnemonic to CyberLab.

"To charge" fields

Both request definitions and tariffs have a To charge field in GLIMS. The value of each of these fields is exported in a dedicated field in the csv file.

HC provider identification/mnemonic of price list

The price list HC provider field of the csv file allows the export of price lists that are linked to one or more specific issuers in GLIMS:

This field contains the identification or the mnemonic of the HC provider to which the price list is related. If several issuers have the same price list, then all the request definitions of this price list are exported for each issuer.

Issuer of request definition and type of request definition issuer

Issuer of request definition

The issuer of request definition field in the csv file allows the export of the specific issuer linked to the exported request definition in GLIMS:

This field contains either the identification or the mnemonic of the request definition issuer.

Type of request definition issuer

This field indicates the type of the request definition issuer, which can be any type of the following list:

1 = Person, 2 = Organization, 3 = HcProvider, 4 = Ward, 5 = Institution, 6=Lab, 7=Department, 8=Firm, 9= Fund, 10 = Health Office, 11=Company, 12=Bank, 13= Study

In the case of panel request definitions

Basically, when a request definition is linked to a particular issuer, then the issuer information is exported together with the request definition information. The situation becomes more complex when this request definition points to a panel request definition whose members are themselves linked to no or different issuers.

In such cases, the following rule applies:

A panel member is exported if
  • the issuer to which it is linked is identical to the issuer of the panel request definition (more accurately, identical to the issuer of the request definition that points to the panel request definition), or
  • this panel member is not linked to any issuer.
Examples

Examples

1.

Request definition panel Z has issuer A and contains three members: M1, M2, M3.

M1 has no issuer

M2 has issuer A

M3 has issuer B

--> If issuer A requests Panel Z, M1 and M2 will be exported, M3 will not be exported (independently of whether M3 is a simple request definition or a panel request definition) because it is specific to issuer B.

Obviously, since Panel Z is specific to issuer A, no issuer but A can request it.

 

2.

Request definition panel Z has no issuer and contains three members: M1, M2, M3.

M1 has no issuer

M2 has issuer A

M3 has issuer B

--> Like in example 1, if issuer A requests Panel Z, M1 and M2 will be exported, M3 will not.

--> If issuer B requests Panel Z, M1 and M3 will be exported, M2 will not

--> If issuer C request Panel Z, only M1 will be exported.

Non-exported data

  • Discount data is not exported.
  • MISPL logic is not exported.
  • Billing code assignment, Tariffs and Price formulas are not exported if the export time lies outside their validity start and end dates.
