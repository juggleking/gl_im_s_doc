Export configuration data for CyberLab - Request codes and request forms

Introduction

To export request code and request form information, the option Export request codes of the CyberLab export window should be enabled.

Output file: "RequestCodes.xml". This file contains:

  • a list of all request codes
  • a hierarchically structured list of request forms containing request codes

Request codes: fields exported to CyberLab 9.11 and higher

Additionally to the commonly exported data, the following fields are also exported if your version of CyberLab is 9.11 or higher and if you chose v9.11 and above in the CyberLab version field of the CyberLab export window.

  • <RequestCode.Explicit>: indicates whether the request definitions are explicit or not. CyberLab can use this information for panel members.
  • <RequestCode.Enabled>: indicates whether the request definitions are active or not. CyberLab can use this information to update existing request definitions.
  • <RequestCode.Universal>: indicates if the underlying request definitions are universal or material/issuer specific.

  • additional information about each valid request definition of the request code (the validity is based on the Start date and End date of the request definition):

    • <request definition.IsDefaultRequestable>

    • <Requestable.Issuer>

      • <Issuer.Type>
      • <Issuer.Mnemonic>
      • <Issuer.InternalID>

      • <Issuer.IssuerCodes>

        • <IssuerCode.Code>
        • <IssuerCode.Provider>
    • <Requestable.Type>
    • <Requestable.TemplateMnemonic>
    • <Requestable.Discriminator>

  • additional information about preferred materials/specimen volume. For each material:

    • <Material.Code>: mnemonic of the material
    • <Material.Unit>: unit used to measure the material
    • <Material.Dimension>: dimension the unit represents

    • <Material.Value>: maximal size of the material

      Warning

      Warning

      Units and values are exported as they are, without conversion.

      Note

      Note

      For property request definitions, all input materials of the preferred procedure will be listed. Moreover, the Consumption of the property output is exported instead of the Maximal size of the Material.

      For microbiology request definitions, the MicrobiologyProcedure material will be listed.

      For pathology request definitions, the PathologyProcedure material will be listed.

      For material request definitions, the material itself will be listed.

      For blood request definitions, the BloodProduct material will be listed.

  • If a request definition is a panel:

    • <PanelMember.RequestCode>
    • <PanelMember.Discriminator>

Request forms

CyberLab uses the hierarchical structure Platform > Discipline > Section > Test/request definition. Upon XML export, GLIMS automatically maps the hierarchical structure of its request forms (request form -> request tab -> request tab member) onto the CyberLab structure.

Unique identifiers when using CyberLab 9.11 and higher

Several tags contain a unique identifier. When configuration data changes in GLIMS, these identifiers allow CyberLab to notice that existing information has been modified and should be updated:

  • The section tags in the request code information contain a unique identifier composed of the discipline's key followed by "-01".
  • The tags of each request form, request tab and request tab member comprise a unique identifier composed of the id and creation time of the corresponding record.
