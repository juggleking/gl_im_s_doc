Denormalized import

Introduction

Use this function to import data into the database from text files adhering to the format for denormalized import and export. Denormalized import can be accessed from Start > System management > Database > Import.

Usage

Basic Page

Use script

If checked, Script file is enabled. Else, Data file is enabled.

Script file

Is a file containing a line for each data file to be imported. Each line contains the data file path (relative to the location of the script file itself), a separating space and a comma-separated list of columns to be imported. If this list is unspecified (as designated by a question mark) or omitted, all columns are imported.

Data file

Is a text file from which records need to be imported.

Select columns

If checked, a dialog is shown where you can interactively select the columns to be imported.

Column selector list

Is a comma-separated list of discriminator tags used to pick one column out of a list of tagged alternatives matching a single field.

Stop on record rejection

If checked, current csv file importing and script processing are discontinued whenever any record is rejected.

Create new records

If checked, records found in the text file but not in the database will be created.

Update existing records

If checked, records found in the text file and also in the database will be updated if the database field contents are different from those in the text file.

Enforce auto id usage

When set, and no database record can be found by remote id, the procedure will try to find the database record by id.

Example

The import file contains a column "Id" and a row with value <Id>.

  1. The import routine will first try to find a site attribute value for "_RemoteMasterId" of the processed table matching <Id>.
  2. If no matching value can be found, and the option Enforce auto id usage is set, the import program will try to find the record of the processed table with id matching <Id>. This will only work correctly when importing data from a file originally exported from the same database (or a copy).

Advanced Page

Character set

Identifies the character set used in the text file. If left unspecified, no character set conversion is applied.

Languages

Allows you to specify a list of languages (double-clicking in this field will open a browser in which you can select one or more languages). For each of these, translations will be imported for any multilingual fields. Any other translations present in the text file will be ignored.

Confirm database changes

If checked, a confirmation dialog will appear prior to any record creation or update. Old and new values can be viewed alongside.

Log database changes

If checked, any record creation or update will be logged to a log file. Else, only rejections will be logged. The log file has the same name as the input file, but with ".csv" appended. The log text appears as an additional column at the far right.

Simulated

If checked, no effective changes will be applied to the database.

Support for non-identifiable records

Some tables have no associated human-readable identification for their records. On import, record matching is possible only if a remote master identification column is supplied. This is the auto-generated internal record identifier out of the source database where the data were exported from. These can be matched to the value of a special site attribute called "_RemoteMasterId" (defined in the database in which data is imported). Note that only one remote master identification can be stored per record. It is therefore not possible to import non-identifiable records from multiple source databases. For configuration tables, the site attribute is automatically created. For other tables, it should be created manually in the database in which data is imported. If no matching is possible, import will always create new records.

Denormalized export

Introduction

This function exports database contents into human-readable text files, one per table. The data are output to "comma separated value" files with name <TableShortName>.csv. These files adhere to the format for denormalized import and export. They can later be imported (typically into another system) with the matching Denormalized Import function.

Denormalized export can be accessed via Start > System management > Database > Export or via the Tools tab of the contextual ribbon (in which case it applies only to the selected records).

Usage

Basic Page

Use script

If Use script is checked, the Script field is enabled. Else, the Tables field is enabled.

Script

Is a text file containing a line for each table to be exported. The lines contain two space-separated fields: the table name and a comma-separated list of field paths. If this list is specified, only the listed fields are exported. If it is unspecified (as represented by a question mark) or omitted, a default is applied.

Tables

Allows you to select, by double-clicking in the field, one or more tables for exporting data from.

Select fields

If checked, a field selection dialog is shown for each selected table, allowing you to interactively pick the fields you want to export.

Folder

Is where the output files containing the exported data will be created.

Advanced Page

Languages

Allows you to specify the languages for which translations of any multilingual fields should be exported. Double-clicking in this field will open a browser in which you can select one or more languages.

Separator

Is either Comma, Semicolon or Tab. If you match this with the List separator in your Windows Regional Settings, it will be easier to open the export files with MS Excel. Your latest selection is proposed by default.

Updated since

If you specify the minimal last update date (and if time stamps are supported in the table to be exported), only records updated since the specified date will be exported.

File name style

Is either Table short name or Table name. It determines how the output files will be named (for example, the User table could be exported either to "sc_User.csv" or "usr.csv"). Your latest selection is proposed by default.

Support for non-identifiable records

Some tables have no associated human-readable identification for their records. If they are configuration tables, the auto-generated internal identifier will be exported. Subsequent import into another database will match these internal identifiers with earlier imported records from the same source database using a special site attribute called "_RemoteMasterId".

Denormalized import/export format

Introduction

This document describes the format for text files used by Denormalized import and Denormalized export. We will refer to these files as csv files, because their extension is typically ".csv" (for "Comma Separated Values").

Each file is scoped to a table in the database. The file consists of a header with general information, and a body. Each line in the body corresponds with a record in the table. Individual fields of the record are separated by a separator character.

"Denormalized" means that any "parent" records referenced from a record are identified by a human-readable identification, rather than by a numeric reference into yet another text file (which would be the "Normalized" case).

Header format

Introduction

The application supports two alternative header formats: a brief format (better suited for easy csv file editing with Microsoft Excel) and a verbose format (with extra features, such as default value assignment). Denormalized export always exports to the brief format. Denormalized import supports both brief and verbose format.

Verbose header format

The verbose header section is contained between two delimiter lines: one containing the string "%%SH" (Start header) and one containing the string "%%EH" (End header). Between these two are the directive lines:

Name

Syntax

Comment

Directive

SeparatorDirective|TableNameDirective | FieldListDirective | DefaultListDirective

SeparatorDirective

Separator = ( Comma | Semicolon | Space | Tab )

TableNameDirective

TableName =TableName

Mandatory

FieldListDirective

FieldList =ColumnIdentifier { SeparatorColumnIdentifier }

Mandatory

ColumnIdentifier

FieldName { .FieldName } [ :LanguageCode ] [ #Discriminator ]

FieldName

For site attributes, use leading tilde ('~') character. For normal fields, use names as in MISPL (without table short name prefix).

LanguageCode

Used to identify the translation used for multilingual field values.

Discriminator

Any string of characters except delimiters; used to discriminate between alternative columns for the same field.

DefaultListDirective

DefaultList =Assignment { SeparatorAssignment }

Assignment

ColumnIdentifier=FieldValue

Brief header format

The brief header consists of a single line containing only column identifiers (with syntax as in the verbose header format) separated by separator characters. The separator is auto-detected. The table is auto-detected from the file name root, which should match either the name or the short name of the table (e.g. sc_User.csv or usr.csv). Default value assignments are not supported in this format.

Body format

In the body, there should be one line per record to import (except if the record contains fields containing new line characters). The fields should occur in the sequence of the FieldList directive, and should be separated by the separator. If field values happen to contain either a separator character, a double quote character or a new line character, they should be surrounded by double quotes. Embedded double quotes must be escaped with an extra double quote.

Enumerated fields must be specified by their string value. For instance, for the 'User.Type' you may write 'Guest'.

Date fields must be specified as "YYYY/MM/DD". Time fields must be specified "HH:mm:SS". Date/time fields must be specified as "YYYY/MM/DD HH:mm:SS"

Comment lines are preceded by "%%".

Support for CLOB / BLOB fields

Import/export of table fields of datatype "BLOB" or "CLOB" is supported.

Export

When exporting records from a table with "CLOB" or "BLOB" fields, an export file will be created with as CLOB/BLOB field data the name of a CLOB/BLOB file with format

  • CLOBS: <TableShortName>!<FieldName>!<CodePage>!<Id>.blb
  • BLOBS: <TableShortName>!<FieldName>!<Id>.blb

<CodePage> is the code page used in this session, example "UTF-8". <Id> is the id of the table record.

For each exported record, a file with matching name is created in the same folder as the .csv file. This file contains the data from the exported BLOB/CLOB column. In case of CLOB this is in text format, for BLOB it is in binary format.

Import

When importing records from a table with "CLOB" or "BLOB" fields, the import file will be examined. For CLOB/BLOB fields the matching file will be searched in the folder containing the .csv file. The import will fail if the .blb file is not available.

Before attempting to import files, the data in the .blb files will be compared with the data in the database. For CLOB fields this is a "RAW" comparison, for BLOB fields this is a byte to byte comparison, although the import program will apply other, more performant comparison techniques first.

Remarks

  1. If you want to import units and dimensions, you should:
    • First import the dimensions using only the Name field.
    • Then import the units.
    • Finally, import the dimensions again, this time including the Base unit field.
  2. When importing properties or materials, you should exclude the Default request definition field.
  3. Due to the fact that inserting new nodes in a property classification may lead to problems when trying to import this modified classification in a target system which already has the rest of the classification. To avoid this problem, we advise either to use the import option only for the initial creation of the property classification and not for later updates, or to completely clear the correspondent classifications before doing an update via the import program.
  4. The records of the Code table can be exported and imported providing the records of the coded table can be externalized via a single word (example: the record referred to by a property code can be uniquely identified via the property mnemonic).
  5. Importing the um_Option table is currently non possible due to the fact that GLIMS automatically creates a default menu for each table when there is no um_Option for this table.
  6. When editing .csv files in MS Excel and then saving your changes, MS Excel will add extra comma's in the header of the file. These must be removed in order to be able to use the file for import.
  7. Columns containing dates: GLIMS always used the year/month/day format, independently of the site configuration. However, when editing .csv files with MS Excel, the latter may interpret these dates differently (using the local date format). When saving the .csv file with MS Excel, these date can either become unacceptable or may have changed (the date is still valid but has changed !!!). So be very careful when editing .csv files containing dates with MS Excel !
