Blood bag / Blood selection export & import

Introduction

This document describes how you can export blood bags and optionally blood selections from one Glims system and import these data in another Glims system.  This can be used to synchronize two Glims systems after a major upgrade.

Export blood bags and blood selections

Introduction

This program exports all blood bags and possibly blood selections into two different files.  These files can then be imported again in another Glims system.

Click Start -> System management -> Database -> Specific export -> Blood bag data

Options

When starting the program, the following options are available:

For recipients, export PIN

Decides whether the PIN should be exported for each blood bag recipient

For recipients, export internal id

Decides whether the person's internal id should be exported for each blood bag recipient.

Also export orders and blood selections (in separate file)

Decides whether the orders (containing blood requests) and their blood selections should also exported. If so they are written to a file called "bloodrequests.txt" in the session's temp-directory.  It was intended for using this program after a version upgrade, for synchronizing the upgraded database with the old database.

Only bags and blood selections created or updated since

Blood bags or blood selections created or changed BEFORE this date will not be exported.

Only bags and orders received since

Blood bags or orders received BEFORE this date will not be exported

Export typing / blood selections:  allow to exclude the blood bag typing and blood selections in the export file (typically used when merging data from one GLIMS system into another).

Exported files

The program can generate 2 files in the GLIMS temp-directory

1. "bloodbaglist.txt" (always created)

This file contains all created or modified blood bags, together with any related blood selection(s) and crossmatch result(s). Each line mentions (field separated by '|'):

- the bag internal id
- the bag external id
- the bag donor
- the bag product spec. mnemonic
- the bag's department mnemonic
- the bag's typing mnemonic
- the blood group (readable, untranslated)
- the rhesus-D (readable, untranslated) 
- the list of present or absent antigens in the format: [+/-]antigenmnemonic , ... 
- the bag's production date-time
- the bag's receipt date-time
- the supplier mnemonic
- the expiration date-time
- the transfusion date-time
- empty field (patient PIN)
- the receiving patient's internal id
- the transfusion hc provider's mnemonic
- the bag's status (readable, untranslated)
- the list of non-discontinued blood selections (blood selections and fields separated by ':') with the following fields:

* order internal id
* Bloodselection status (readable, untranslated)
* crossmatch request definition mnemonic (or empty if no crossmatch)
* crossmatch result value ('?' if no value yet; empty if no crossmatch)
* crossmatch result status (readable, untranslated) (empty if no crossmatch)

- on the following lines:

if the bag is transfused: the transfusion reactions
if the bag is not transfused: the bag's comment field.

- on the next line: "---" (to indicate the end of the bag's info)

This file should be imported in the other database using 'ap_imbb' or 'System management->Import->Blood bags'.

1. "bloodrequests.txt" (created when the option "Also export orders and blood selections (in separate file)" is set)

This file contains all orders and the number of blood requests they should contain.  Each line mentions (field separated by '|'):

- order internal id
- blood product mnemonic
- blood request definition mnemonic
- the urgency
- the number of requests in this order, for this request definition, with this urgency.

This file should be imported in the other database using 'ap_imbsel' (see below "Import blood requests").

Import blood requests

The program is called 'ap_imbsel' and should be run from Start -> Development -> Run4GLprocedure.

This program reads a file in the format of the "bloodrequests.txt" file generated by ap_exbb.p.  It searches for the order, checks if there are enough blood selection requests in that order, for that specific request definition, that blood product and that urgency.  If necessary, it adds blood selections (with status 'initial) to that order.
