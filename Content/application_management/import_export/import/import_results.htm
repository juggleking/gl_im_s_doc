Result import

Prerequisites

This result import program can only be used to import historical data (e.g. when switching from a previous LIS to Glims).It MUST NOT be used to import routine data.For the latter, MIPS can provide dedicated communication drivers.

The import file should comply with the following rules:

  1. The character used as delimiter between field values in the file is a space.
  2. All character fields must be enclosed in quotes (" ") if they contain any delimiter characters (i.e. spaces).
  3. If you want to import any quotes contained in the data, replace them by two consecutive quotes ("").
  4. If you do not wish to specify a field, you should put an unquoted question mark (?).
  5. Date fields are interpreted according to the date format (-d) and century (-yy) startup parameters of your GLIMS session (see session.pf, typically "-d dmy" and "-yy 1950"). However, it is advised to specify years in a 4-digit format (i.e. including the century).
  6. Time fields can be specified in HH, HH:MM, HH:MM:SS or HH:MM:SS+DDD format. DDD allows to specify a delta time (in seconds) which is added to the object time in order to obtain the real object time of the result. This allows to import results from dynamic tests (e.g.dextrose tests) that each have the same base time (and therefore the same order), but a different delta time.
  7. Specify numbers with decimal points (no commas).
  8. The import file must always end with a new line.
  9. Each record must meet the following format:

Activation

For performance reasons it is recommended to run the import program using the TTY interface on the server. The result import program is integrated as the ‘ImportResult’ method on ‘gp_Site’. Its standard location is ‘System management => Import => Results'.

Parameters

Fields

The File name of the import file. You can use the FIND key (TTY) or double-click (PC) to browse through your file system.

If the import file character set doesn’t match the GLIMS character set, you can specify an alternate input file character set. The name that you specify must be a valid code page name available in the $DLC/convmap.cp file (a binary file that contains all of the tables that Progress uses for character management). For more information please refer to the Progress ‘System Administration Guide’ that was provided along with your Progress license.

The Pin provider is the correspondent that assigned the PIN contained in your import file. Typically this will be the ‘Institution’ record identifying a hospital. If you have defined an ‘Identification source’ in your order entry options screen, it will be proposed as default value.

The Coding system field allows mapping external property codes onto GLIMS property mnemonics by means of PropertyIdentifications. It optionally references a PropertyIdentificationSet containing the mapping between the external codes and the GLIMS mnemonics.

Remove leading spaces: by default, the "Result import" program removes any leading and trailing spaces from the result string during import. This option indicates whether or not leading spaces should be removed or kept. Note that trailing spaces will always be removed.

When enabling the Ignore data type errors toggle-box, result values that don't match the property data type will not be rejected. Instead, they will be stored in the Result.RawValue member. Think twice before using this option. Results imported that way become unusable for some purposes, such as history graphs, statistics, etc...

If orders have to be created (enable the Create orders toggle box), it is possible to specify the default issuer for these orders. Default means that this issuer will be used if not overruled by the IssuerInternalId field of the input record. If more than one result is imported for the same order, each result will individually set the orders' issuer. So, it will be the last result of each order that will determine the actual issuer and object time. Orders are created per object time.If supplied, and the 'Create orders' toggle-box is checked, the order internal id of the import file will be used instead of an auto-generated identification. Make sure to provide unique identifiers that will not conflict with your daily routine numbering!Note that you can only user the "Create orders" option if the user who performs the import has at least one employee assignment !

The Create reports toggle box allows to indicate that the default reports should be created for these orders. Be careful with this function and use it only if this is really necessary !

By means of a simple rules wizard, the import program can be instructed to perform special processing for some pseudo-properties. If one these properties is encountered during import, it is processed according to the specified rule. When importing pseudo-results, possible comment in the import file will not be taken into account.

Output

During import some statistics are issued on the screen indicating the progress. Also some ‘debug’ files are generated:

Filename Contents
<ImportFileName>.rej All records rejected because:
  • the PIN field is empty or unknown, or the PIN was not found in the GLIMS database
  • the date field is empty or unknown
  • the Property doesn't have an associated rule or was not found in the GLIMS database
  • the result string didn't match GLIMS's Property definition and 'Ignore data type errors' was not enabled

This file can be used for subsequent import runs after correcting the reported problems (as only the problem cases are left).
<ImportFileName>.pin PIN’s that were not found in GLIMS. The affected results are of course not imported.
<ImportFileName>.prop Properties that were not found in GLIMS. The affected results are of course not imported
<ImportFileName>.rslt Results that couldn’t be converted because they didn’t match the GLIMS Property definition. E.g. alphanumeric characters in a numeric result, choice not found for an enumerated result, etc.

If the 'Ignore data type errors' toggle-box was enabled, although not rejected, erroneous results will still be logged to this file.
<ImportFileName>.min Rejected specimen internal id's because:
  • The Specimen could not be created due to a missing or invalid material
  • The Specimen already exists in the GLIMS database, but its object, material or sampling time doesn't match the imported result.

These errors are considered not to be fatal. The result will not be rejected, but its specimen reference will be unknown.
<ImportFileName>.mat Materials that were not found in GLIMS.
<ImportFileName>.ord Orders that were rejected because the same order was referred before, but for a different object.
<ImportFileName>.issr Issuers that were not found in GLIMS.

Records with format errors cannot be processed and will cause an immediate error message, with the logical line number on which they occurred. In that case, the operator has the choice to continue or abort the import.
