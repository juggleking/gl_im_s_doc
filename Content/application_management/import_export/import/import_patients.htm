Import patients from file

Prerequisites

  • Your site preferences should specify your current organization. This is the last field in the More page of the 'Specific site preference set' editor. The ‘Country’ of the specified organization will be used as the default country for patient import. Make sure to set the country’s ‘Address type’ correctly.

  • The import file should comply with following rules:

    1. The character used as delimiter between field values in the file is a space.
    2. All character fields must be enclosed in quotes (" ") if they contain any delimiter characters (i.e. spaces).
    3. If you want to import any quotes contained in the data, replace them by two consecutive quotes ("").
    4. If you do not wish to specify a field, you should put an unquoted question mark (?).
    5. Date fields are interpreted according to the date format (-d) and century (-yy) start-up parameters of your GLIMS session (typically "dmy"). However, it is advised to specify years in a 4 digit format (i.e. including the century).(see session.pf)
    6. Specify numbers with decimal points (no commas).
    7. The import file must always end with a new line.
    8. Each record must meet the following format:

Activation

The patient import program is integrated as the ‘ImportPerson’ method on ‘gp_Site’. Its standard location is System management -> Import -> Persons. For performance reasons it is strongly recommended to run the import program using the TTY interface on the server.

Parameters

The patient import user-interface:

Fields

The File name is the name of the patient import file. You can use the FIND key (TTY) or double-click (PC) to browse through your file system.

The Attribute file name is the name of a separate file which allows to import person site attributes. This file should be formatted as follows:

  • Site attribute name
  • PIN
  • Site attribute value

If the import file character set doesn’t match the GLIMS character set, you can specify an alternate one. The name that you specify must be a valid code page available in the $DLC/convmap.cp file (a binary file that contains all of the tables that Progress uses for character management). For more information please refer to the Progress ‘System Administration Guide’ that was provided along with your Progress license.

The PIN provider is the correspondent that assigned the PIN (and family doctor identification codes if applicable) contained in your import file. Typically this will be the ‘Institution’ record identifying a hospital. If you have defined an ‘Identification source’ in your order entry options screen, it will be proposed as default value.

Output

During import some statistics are issued on the screen indicating the progress. Also some ‘problem’ files are generated:

Filename Contents
<ImportFileName>.rej All records rejected because the PIN or last name field is empty or unknown.

This file can be used for subsequent import runs after correcting the reported problems (as only the problem cases are left).
<ImportFileName>.mncp PostalCodes/Municipalities that were not found in GLIMS. The affected persons will have an unknown municipality reference.
<ImportFileName>.hcpr HCProvider mnemonics/codes that where not found in GLIMS. The affected persons will have an unknown FamilyDoctor reference.
<ImportFileName>.paya Lists FundMnemonic/PolicyNameCode combinations for which the fund or the policy name was not found in the GLIMS database. The payment agreements for that fund and policy name are not created.
<ImportFileName>.lng Invalid language codes will be logged in this file.

Records with format errors cannot be processed and will cause an immediate error message, with the line number on which they occurred. In that case, the operator has the choice to continue or abort the import.

If the problem files are empty after the import phase, they will automatically be deleted.

Remarks

Person comparison

A person defined with the same characteristics will be created in the GLIMS database only once. The comparison of a person is done with the following fields: LastName, FirstName, MiddleName, BirthDate, PersonSex, SpouseLastName, PIN. If all those fields have the same value, an existing person was found and will not be created again. Note that the fact a person record already exists in the GLIMS database does not imply that the remaining import data of that person is skipped.

Postal code - municipality

The patient import program requires that any specified municipality already exists in the GLIMS database. A municipality that is not found is logged to the ".mncp" diagnostic file.

  • The program first attempts to find a unique match for the postal code only. If found, and the name also matches, it is accepted. If the name doesn't match, but no other municipality with the same name and different postal code can be found, it is accepted anyhow. This allows import of misspelled municipality names.
  • If no unique postal code match is found, the combination of postal code + municipality name is searched, like before.

Belgian examples:

  1. A patient is imported with postal code "9052" and municipality "Zwynaarde". In GLIMS, the postal code "9052" is unique, but matches the municipality "Zwijnaarde". If GLIMS contains no other "Zwijnaarde" with a different postal code, then the import will be accepted.
  2. A patient is imported with postal code "9052" and municipality"Destelbergen". In GLIMS, the postal code "9052" is unique, but matches the municipality "Zwijnaarde". Moreover, "Destelbergen" is also found with code "9070". The import will be rejected.

Payment agreements

In order to create a payment agreement for the patient, the fund and the policy name must exist in the GLIMS database. The import fields 'FundMnemonic' and 'PolicyNameCode' are fields that are not imported in GLIMS. A payment that already exists with the same fund, policy name and matriculation for the current patient will not be duplicated. The 'FundMnemonic' field is to be set '?' when no payment agreement creation is wanted. Newly created payment agreements will get either the patient birth date or 1/1/1970 as start date, whichever is highest. However, if the fund has an identification code with a start date that is more recent than the patient birth date and 1/1/1970, than this date will be used as start date of the payment agreement.
