Unofficial and plugin releases

Unofficial releases (hotfixes)

Introduction

To provide a quick solution to an urgent problem, MIPS sometimes offers separate .r files which are used by the application. These .r files are not part of the official GLIMS releases.

To manage these .r files, MIPS makes an inventory of all .r files distributed to our customers. These .r files are not used by the application unless they are accompanied by an associated .info xml file which is to be provided by MIPS. This XML file will list all unofficial releases (.r files) that MIPS has registered for a particular customer.

Installation

  1. Copy the .r file and its associated .info xml file in the GLIMS root folder.

    Notes

    • Depending on the type of issue solved by the .r file, the files need to be copied only on the server or both on the server and clients. Please verify this with the MIPS development department !
    • In case of a UNIX server, you may need different .r files for the server and for the Windows clients.
  2. Restart all GLIMS sessions to activate the .r files.

Usage

When starting the application, all .r files in the PROPATH are detected and each one is investigated:

IF THEN
No info file exists all .r files will be renamed to .old
If the encrypted control string in the info file is incorrect, or the database / operating system is incorrect An error message will be displayed and the .r files will be renamed to .err
If the minimal version required for the .r file is not available An error message will be displayed and the .r file will be renamed to .new
If the knock off version for the .r file is lower than or equal to the actual application version. The knock off version is the version in which the solution provided by the .r file is officially released. An error message will be displayed and the .r file will be renamed to .old
the .r file is validated The .r file will be used by the application and is visible in the tab page Unofficial releases of the About box.

Plugin releases

Introduction

In some cases, MIPS must provide functional modifications earlier than the next official release date. This can be due to legal requirements (e.g. billing). These functional modifications can be distributed to customers using dedicated .pl files (plugins). Each .pl file must be accompanied by a file with the same name + extension .info, being an xml file that validates the .pl file.

Installation

  1. Copy the .pl file and its associated .info xml file in the lib folder of the GLIMS installation folder.

    Note

    • Depending on the type of functionality provided by the .pl file, the files need to be copied only on the server or both on the server and clients. Please verify this with the MIPS development department !
    • In case of a UNIX server, you may need different .pl files for the server and for the Windows clients.
  2. Restart all GLIMS sessions to activate the .pl files.

Warning

Do NOT put the new PL-file in the PROPATH variable of the progress.ini file !

Usage

When starting the application, all .pl files that are found in the same directory as func.pl and corr.pl ( i.e. the lib dir), are investigated:

IF THEN
No info file exists for the pl-file The .pl file is ignored
If the encrypted control string in the info file is incorrect, or the database / operating system / server identifier is incorrect An error message will be displayed and the info file will be renamed to .err
The info file's structural version differs from the application's structural version

Or

The info file's knockoff version >=the application's version. The knock off version is the version in which the functionality provided by the .pl file is officially released.

 A warning message will be displayed and the info file will be renamed to .old. The .pl file will also be renamed to .old
The info file's minimal version > the application's version A warning message is displayed and the .pl file is ignored
the .pl file is validated The .pl file will be used by the application and is visible in the tab page Plugins of the About box.
