Order entry: Requests

Introduction

This page contains an embedded browser of requests contained in the order. The browser has ten columns, showing for each request: the template type, template mnemonic, root request definition mnemonic, root specimen sequencer, template description, object date and time, urgency, chargeability and billing mark. On top of this browser are widgets used for the creation of new requests.

Request selection

Request

This field references the root request definition to be used for request creation.

Label

The label of the request field also shows how many requests or root requests (See Details option) are currently in present in the order. 

Pressing Ctrl-F activates a browser where multiple request definition records can be selected. The same search mechanism can be obtained in the embedded browser of the requests page by hitting the Insert key. The request definition browser features an Options button to narrow down the number of request definitions in the browser.  The 'type' settings are remembered per user.

A root request definition is any request definition that is actually used for requesting. If this root request definition is a panel, it will be expanded to its contained panel members, and the browser will show a line for each panel member. Because every line shows the root request definition, you are able to see for each request whether or not it was requested through panel expansion. Each panel member has a sequence number which allows to specify in which order the panel should be requested. This can be important when the panel includes explicit material requests or when the order in the reports depends on the order in which the requests are entered.

The field next to the Request list field allows the user can enter a discriminator. Subsequently entered requests will inherit this discriminator and will receive a dynamic type corresponding with the discriminator format.   This can be useful when several requests with the same dynamic discriminator have to be entered. For more information, see dynamic test requesting

Request syntax

The request syntax allows for instance to:

[-]<rqbl>[=<value>][,[-]<rqbl>[=<value>]]...

where:

  • entering a '-' (minus) in front of the request definition mnemonic will cause removal of the request definition
  • appending '=<value>' to the request definition mnemonic will assign an initial result value to the request definition (e.g. patient weight, urinary volume, ...). Note that this kind of result entry has some drawbacks:
  • Although QC-checks are applied, resulting warnings will be suppressed.

  • The assessment method rejection limit check can not always be applied since the result has not yet been scheduled and the assessment method is not yet known.

    To cope with this, the result is compared with the minimum and maximum of the rejection limits of all the enabled assessment methods for this property.  When the result lies above or below all these rejection limits, it is rejected.  

    Note that no high rejection limit checking takes place when one of the high rejection limits is unknown.

  • In case of panel request definitions: 

Typing Panel=Value results in assigning the value to all properties request definitions of that panel (e.g. Diff=Not enough sample). The entered value is not used for material request definitions within the same panel.

Examples

Na,K,Cl   add Na, K and Cl
-Na   remove Na
WGHT=27   add WGHT with initial value 27
WGHT=72   update WGHT value to 72
Na,-K,Cl=102   add Na, remove K, assign 102 to Cl
Serum=1232343 results in Serum specimen with internal id = 1232343
Diff=NES results in Mono=NES,Neutro=NES,EOSINO=NES,...
SerumAnNaPanel=140 results in Serum with generated internal id and Na with value 140

Request are ordered as follows: existing requests are grouped per root specimen, headed by the root specimen itself; newly typed in requests are displayed on top.

To the right of the request definition spinner are the request object date and time, urgency, chargeability and billing mark fields.

Object date / time

You can specify a default object date in the order entry options.  When no default object time is specified in the order entry options and when opening the order entry screen, none of the time and dates are filled in. GLIMS will then take the system time (as from the database) at the moment the first request is added (date or time not filled in by user).

So, the object time taken (when none is specified), is the time the first request for the order is entered. When an order is saved and one enters the first request for the next new order, the object time is reset and the time the first request is entered will be used for the object time.   The time displayed in the status bar shows the object time of the current order. When entering a new order, GLIMS displays the object time as soon as the first request is entered.

When GLIMS considers the specified object time to be improbable, a confirmation message is shown.  The date range for an improbable object time (as compared to the reception time) can be configured in the order entry options.  This confirmation message is displayed if:

  • The environment variable "OrderEntryObjectTimeConfirmation" is set to "Yes" or,
  • A HIS connection is detected (as sending an incorrect object time from GLIMS to the HIS may result in incorrect admission data sent back to GLIMS).

By setting the environment variable "OrderEntryObjectTimeConfirmation" to "No", the object time confirmation message will be completely suppressed.

Detail

If the Detail option is not activated, only the 'root' request definitions are visible. These are the requests which are actually typed in by the user, without panel expansion. This may facilitate the process of checking whether everything on the request form was requested.  Deleting a root request via the browser will also delete its dependent requests.

If the Detail option is activated, the implicit requests (such as the members of the requested panels, and material requests added by the scheduler) as well as configured urgencies and discriminators are also displayed. Implicit requests (other than panel member requests) can easily be distinguished from other requests: their type is followed by an apostrophe.

Specimen

The Specimen spinner can reference any specimen request in the order. The label of the Specimen field shows the number of root specimens currently present in the order. A root specimen is a specimen which is not derived from another specimen (no centrifuged blood, for example). In practice, first enter a specimen request code (EDTA) in the Request field. From then onwards, all properties you request will reference the same root specimen.

Dynamic test requesting

GLIMS offers several ways to request dynamic test. Please see the chapter Requesting tests with a discriminator.

Dealing with overlapping panels

Sometimes, orders contain requests for panels that have common members.  Requesting these panels is no problem: GLIMS automatically detects that a request is already present and will simply ignore the additional panel member request. However, GLIMS also has to avoid that when deleting a first panel, the members of this panel are not deleted from the order if they are also required for another panel.

To deal with this situation, the following rules are followed:

  • During the requesting phase, the program keeps track of the fact that a test was requested via different panels. This is also visible by the fact that all requesting panel mnemonics are shown in the request mnemonic column of the order entry request browser. When saving the order, this information is saved in the Request.PanelList field.
  • When entering "-X", the program first checks if "X" was one of the original request definitions used for this order. If so, then all the requests where the root request definition mnemonic equals "X" or where "X" appears in the panel list are treated.  Deletion of such a request is done by first removing the "X" from its panel list. The request will only be removed when its panel list becomes empty.

Note

"MONO" + "DIFF", followed by "-DIFF" will result in an order where MONO is still requested, since it was not requested via the DIFF mnemonic.  If "X" was not one of the original request definitions, then all the requests of tests, specimens, microbiology action, ... within this panel are removed from the order, regardless of how they where requested and via how many panels they where requested.   

This will allow entering requests like "HEMA" + "-RBC", indicating that the HEMA panel should be requested, except for RBC, or "HEMA" + "-DIFF", indicating that we want the HEMA panel without the DIFF members (even though DIFF was not used in the positive requesting phase). This also implies that entering "-X" where X is not a panel request definition will always result in the removal of the requests where the actual property, material, microbiology procedure, ... corresponds with the one referred to by the "X" request definition.

Note

Deleting a request by pointing to a request in the order entry screen's request browser and deleting it via the "<Delete>" key will result in the removal of the request, regardless of its panel list.

In other words, when deleting via a mnemonic that was used in the requesting phase, only the requests actually requested via this mnemonic are deleted. When deleting via another mnemonic, all requests corresponding with tests represented by the mnemonic are removed, regardless of their original request mnemonic.

Paired samples

During order entry, you can hit the F9 function key when positioned in the Request field.   

This option allows to add a request for a property which has to be assessed explicitly on an already existing specimen for the same patient (as is often desired in serology/toxicology labs).   Although the request is added to the CURRENT order, it will refer to the existing old specimen (which was originally requested for another order).   How this is reflected on paper reports mainly depends on the report parameter settings (cf. the ReportTemplate.Grouping field with as interesting values GroupByOrderByDate or GroupByObjectTime).

By pressing the F9 function key, a browser is displayed on screen showing a list of known results/specimens of the current object of the order.  It is possible to select the specimen to be reused either from a specimen or result browser.  Your preference can be set with the order entry option Method to select paired specimens.

Upon selection of one of these specimens or results, the order entry program will automatically impose that all subsequent requests for this order use the selected specimen or the specimen of the selected result as root specimen, without disturbing the LowestObjectTime of the current order.

Additional request fields and request browser

Date and time

The Object date and time denote an important time to be associated with the result or specimen. We use the qualifier 'Object' because they indicate at which time in the evolution of the object (patient) a snapshot was taken, typically by sampling a specimen from the object. Note the difference between the assessment time of a result and its object time (sampling time): the result can be assessed at any time after sampling, but it stores a property value (RBC count) describing the object at the object time! The object time has to be provided with the result for interpretation, whereas the assessment time is mainly used for logging purposes (GLP accreditation). GLIMS uses the object time of a property assessment request (Na) to make sure that the assessment will be performed on a specimen with a correct matching object time.

Urgency

The Urgency field allows you to choose one of the urgency levels defined in the site domain Urgency. If no level is defined, GLIMS defaults to a number range of 1 to 9.

Chargeability

The Chargeability field can contain either Yes = "charge" or No ="don't charge". It can also be left empty, in which case the request definition's chargeability will be applied.

Billing mark

The Billing mark field allows you to choose one of the (enumeration) values defined in the site domain BillingMarks.  You can also specify an Order billing mark (OBM), i.e. a default billing mark for the entire order. If the latter is specified, this billing mark is used for each request where no explicit billing mark is specified (including requests added later on, e.g. via MISPL). When an order is rewritten and only 1 different billing mark is used in its requests, then the order inherits this billing mark.

Request flag

The Request flags field allows to indicate that a request was added to the order on the initiative of the lab or on the initiative of the issuer (e.g. ordered by phone). This may have important billing implications. This field behaves a lot like the Billing mark field, but the only possible values currently supported are L to indicate a lab-issued request and Ito indicate an issuer-issued request. These values are stored in the Request.Flags field.

Request form

The Request form button allows the selection of a request form from which requests can be chosen.

contextual menu functions

In the request browser, menu functions can be applied to one or more requests. To this aim, select a request (or several requests using the CTRL key), right-click on the selection and choose one of the following functions:

Internal comment

This function allows to enter comment used for internal purposes (e.g. on work lists).   When adding internal comment to a request definition, the request definition type (first column of embedded browser of the request page) is marked with an asterisk and is recognizable as such on the work list.  You can enter free text or use the Insert key to add a text reference or a text expression.

Note

When choosing the function while more than 1 request is selected, GLIMS will ask for a single comment which will be applied to all selected requests. 

External comment

This function allows to enter comment used for external purposes (on reports). When adding external comment to a request definition, the request definition type (first column of embedded browser of the request page) is marked with an asterisk.  You can enter free text or use the Insert key to add a text reference or a text expression.

Note

When choosing the function while more than 1 request is selected, GLIMS will ask for a single comment which will be applied to all selected requests. 

Apply current request attributes

This function offers the possibility to apply current request attributes, i.e. urgency code, to charge(Y/N), to a selected range of requests. 

Example

Enter a billing mark, select a request and choose the contextual menu function Apply current request attributes. The billing mark will be indicated with the request. In order to remove this billing mark, just type a blank as billing mark, select the request and choose the same function. The billing mark will be cleared.

Apply current object time

This function takes Result.DynamicType into account. For instance, a relative dynamic test with object time 08:20 and reference time 08:00 will get object time 09:20 and reference time 09:00 after changing the object time from 08:00 to 09:00 using this function. 

Note

The object time (date) can only be changed if the order has not yet been saved.

Change root specimen

When directly requesting materials and then assigning requests to these specimens, we speak about root specimens. The Change root specimen contextual menu function allows to change the root specimen of a selected request. Activate this function and a box will appear with the available root specimens.

Property blocking period and property reusability

Blocking periods

Fixed blocking period

It is possible to specify a so-called 'blocking period' for each property. This is the minimal time span in days that should exist between two requests for the same property and the same patient. This time span can be specified in the Property.BlockingPeriod field.  Note that the property blocking period check ignores results that were not assessed (i.e. Result.AssessmentCount = 0). This may occur when a result code is applied (e.g. "NM = No material received") with ResultCode.Assessed = NO.

The blocking period check is executed against the latest effectively requested and assessed result, ignoring any already blocked requests in between. Therefore it is necessary that a 'blocked' result can be distinguished from a normal one.  This has been accomplished by the .Original field referring the original (assessed) result, while the .IsDuplicate field is still = NO. This means: it isn't a copy, but it refers an original result record (that caused the blocking). 

This also facilitates putting comments on reports. The previous result can be easily accessed via .Original, as in the following example:

Example

DateTimeToString(.Original.ObjectTime, "%x %H:%M")

Note

When an existing result within the blocking period is detected during order entry:

  1. For explicitly requested properties, if a blocking period applies,
    • A question will appear in manual order entry (The user is offered the possibility to ignore this and create a new request, reject the request, or accept the blocking period and create a result with a message in its value to warn the issuer about this. In the latter case, the result's value is filled with the text in Property.BlockingPeriodMessage (or with a fixed text when not specified), and is validated automatically without being scheduled. The new request will be created with ToCharge = NO.).
    • In electronic order entry, the result is automatically blocked.
  2. For implicitly requested properties, if a blocking period applies,
    • The result is automatically blocked.

GLIMS also supports decimal values such as "0.5" (half day) as blocking period.  

Calculated blocking period

It is also possible to define a variable blocking period for each property.  The blocking period could for instance be different for general practitioners and specialists.  To support this, you can use the field "Calculate blocking period" for properties.  This field allows specifying an Order-based, Integer-valued MISPL expression.  If the MISPL expression returns an integer, the latter will be considered as the blocking period.  If the MISPL expression returns a "?", the fixed blocking period will be used (RETURN StringToInteger(?);).

GLIMS also supports decimal values such as "0.5" (half day) as blocking period.  

Note

When adding a request to an order, GLIMS will only considerAvailable specimens when checking if the blocking period is respected.

This implies that it is possible to request a request code multiple times for the same object within the defined blocking period, as long as the specimens on which the requests are scheduled are not available yet.

Result reuse

Result re-usability

When the existing request also lies within the property's re-usability period, a chance is offered to reuse this existing result. In this case, a warning about the blocking period is added to the Reuse dialog shown to the user. In the Reuse dialog, just as in the Blocking Period dialog, it is also possible to completely cancel the request (except when the request is automatically added via the scheduling process, e.g. as input for a calculated result).

When a result is reused and the new field Property.ReusabilityComment has a value, this text will be added to the duplicate result's external comment, thus allowing to indicate the reuse of an existing result on the report.

Practical example

PSA cannot be requested again within a time span of 14 days:

Example 

Blocking period = 14
Blocking period message = (gp_Text) Despite of the new request on {=DateTimeToString(.ObjectTime,"%d-%m-%Y")} 
(i.e. within {=FractionalToString(.Property.BlockingPeriod,"%2.0f")} days 
after {=DateTimeToString(.Object.GetResult("PSA", ?, 2, ?).ObjectTime,"%d-%m-%Y")}), 
{=.Property.Mnemonic} was not determined again.
