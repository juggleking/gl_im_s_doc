Order entry options: Scheduling

Automatic sample login

Determines whether the sampling actions, required to satisfy the requests, are scheduled or assumed complete. If activated, all specimens are assumed to be available : their status Expected will change to Available.

Note

This only applies to root specimens (i.e. specimens that are output of a sampling procedure). If this option is set, root specimens are automatically confirmed (status = Available) and no sampling actions are created. The receipt time of the specimens is set to the current time.

Example

Example

A specimen is sampled at 11 am.

The lab receives the specimen and creates an order with this specimen at 2 pm.

--> The sampling time of the specimen in 11 am. Its receipt time is 2 pm.

If the root specimen is to be processed into sub-specimens (through a dedicated procedure), these sub-specimens will NOT be confirmed automatically. For the sub-specimens to be confirmed automatically, the option Auto-login of the material output (of the processing procedure) must be activated.

To sum up, the combination of Automatic sample login and Auto-login allows a full automatic confirmation of both root and work specimens.

The general setting Auto sample login per department allow you to specify that this automatic sample login (when activated in the order entry options) should ONLY be done for specimens which are input to at least one procedure of the user's own department.

Note

When adding requests to an order AFTER the order has been created, the schedule options for AllowSpecimenReuse, AllowResultReuse and AutoSampleLogin are reset to the values during the creation of the order.

Mark order as synchronized

With this option a site function on the Order table can be defined, which returns a Logical and thus manually updates the Synchronized flag of the order in question to Yes.

MISPL

RETURN TRUE;

In this case, any subsequent object time forward / backward synchronization will have no further influence on the order, which is useful if no more changes to the sampling time in the order are intended. The Synchronized value of an order can be checked with the following site function on the Order table, which returns a Logical value:

MISPL

RETURN .Synchronized;

This functionality is also available for the Create order series function, via the Order entry User options (Order entry > button User... > Order activation).

Allow to reuse specimens from previous orders / Allow to reuse results from previous orders

Can be used to enable or disable re-use of existing specimens and results respectively, regardless of the configured re-usability values (material's re-usability and property's re-usability).
  • GLIMS will only propose to reuse a specimen if the "Specimen scope" defined for the involved procedures is identical.
  • For explicitly requested properties and materials:
    • If a re-useable result/specimen exists, a question will appear in manual order entry. In electronic order entry, the result/specimen is automatically reused.
  • For implicitly requested properties and materials:
    • If a re-usable result/specimen exists, it is automatically reused.

Note

The Automatic sample login, Allow specimen re-use and Allow result re-use settings used during the last manual update of the order are stored and only when adding requests via MISPL, these settings are reused, i.e. not when manually adding requests to the order.

Moreover, the action scheduler will not attempt to schedule reused results or specimens, even when they were never scheduled before. This is especially important when reusing old imported results or specimens, for which there is no known material output or property output. Scheduling these old results or specimens could lead to undesired effects (e.g. executing a property trigger on a reused blood group).

Note

When adding requests to an order AFTER the order has been created, the schedule options for AllowSpecimenReuse, AllowResultReuse and AutoSampleLogin are reset to the values during the creation of the order.

Action scheduling mode / Postscheduling mode / Report scheduling mode

These options allow the user to determine for each phase of the order post-processing if it should be executed locally or on the server.

GLIMSRemoteCalls

The GLIMSRemoteCalls environment variable needs to be set to YES. If GLIMSRemoteCalls is set to NO, then the action scheduling, post scheduling and report scheduling will be executed locally, regardless of the Action scheduling mode, Postscheduling mode and Report scheduling mode.

Action scheduling mode

This option determines how the action scheduling and the evaluation after scheduling are executed. Possible values are:

Local

Execution on the client. Use this option if interaction with the user is required in the procedure eligibility, result output or evaluation after scheduling MISPLs.

Serversync (default value)

Use this option

  • when no interactive MISPLs are used and
  • when the user may need to specify material variables for the implicit specimens created during action scheduling or
  • when the post-scheduling or report scheduling phase will be executed locally.

Serverasync

Use this option when interaction with the user is not necessary during the entire order post-processing. This option is only possible when the post-scheduling and report scheduling phases are executed on the server as well. The entire order post-processing will then be executed asynchronously on the server while the user can continue working in GLIMS.

Notes

  • Any error messages from action scheduling will not be shown to the user.
  • Material variables for implicit specimens that have been created during action scheduling cannot be asked for, and the values of auto-prompt results that have been added automatically cannot be asked for.

Postscheduling mode

Local

Use this option if interaction with the user is necessary via interactive MISPLs.

Server (default value)

Execution on the server which is faster than execution on the client but interaction with the user is not possible.

Note

The post-scheduling phase consists of:

  • Microbiology action post-processing (Next station MISPL of microbiology procedure and carrier generation)
  • Genetic exam post-processing (approach plan creation)
  • Order set determination
  • Specimen size calculation
  • Label printing
  • Obtaining patient information from external systems

Report scheduling mode

Local

Use this option if interaction with the user is necessary via interactive MISPLs.

Server (default value)

Execution on the server which is faster than execution on the client but interaction with the user is not possible.

Evaluation after scheduling

This evaluation is run after the action scheduling phase (if required) and before the post-scheduling (specimen size calculation, label printing...) and report scheduling phases.

When the evaluation after scheduling is executed on the remote server (see Action scheduling mode), any errors resulting from this evaluation are simply logged but have no influence on the commit process.

The evaluation after scheduling can be run on the remote task server for performance reasons. When the evaluation after scheduling is run, the action planning phase has passed. The latter may be important when you need to inspect the planned actions.

Scheduler verbosity

Determines the amount of messages sent by the scheduler, upon committing the order's new requests. Use it to verify your procedure configuration. Depending on the value entered (1-9) the user is forced to justify each step in the order procedure (e.g. material, properties, ...).
