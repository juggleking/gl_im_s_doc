Order entry: object

Introduction

The Object page allows selecting the person, animal or lot for which lab requests are to be executed.  The fields shown depend on the object type chosen in the Type field.  In the order entry options, you can set a default Object type that will be used by default whenever you start the order entry program.

The most important object fields are shown in the Main fields section.

Persons

Fields

PIN

Allows looking up a person by its identification code. 

PIN search in local Glims database

A person can have several identifications from different identification sources.  The identification source used for looking up the specified identification is determined as follows:

  1. The order entry option Person identification source.
  2. The user preference Default identification source.
  3. The identification source with the lowest sequence number in the PIN provider group specified in the general settings .

  4. The person identification source is specified on SpecificSite level (Start -> System management -> Customize -> Glims general).

The main page of the order entry options also has a field Identification source access which determines if the identification source is hidden in the order entry screen (hidden), if it is displayed but cannot be changed (read-only) or if it is displayed and can be changed (update).

PIN search in external system (e.g. HIS)

If the identification source specified in the Default identification source field of the user preferences refers to a translator (which is active), a Query button will be shown which allows looking up the PIN in an external system.  The translator specified will handle the query to the external system.  If no identification source is specified in the user preferences, Glims will check if a person identification source is specified on SpecificSite level (Start -> System management -> Customize -> Glims general).

The Identif. source access in the main page of the order entry options screen will determine if the user can see and/or access the source used for the external query.

When one or more patients are found in the external system, Glims will show a selection screen.  For more information about this screen, please see the chapter Querying persons in an external system.

Note

  • If both the PIN and encounter are specified, two query buttons will be displayed that allow to query the patient either by PIN or by Encounter.
  • F8 will launch a query by PIN if the field PIN has focus. 
  • F8 will launch a query by encounter if the field Encounter has focus.

Note

  • The Order Entry OMR scan program can read the Pin Provider from an OMR form. The scanned Pin Provider will then be visualized in Order Entry (unless the Identification source access field was set to 'hidden', see below).
  • GLIMS supports multiple patient identification sources simultaneously. However, only one identification source can be active at a time during order entry!

Internal id

Allows selecting a patient by a GLIMS internally assigned identification for the patient.

Unpaid amount

When selecting a patient the field Unpaid amount is calculated.  This is the unbalanced amount of all fixed invoice summaries of that patient.  For orders with object type Animal, this is the unbalanced amount of all fixed invoice summaries of the animal's owner.

Note

the calculation of this amount, which can take little time, is disabled when the field is hidden (F12 for screen customization).

Access to person relation browser

This button is displayed whenever a person is selected in the object page of the order entry screen (and when the HLA or Genetics license is available). It allows access to the person relation browser. When the selected person has relations, the text next to the button will indicate how many relations there are.

Updateable fields

When selecting a patient in GLIMS, some fields can be updated from within the order entry screen:

  • Person.LastName
  • Person.FirstName
  • Person.BirthDate
  • Person.Sex
  • Person.Address
  • Animal.Name
  • Animal.Owner
  • Animal.BirthDate
  • Animal.Sex
  • Lot.Supplier
  • Lot.InternalId
  • Lot.ExternalId
  • Lot.ProductionDate

Note

When the user does not have the Person update privilege, he will not be allowed to change person or correspondent data in this page. The fields will either be disabled upon choosing a person, or he will receive an error message upon trying to commit his changes.

Automatic creation of identification and encounter for new patients

When a patient is created via the order entry program and the current person identification source (or PIN provider) has a valid MISPL function in Correspondent.IdentificationCode, this function is evaluated in order to calculate a new, unused Identification Code. When successful, a new identification record will be created, connecting the patient with the PIN provider and the calculated code will be assigned to it.

The same holds true for encounters: when no active encounter exists for the currently chosen patient in the currently chosen institution (via the Ward field), and the institution has a valid MISPL function defined in Institution.EncounterExternalId, this function is evaluated in order to calculate a new, unused encounter external id. When successful, a new encounter is created, using the Encounter Type as entered in the order entry screen, connecting the patient with the institution, and the calculated external id will be assigned to it.

The encounter type field Automatic creation allows customizing the automatic creation of encounters in GLIMS.  If set to 'Yes', GLIMS allows automatic creation of encounters for this encounter type.  If set to 'No', GLIMS does not allow the automatic creation of encounters for this encounter type.

Manually creating and updating encounters

Creating or updating an encounter via the order entry screen requires explicit confirmation by the user, in order to avoid erroneous encounter creation or update. When an encounter has been chosen in the order entry screen (either explicitly or automatically by selecting a patient) and the user changes the encounter type, external info or the external id (to a non-existing value), the program will assume that an update to the last selected encounter is desired. If no encounter was selected prior to committing the screen, the program will assume that a new encounter has to be created.

When the field Institution.UniqueExternalId is set to TRUE, you will not be allowed to manually create an encounter with an external id that already exists in the database for the same institution. It will also not be possible to modify the external id or institution of an existing encounter when an encounter with the same external id and institution already exists.

Such updates coming from so-called 'trusted' sources will only result in a warning message. For the moment, the URL brokers (including all ZIS-connections) and the 'External patient manager' (used by the order entry patient query buttons) are considered to be 'trusted' sources.

Note

Update and create privileges on the encounter table are checked: when a user does not have update/create privileges, he will not be allowed update/create an encounter. The same holds true for the update/creation rights on the person table.
