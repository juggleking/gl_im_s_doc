Phone list - Phone the results

Contents: Introduction - Order query - Outline - One by one - Multi-user support

Introduction

The phone list program is available via the main menu through Start -> Routine -> Orders -> Phone list. The program has two modes of operation: Outline or One by one:

The last choice is remembered per user. If the function is pre-parameterized this initial dialog is skipped.  

The phone list program contains a composite query. It always starts with an order query (the so-called "Master" query), yielding all orders for which something is to be phoned.  For each individual order, a separate query is performed on all results of the order (the so-called "Detail" query).

You can also phone all results from a single order using the Order.PhoneThrough() function.

Order query

After selecting the view type, a query dialog allows looking for orders of which you would like to phone through the results.

Sampled/ Received

Allows querying on the order's sampling date/time or reception date/time.

Minimal result status

This is a very important option. It determines which orders will be selected for display. The effect on the order query depends on the view type :

One by one

An order will only be selected if at least one of its results has reached this minimal status.

Note

The Minimal result status also has an effect on the (detail) result query. Results that have not reached the minimal result status (or results with a status lower than Available) are not supposed to be phoned. Therefore, their value is hidden. That way, you can see there is a result to be phoned, but it isn't available yet. Instead of the value, the status is displayed in order to have an indication when it is to be expected. To avoid confusion, the status is shown in a different font (italic + not bold).

Note that, if the status indication is Available (which may happen if the minimal result status is set to Confirmed or Validated), you can still press F6 where the value can be seen anyhow.

Outline

In Outline mode, whether or not the Minimal result status is taken into account during order selection depends on the Only Orders with Results option:

  • With Only Orders with Results checked, an order will only be selected if at least one of its results has reached this minimal status.
  • With Only Orders with Results unchecked, all orders with results marked as To be phoned are selected, regardless of the Minimal result status.

The Minimal result status is taken into account to decide if a result will appear in the Available or Pending column: results below the Minimal result status will be displayed in the Pending column, while the others will be displayed in the Available column.

Only complete

This option is closely linked to the Minimal result status option (see above). It means that only orders should be shown where all results to be phoned have reached the Minimal result status. This option behaves identically for both view types.

If this option is enabled, the Pending column of the order browser is hidden as it is empty by definition in that case.

During the query, a progress indicator shows how many orders are verified, and how many of them are eligible to be shown in the phone list program. Orders may not be selected due to: order shielding, inappropriate result statuses or mismatching filter criteria.

General functionality

By default, only the results to be phoned or already phoned are shown.  

This button provides direct access to the phone log.
The complete order can be viewed anyhow by pressing the Show complete order button on the main screen, or by checking the "Show complete order" toggle-box in the options screen.  The button and the toggle-box have exactly the same effect and are coupled. If enabled, all results of the current order are shown, using the default property classification. Any classification specified in the order query screen is ignored in this case. 

Results with a lower status than the minimal result status are shown, but their value is masked. This is done by putting the result status in the value cell, in a different (italic) font.

The results which have to be phoned through are marked with a P in the second column. You can select or deselect results by hitting the space bar. This implies that you can also phone through results which were originally not marked as such. 

At the top of the screen, you can select a phone number which should be used to phone through the results.  The following contact data will be displayed:

  • Order's phone number (specified in the order entry screen)
  • Issuer phone numbers
  • Agent phone numbers
  • Ward phone numbers
  • Issuer's practice phone numbers

  • Patient's family doctor, mother and company phone numbers

For all contacts, the following contact data will be displayed:

  • Correspondent.Phone
  • Correspondent.Phone2
  • Correspondent.MobilePhone
  • Correspondent.Fax
  • Correspondent.Fax2
  • Correspondent.Email

By making use of the SpecificSite field Phone number sequence (Start -> System management -> Customize -> GLIMS), it is possible to influence which phone numbers are presented by the phone list program, and in which sequence.   When you want to change the default behavior, supply a comma-separated list composed of one or more of the following keywords: "Order", "Issuer", "Agent", "Ward", "Patient" or "Private". The phone program(s) will:

  • Only show the phone numbers for the tags included in the list.
  • Show them in the specified sequence.
  • "Order,Issuer,Agent,Ward" is the default.

Note

  • The keyword "Patient" includes the patient's family doctor, mother and company phone numbers
  • The keyword "Private" includes the patient's phone 1, phone 2 and mobile phone
  • If the issuer phone numbers are shown, the phone numbers of the issuer's practice will be shown as well.
  • Make sure that the list does not contain spaces between the keyword

When selecting a phone number, you use the F6 key to view the details of the contact person.

By hitting the Page Up and the Page Down key, you can scroll through the different orders.

The property column displays:

  • The multi-lingual property description. The language to be used is searched in the following order until a value is found: the report language of the order (Order.ReportLanguage), the language of the issuer, the site language, and finally the user language.
  • The property short name if no description was found.
  • The property mnemonic if the short name is empty.

An icon is available in the column header which allows switching between the property description and the property mnemonic.

Request mark and urgency columns

The phone list program has a request mark column that indicates the urgency and phone status. An urgency level > 1 is signaled by means of an exclamation mark ("!"). A result that is to be phoned is flagged by an uppercase "P" (or "T"), while a result that has already been phoned is flagged by a lowercase "p" (or "t").  

Next to the request mark column, the urgency column displays the single-character urgency code as defined by the "Urgency" site domain, but only if the urgency value > 1.   Actually, this new column makes the "!" indication superfluous, but the latter was left for consistency reasons.

Result mark column

  • If a result is unsolicited, an X will be displayed in the column with the label U.   It is possible to create a request for an unsolicited result by double-clicking on the X.
  • Click here for more information on result flags.

Browser columns

The previous result values are displayed in right-side hand cells. The dynamic time discriminator (dynamic panels) of a result is also taken into account when searching for a previous value for this result.

On entry, any available comment is displayed in the box at the bottom left-hand side of the screen. The result can be zoomed by means of mouse double click, RETURN or F6.  If the result is updateable, the Result.Modify() dialog is invoked on RETURN or F6. Else, the standard result editor is called in view mode.

Buttons

Final

Phone the currently available results, mark them and remove the other non-available results from the list. This happens silently, without any subsequent user interface. This option aims to remove the order permanently from the phone list.

Note

However, if any of the (phoned) result values is changed later on, the order will re-appear anyhow.

Partial

Phone the currently available results, mark them and leave the others untouched. Again, no user-interface follows. This option will cause the order to re-appear as other results become available.

Comment

Simply add a phone comment to the order without any further processing. The entered comment will become visible in the order overview browser (in front of the standard issuer comment, if any).

Options

The Options button allows to further narrow down your result selection criteria:

General

Result status from-to

These options have absolutely NO relation with the minimal result status option of the order query described above. In the (detail) result query their meaning is: only display results within this status range. This applies to any result that is eligible to be displayed.

Enforce comment

when checked, the user will always be prompted for comment when using the "Final" or "Partial" buttons.  The initial value of this option is 'NO'.  The gp_site.PhoneList() or Order.PhoneThrough() functions can both be preparameterized however to impose all users entering phone comment.

Show complete order

The default value of this option is "NO". It means that only results to be phoned or already phoned will be shown. When this option is set to "YES", all results of the current order will be shown, using the standard property classification.

Even when the Hide unsolicited results toggle-box is checked, to be phoned unsolicited results will be shown in the browser.

Remarks 

  • Results without a value will not be displayed in the phone list.
  • The result browsers can detect that the displayed result value is too wide to fit in its column. This may lead to misinterpretation. Therefore, this is made clear to the user by replacing the last part of the value with the '@'-sign.  

  • Note

    Such a situation may sometimes quickly be resolved by double-clicking the frame title in order to maximize the browse window.

  • As there is no means to make results with Property.Visible <> "Yes" visible in the phone list program, such results can never be processed and may cause an order to keep on re-occurring on the phone list without (at first sight) a reason.  Therefore, several programs disallow putting such results on the phone list:
    • The Minimal/Maximal urgency trigger on SpecificSite
    • Result.Modify()
    • Result.ToBePhoned()
    • The AssessmentMethod.PhoneSeverity mechanism

Warning

When trying to phone an already phoned order: to avoid a user from inattentively phoning an order that has already been phoned, some mechanisms have been built-in that should draw the user's attention to such a situation:

  • A transient message "Nothing to be phoned" now appears for about 1 second, unless earlier removed by pressing any key.
  • The browser background color changes from white to gray.
  • The Final and/or Partial button may become insensitive.

Multi-user support 

The phone list program provides multi-user support in both One by one and Outline mode. This means that users, which are simultaneously processing the same phone list, will not interfere. Orders processed by one user will automatically be skipped or hidden for another.

View type "One by one"

When an order is shown, it is marked (or locked) as being phoned for the current property classification. This lock is held until the user leaves the phone list program completely or re-executes the query.  When moving forward through the query, each new order is checked whether another user has already marked it for the same property classification. If so, the order will be skipped.

View type "Outline"

For this mode, multi-user support has been implemented as follows:

  • Orders are allocated in blocks. A block is a group of orders to be phoned together. This grouping depends on the Sort by option, and is visualized by means of the alternating background color of the first two columns. From the moment that one order of a new candidate block is marked as being phoned by another user, the complete block will be skipped.
  • A block is allocated to the current user from the moment that either the Phone through or Skip button is pressed while the block is focused. Explicitly skipping a group of others is interpreted as "I had a quick glance at the orders and decided not to phone them for the moment". As a result, another user shouldn't be bothered with the same group of orders.
