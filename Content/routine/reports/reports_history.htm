Basic routine -> Reports -> How to add a complete history on a text report ?

When activating the Add history toggle-box in the report generation user interface, GLIMS will add the complete history of the current patient in the form of cumulative columns to the current report page when printing the report, much the same way as cumulative columns are added to a standard cumulative order report.  An additional and mandatory "since" date field limits the history to a specified start date.

Regardless of any history or previous result settings, the report generation program will not add previous orders to the report when the report template does not specify that there is more than 1 result column (i.e. ReportTemplate.ColumnCount > 1).  When the total amount of columns exceeds the maximal number of columns defined in the report template, multiple pages (or 'Subsections') will be printed for the current report, each having its own Subsection Number (starting from 1).

The sub section number of a report page (and the amount of subsections) can be accessed in the header or footer of the report by using the {SubSection} and {SubSectionCount} variables. The fact that {SubSectionCount} > 1 indicates that we are dealing with a multi-section report page.

Also note that the columns added take the cumulation settings of the original report into account:

  • The original report may state that only orders of the same issuer should be allowed as cumulative columns.
  • When the original order was a study order, only orders of the same study are shown.
  • Study orders are not shown on non-study reports.
  • The DefaultReport.PreviousResults settings (None/Property/Chapter/All) are taking into account, unless when set to 'None', in which case 'All' is assumed.

Although adding history this way can be used on any text report, it obviously works best for order-scoped reports, since the combination of adding cumulative columns to an object or encounter report page (who already have a number of historical columns on each page) may produce confusing effects.

Note

The Section Number of an object, year or encounter report can also be accessed in the same way as the above described subsection, by using the {Section} variable (as alternative to the more generic &DocSeqNo reference).
