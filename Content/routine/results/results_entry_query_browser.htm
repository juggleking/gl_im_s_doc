Result entry, query and browser

Contents: Result entry - Result query - Extended result query - Result browsers

Result entry

There are three widespread ways to enter results in GLIMS:

  • Soliciting a request definition in order entry and instantly entering a result: hb=5, chol=140 etc. This method works very efficiently, but does not offer any work list functionality. It might occur, that the length and weight is directly entered in the order. If so, it is much easier to enter len=165,gew=65.
  • On the order table, you can select the popup function Edit result & specimens: it is not the fastest way, but a good way to enter results, as it gives all work list functionalities, such as choices, ... (see Editing a work list).  Please also see the chapter User interface: grid options.
  • Through a work list.  The result entry function is activated from the main menu (Start -> Routine -> Work lists -> Result entry) and delivers a work list selection screen.

Advanced editor

If this option is activated, the Work list editor opens in the advanced format. If it is not activated, it opens in the classic format.

Entering a value

The value you enter in the result cell is automatically checked against a number of rules. These rules may prevent the result from being stored, if it is obviously wrong. They may also automatically set result attributes such as severity and comments.

For numerical results, the expected unit is shown on the status bar at the bottom of the screen. Unless a unit is explicitly specified, the expected unit will be assumed. If a different unit is specified by the user, it will be accepted if a conversion to the expected unit is possible.

Unavailable specimens are automatically confirmed upon result entry.

The following is an explicit list of actions performed on input values:

  1. The quality of the channel (assessment method) is checked. If it is Unreliable (QC alarm severity was transgressed or manually set to Unreliable), the user is given the choice to either abort input until reliability is restored, or to continue input. In the case, the result will be marked internally as obtained through an unreliable channel.
  2. The raw input value is lexically converted by the method's conversion expression, if one is specified.
  3. The raw input value is parsed into an internal, normalized representation. This representation will be stored permanently, along with the raw input value.
  • For numerical results, the representation consists of a comparator, a mantissa and an exponent, expressing the value in the expected property unit, with the expected precision. The parser takes care of unit conversion and can deal with scientific notation (for example, 1e-3 instead of 0.001).
  • For multiple choice results, the representation consists of a reference to the selected choice. The parser can deal with a viable prefix instead of a complete choice value.
  1. For numerical results, method specific range checks are performed. Range checks may truncate or reject the input, as specified in the associated assessment method record.
  2. For numerical results, norm and delta norm checks are performed. These checks set the associated result severity levels.
  3. Any non-trivial severity levels and generated comments are displayed in an alert box. The user is given the option to cancel storage of the input value.
  4. The result is stored. It is also automatically confirmed or validated if allowed by the configuration.
  5. Trigger code, defined at the property level, is executed. This code may cause reflex testing (also called cascade requesting), as illustrated in the example below.

Zooming to the result editor

In most of the programs, when zooming into a result (F6), not the standard result editor is shown but the Result.Modify() screen. 

The Result.Modify() screen has several advantages in comparison to the classic result editor:

  • It only contains the most essential information and shows this on one page.
  • It allows to zoom large (string) result values.
  • It allows to expand text modules embedded by means of "{< }".
  • It provides immediate access to the phone functions.

Users for which access to the Result.Modify() function is denied will also get this screen, but all widgets will be read-only. The Expand buttons are always enabled, but users with insufficient privileges will be unable to save the expansion.  Click here for details on privilege handling by this program.  

The standard result editor is available under the Detail button of the Result.Modify() program:

External comments are displayed on outbound reports.  Internal comments are for internal use only.

The Log field (also available via the Result.ViewLog() method) gives a detailed description of all important events that took place for this result and its predecessors. This logging also indicates when a result has been modified, repeated, selected, .... together with the entered value, the rejection reason etc.

Example

Result query

Incomplete results query

The result query is available from the main menu via Start > Routine > Results > Incomplete results. The results are first sorted on specimen internal id and then using the default property classification.

Status range

The status Validated is deliberately not added in order to avoid a too large number of results.

Executing department (group)

Allows to search for orders that have at least one action that is executed in the specified executing department (group).  This is determined by the procedure's department.

Filter

If the provided query options are still not sufficient, you can still create a user-defined MISPL filter.

Extended result query

To allow querying on validated results and QC results, you can use the extended result browser. It supports the same selection criteria as the result statistics program. The results are sorted by specimen internal id + order id + object time. This query is available from the main menu via Start > Routine > Results > Extended query.

Tip

Result query and browser performance can be improved if you:

  • exclude discontinued results. The inclusion of discontinued results is likely to significantly increase the number of results considered, and so reduce performance.
  • set as many query options as possible.
  • use small ranges where a From / to value can be entered. For example, reducing the Object date range can help improve performance.

Result browsers

There are several result browsers in GLIMS: confirmation and validation screen, result overview etc.
