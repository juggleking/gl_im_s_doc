Preliminary configuration

The GLIMS-VALAB connection requires a license key.  There is an evaluation period of 30 days.  Before you can use the Valab functionality in GLIMS, the Valab interface should first be defined (as a regular analyzer connection).  The translator must use the "valb" driver.  Before the Valab program is started in GLIMS, the translator should be running.

In order to map the properties in GLIMS with the codes used by Valab, you must define a coding system with a property code set that will be selected for validation on VALAB (A-tests), or as extra informational tests (E.g. C-test).   You can specify the coding system to be used in the function setup.  If multiple properties should be sent with the same Valab code, you should use different discriminators in this coding system.

Example

Property Code
K A4
Cl A8
Info-C1 C
Info-C1 C
Info2 O
  |_ First character obliged. Indicates type of Valab test

Note

Test mapping may not be realistic in this example.

Validation of multiple results having the same test code in Valab: The GLIMS Valab program supports validation of results of an order that have the same test code on Valab. This is typically an issue for morphology results. In Valab, morphology is represented as one test, named 'A57'. In GlIMS however, morphology might be stored in several results. GLIMS sends multiple result values for the test 'A57' (as Valab expects it to be). Valab interprets the results but will send only one validation flag back.  The GLIMS Valab program is able to store the validation flag for all results of 'A57' of that order.  To activate this, one should adapt the Valab codes defined for the properties in GLIMS: Add the '*' (asterisk) character at the end of ALL A57 codes.

Station specific property codes for Valab validation

The GLIMS Valab validation program also supports the station-specific property codes defined through an AssessmentMethod-based code set. In the Valab coding system, you can add a new code set for the AssessmentMethod table. Enter the assessment method record as station-mnemonic/property and the code value representing the Valab test. The codes in the assessment method code set have priority over the ones defined in the property code set.

Warning

Each alternate code should have also a code in the Property code set.

Example

Record Value (Code)
Hit917/K A4
Hit912/K A5

The Valab program in GLIMS

The Valab program can be started manually by choosing the contextual menu function 'Valab validation' from an Order record or via a command:

  • Table: Order
  • Query: Review
  • Function: ValabValidation

Note

When running the Valab program via a command, make sure that 'Login user' of the service which serves this command (via a task scheduler) has at least 1 employee assignment.

The Valab parameters are passed via the Valab method interface. 

Translator

Select the translator that will be used for the Valab communication.  In sites where several Valab connections are used, you can define a separate translator for each communication channel. 

Note

You must use the driver "valab".

Coding system

Coding system to be used for communication with Valab.

Classification

In some cases, Valab requires a group of results to perform a correct validation. GLIMS allows to group results before sending them to Valab.

The option Classification allows to select a property classification. A group is defined as a property classification node with Reportable set to When complete, having child nodes for properties. Of course, only properties available in the property code set of the Valab validation coding system are considered.

Mode

Translator  

Coding system  

Classification

Mode

Default  

  • This mode is only used for backward compatibility.
  • GLIMS will only send confirmed results to the Valab system.
  • Results in GLIMS can be validated after validation by the Valab system.
  • Object attributes and specimen variables are never sent to the Valab system.
  • Options 'Store validation flag' (by default not set) and 'Alternate codes' are ignored.
     

Extended  

  • When the Valab - GLIMS connection is used in a production environment.
  • Results in GLIMS can be validated after validation by the Valab system.
  • GLIMS will send all confirmed and validated results of an order to Valab if the order has at least one confirmed result (of type A) which was not already rejected (or validated) by the Valab system. The validated results of the order are sent along as information for the Valab system.
     

Evaluation  

  • When the Valab - GLIMS connection is used in a testing environment.
  • Results in GLIMS will NOT be validated after validation by the Valab system. This enables the user to consult Valab validation flags (raw flags) in GLIMS' Order review programme without results having already been validated.
  • Any result that is confirmed or validated will be sent to Valab, whether or not the result has already been sent to Valab.
     

Not confirmed results

If set, the minimal result status for sending results to Valab is 'Available'. If not set, results should have at least the status 'confirmed'.  This parameter only has effect in modes 'Extended' and 'Evaluation', not in mode 'Default'.

Hide issuers

If set, issuers will not be sent to Valab.

Hide object attributes

If set, object attributes will not be sent to Valab.

Hide specimen variables

If set, specimen variables will not be sent to Valab.

Strict numeric values

If set, results for Valab codes beginning with an 'A' will be sent to Valab as strictly numerical. This means that alphanumeric comments added to the numeric result will be truncated. If not set, numeric values with comment appended will be sent to Valab.

Expand result values

Allows to indicate that result values should be expanded when transmitted to the Valab expert system.

Note

When Expand result values is disabled, GLIMS will remove the leading '{<' and trailing '}' from the result value being transmitted to the Valab expert system.

Store validation flag

Allows to store the validation flag ({<VALAB-V>}) in the internal comment of the result when Valab validated it.  

Note

The validation flag is never stored in 'default' mode.  This option cannot be used in combination with the 'Use raw Valab flags'.

Allow partial validation

By default, results will only be marked in GLIMS as 'Validated' when both the Valab order validation flag and result validation flag are set to 'Y'.  If the 'Allows partial validation' option is set, only the Valab result validation flag will be taken into account.  This implies that individual results of an order can then be validated in GLIMS even if the Valab order validation flag is not set to 'Y'.

Use raw Valab flags

If set, GLIMS will store the flags received from the Valab system. Check the chapter Raw expert system flags for more information.

Alternate codes

This refers to an Order-based MISPL expression returning a string. This field allows providing alternate codes to be transmitted to the Valab system.
Alternate codes can be provided for the order's issuer and ward.  Pass the alternate codes by putting them in a tagged list. The tagged list is the return string of the function.

  • To pass the alternate issuer code, use the tag 'issuer' as in: CodeList : = PutTag("Issuer", IssuerCode, CodeList, "\");
  • To pass the alternate ward code, use the tag 'ward' as in: CodeList : = PutTag("Ward", WardCode, CodeList, "\");

Note

  • Both issuer and ward tags can be put in the same tagged list.
  • The list should have the '\' delimiter in order to work properly

Example

/* returns as alternate issuer code the last 3 characters of the RIZIV identification code assigned to the issuer */
STRING CodeList;
STRING IssuerCode;
STRING RizivNr;
RizivNr := .Issuer.Identification("RIZIV");
IssuerCode := Substr(RizivNr, Len(RizivNr) - 2, 3);
CodeList := PutTag("Issuer", IssuerCode, CodeList, "\");
RETURN CodeList;

Validator

When specified, the results will be marked as validated by the specified user. If not specified, the validator is the user who started the Valab validation. The validator can typically be used for Valab validations executed by a command.

Object and patient data

GLIMS also sends object attributes and specimen variables information to Valab. The information is only sent if the Valab validation mode in GlIMS is NOT set to 'Default'.

Object attributes

Only attributes with a history before the current date are sent.  The object attribute mnemonic is sent in a C record to Valab.

Specimen variables

Only specimen variables for specimens having selected results (for sending to Valab) are sent to Valab. The information is sent in a C record to Valab in the following format:  Material variable + ':' + Specimen variable choice mnemonic.

Ward mnemonic

The mnemonic of the order encounter stay ward is transmitted to Valab in a O record.

Order issuer internal id

The order issuer internal Id is transmitted to Valab in a O record. This can be disabled by setting the GLIMS Valab parameter 'Hide Issuers'.

Storing Valab flags

There are two ways to store the Valab flags in GLIMS.  

  • Using the raw expert system flags
  • Storing the flags in the result internal comment

Using Raw expert system flags

If the parameter Use raw Valab flags is enabled, GLIMS will store the flags received from the Valab system in GLIMS.  As the parameter Use raw Valab flags is passed to the valb translator, the correct version of the translator is required.   In that way, the translator will know it has to transmit the raw values of the Valab validation flags to GLIMS.  Contact MIPS for an up-to-date version of the translator.

Analogous to the result severity and comment fields, expert system flags are also reset on a result value change. Any new value should be re-evaluated.  Also, when a result is being re-used, the cloned result will NOT inherit the expert system flags of its original. The original result was evaluated by the expert system in its order context, which is different for the duplicate.

The storage of order expert system flags allows easy detection of orders which have been treated by Valab (via the order query).  Possible values are described in the Valab/GLIMS flags relation table above.

Note

  1. When a request for a property within the VALAB Property Coding System is added to an order which already had an expert system flag, this expert system flags field is marked with the 'needs re-evaluation' flag (the asterisk character *).  The same happens when an unsolicited result is added to an order.  When the value of a result which already has an expert system flag is changed, the result's expert  system flag is cleared and the order expert system flags are marked with the 'needs re-evaluation' flag.  The goal is to make sure these orders are selected again by the VALAB query, which typically does a selection via this order expert system flag.
     
  2. 'Pending' results: when transmitting an order to Valab, GLIMS will set the '...' value in the expert system flag field of the result in case the result is pending. Pending means the result was not transmitted to Valab. This can happen for instance when no result value was available yet at transmission time. Results are only marked as pending when:
      
    1. GLIMS found at least one result that can be transmitted
    2. The result property has a Valab code starting with 'A'
    3. The Valab mode was set to 'Evaluation' or 'Extended'

Storing the flags in the result internal comment

If the parameter Use raw Valab flags is not set, the Valab flags can be stored in the result internal comment.  The 'V' flag (which indicates that a result has been validated) will only be stored if the parameter 'Store validation flag' is enabled.  Valab also sends flags when a result has NOT been validated.  In order to store them in the result internal comment, define texts modules with mnemonics like:

'VALAB' + '-' + Code

Where Code corresponds with the flag defined in Valab.

Each Valab code is interpreted as one character. A flag consisting of several characters is split into different GLIMS text mnemonics. Small letters are distinguished from capital letters by the prefixed character 'S'.

Examples:

Valab code GLIMS text mnemonic Text Description
A VALAB-A ....
a VALAB-SA ....
> VALAB-GT ....
< VALAB-LT ....

Since GLIMS Valab verifies the result internal comment for Valab flags to determine if a result has already been sent to Valab, one should manually remove the Valab flags from the internal comment when modifying the result. Valab flags have a syntax like {<VALAB-...}.

Logging

The GLIMS-Valab connection adds logging to results and orders referring to the flags (results) or approval (Rejected/Approved) received from the Valab expert system.  The logging will have the format as in the following examples:

Examples

Order log example:

------------------

23/07/03 10:49:58 Evaluated by BIO [Valab: 'Rejected']

23/07/03 10:51:22 Evaluated by BIO [Valab: 'Approved'] 

Result log example:

-------------------

23/07/03 10:51:06 Modified by BIO [5]

23/07/03 10:51:14 Confirmed by BIO

23/07/03 10:51:22 Ext. evaluated by BIO [Valab: '']

23/07/03 10:51:21 Ext. validated by BIO 

Predecessor:

23/07/03 10:48:30 Modified by BIO [6]

23/07/03 10:48:39 Confirmed by BIO

23/07/03 10:49:58 Ext. evaluated by BIO [Valab: 'A>']

23/07/03 10:51:05 Replaced by BIO [Bad result, replace it]

The empty flag ('') in the result log on 10:51:22 implies that Valab did not reject the result value and that the result in GLIMS may be validated. The GLIMS Valab interface must receive an approval for the whole order in order to validate a result in GLIMS. There is one exception: if the parameter option 'allow partial validation' in the GLIMS valab interface is set, single results may be validated in GLIMS, independent if the order is still not approved by valab.

The Valab validation program also generates general logging: Start->System Management->Logging->Log Types->VALAB

Integration in Order review program

In order to integrate the Valab validation in the order review program, you must identify the external expert system in specific site (System management > Customize > GLIMS).  The Expert system field has three possible values:  Valab, Other, or ?.  Specifying anything different from the unknown value will activate expert system support in the order review program.  

This implies that an "Expert system flags" column will appear in the order review program, next to the standard "Flags" column.  The column displays the Result.ExpertSystemFlags field, i.e. the result's appraisal sent by the expert system.  

Moreover, the order review query will allow querying on the appraisal of an external expert system ('Valab status' field).

Remarks

  • The validation method of results validated by Valab is automatically set to 'external'.
  • Dynamic results are sent more than once to Valab, since there is only one property (and thus Valab code) for dynamic tests.
  • It might be necessary to split an order when it contains multiple instances of the same property, because GLIMS Valab will send them both as the same Valab test.
